Phase 12 (a) - TX doorbell moderation A/B

  • Toggle: echo {1,0} > /sys/module/vnic_l2/parameters/tx_coalesce.

Result

tx_coalesce TX (guest->host) Retr
0 (per-packet doorbell) 414 Mbit/s 0
1 (xmit_more batch) 1.86 Gbit/s 0

~4.5x, zero added retransmits. The off figure matches the L-09 per-packet baseline (~350-414 Mbit/s), so the gain is attributable to removing the per-packet writel(TDT) MMIO VM-exit (D-32 / background-knowledge topic 7.2). It closes most of the L-09 TX/RX gap: TX 0.35 -> 1.86 Gbit/s vs RX ~3.77 Gbit/s (RX was already NAPI-batched).

TCP congestion window (cwnd) - the side effect that confirms the cause

The per-second Cwnd column moved very differently between the two runs (both 0 Retr, so this is not loss-driven):

t (s) off cwnd on cwnd
1 132 KB 204 KB
2 156 KB 228 KB
3 238 KB 280 KB
4 238 KB 305 KB
5 238 KB 329 KB
6 238 KB 397 KB
7 238 KB 1.51 MB
8 238 KB 1.51 MB
9 351 KB 1.51 MB
10 351 KB 1.51 MB

Why. With off, every packet costs an MMIO VM-exit, so the send path drains slowly - the achievable send rate is low, and since cwnd settles around rate x RTT, it stays small (~130-350 KB) and TCP never fills the pipe. With on, the doorbell is amortized over the burst, the send path is no longer exit-bound, the realized rate jumps, and cwnd grows ~6x (to 1.51 MB) to match. Crucially Retr = 0 in both - the small cwnd in off is not congestion/loss, it is the send path being throttled by the doorbell, which TCP simply reads as "this link is slow". So the cwnd curve is independent confirmation that the bottleneck was the doorbell exit, not the wire or loss.
