1.1 · TCP TX flow (HTTP GET) - source -> routers -> destination

Encapsulation adds one header per layer (ports identify the connection, IP the host, MAC the next hop). Blue = encapsulate, green = decapsulate. Routing & ARP detail on the next slide.

1.2 · IP routing, forwarding, and the next-hop MAC

Routing (the sender)

  • The route table maps a destination to an egress interface + a next hop (the gateway, or the dest if on-link).
  • The frame's destination MAC = the next hop's MAC, never the final host's (unless same subnet).

ARP

  • "who-has <next-hop IP>?" broadcast -> the owner replies "is-at <MAC>".
  • Cached in the neighbour table; that MAC fills the Ethernet dst.

Forwarding (a router in the middle)

At each hop the IP packet is delivered up to L3, re-routed, and re-framed:

  • IP addresses stay the same end to end.
  • TTL is decremented (and the IP checksum updated).
  • The L2 header is rewritten: new src MAC = this router's egress MAC, new dst MAC = the next hop's MAC (resolved by ARP).
So: one IP journey, many Ethernet frames - the dst MAC is rewritten at every hop, the IP is not.

1.3 · RX flow (HTTP response) - decapsulation up the stack

Each layer verifies then strips its header. The NIC driver owns L1/L2 (DMA, MSI, NAPI - the rest of this deck); the kernel stack owns L3/L4 up to the socket.

2.1 · Anatomy of a PCIe NIC

PCI config space
vendor/device ID, class, the BAR sizing + the MSI capability - how the host enumerates & maps it
BAR0 = MMIO registers (CSR)
the control surface: identity, control/status, ring base/head/tail, interrupt cause/mask
Descriptor rings (in host RAM)
TX + RX arrays of {addr,len,flags,status}; the device DMA-reads/writes them
DMA engine
reads descriptors + packet buffers, writes RX data + completion status, by bus address
MSI / MSI-X
completion interrupt = a memory write to the CPU's APIC

The contract

  • The driver talks to the device only through (a) MMIO register reads/writes and (b) shared-memory descriptor rings.
  • The device reaches host memory by DMA at the bus addresses the driver programmed.
  • Ownership of each descriptor flips between driver and device via a status (DD) bit + memory barriers.
  • The driver kicks the device with a doorbell (a tail-register MMIO write); the device answers with an interrupt.

2.2 · How the host sees the NIC - address space, MMIO & DMA

The CPU reaches RAM directly (load/store) and the NIC's registers through the BAR0 MMIO window (readl/writel - e.g. the TDT doorbell). The NIC's DMA engine is a bus master: it reads/writes the descriptor rings + buffers in RAM itself; completions raise an MSI. The transceiver (PHY) drives the wire.

2.4 · Where the descriptor ring lives - host RAM vs device MMIO

Two models

  • Ring in RAM (mainstream - VN1000 / e1000 / virtio): the descriptor array is in coherent DMA memory; only the head/tail (TDH/TDT) are MMIO. The CPU writes descriptors with cheap cached stores + one doorbell; the device DMA-reads a burst (with an on-chip prefetcher).
  • Descriptors in device MMIO (minority): the descriptor is a register the CPU writes - so the ring is small and fixed.
CPU MMIO writes are uncached and costly, so writing a whole descriptor per packet does not scale - RAM ring + one doorbell wins for throughput. MMIO descriptors only pay off for tiny/embedded or ultra-low latency (Mellanox BlueFlame).

Textbook datasheet example: Realtek RTL8139

  • TX has no ring in RAM - just 4 fixed register pairs in the chip: TSAD0..3 (the packet's DMA address -> RAM) + TSD0..3 (length / status). The driver round-robins NUM_TX_DESC = 4.
  • Matches "descriptors in MMIO, ring fixed at 4, buffer in RAM" exactly.
  • Its cost (only 4 in-flight TX) is why RTL8169 / e1000 switched to an in-RAM ring - the VN1000 model.

VN1000 deliberately uses the mainstream RAM-ring model (background-knowledge topic 8).

3.1 · Linux PCIe NIC driver framework

net_device + net_device_ops
ndo_open / stop / start_xmit / get_stats / change_mtu
↓
core datapath
TX/RX rings, the DD handshake, NAPI poll - hardware-independent
↓
HAL seam
rd32/wr32, ring alloc, map/unmap, irq, init - the one place hardware-specifics live
↓
device backend
MMIO + DMA + MSI to a real (here: emulated) PCIe device

The lifecycle

  • Init (module_init) - pci_register_driver() with an ID table; the PCI core then calls probe on a match (L1 has no bus, so it builds the netdev directly).
  • Probe - the driver claims the matched device, maps the BAR, allocs the rings, and registers the netdev (it appears, but DOWN).
  • Open - post RX buffers, request the IRQ, enable the device, start the queue.
  • TX / RX - the steady-state datapath (next slides).
  • Stop / Remove - disable, quiesce, free, unregister.

The next slides walk each piece: TX, RX, init/probe, MMIO & barriers, DMA, ISR, cleanup.

3.2.1 · Linux NIC Driver — TX Packet Transmission Path

TX in five steps (A-E)

  • A - init: alloc + map the TX ring; program its base + head/tail (TDH/TDT) into the CSR.
  • B - submit (ndo_start_xmit): DMA-map the skb, fill a descriptor (clear DD = NIC-owned), dma_wmb, then ring the TDT doorbell.
  • C - device + DMA: the NIC DMA-reads the descriptor + packet, sends it on the wire, then writes back DD (done = driver-owned).
  • D - completion IRQ: it raises a (coalesced) MSI; the ISR acks ICR, masks the cause, and napi_schedule().
  • E - cleanup: the NAPI poll reaps DD descriptors, dma_unmap + frees the skb, wakes the queue, re-arms the IRQ.

3.2.2 · TX ring — DD ownership & the driver's cursors

Driver produces, device consumes. DD is the ownership bit - green DD=1 = driver-owned (completed, reclaim it), amber DD=0 = device-owned (in flight). The hardware pointers TDH/TDT sit beside the driver's two software cursors: NTU (next_to_use, the producer - next slot to fill) and NTC (next_to_clean, the reclaim point). NTU may lead the published TDT when the doorbell is batched.

3.3.1 · Linux NIC Driver — RX Packet Reception Path

RX in five steps (A-E)

  • A - init: pre-post an empty buffer into every RX descriptor (Model A); program the ring base + RDT.
  • B - arrival + DMA: the NIC DMA-writes an incoming frame into the next buffer, sets len + DD, advances RDH.
  • C - interrupt: it raises an MSI; the ISR acks, masks the cause, and napi_schedule().
  • D - NAPI + upstream: the poll reads DD descriptors (dma_rmb), skb_put + eth_type_trans, then napi_gro_receive() up the stack.
  • E - cleanup + recycle: refill the slot with a fresh buffer, advance RDT; re-arm the IRQ once drained.

3.3.2 · RX ring — DD ownership & the driver's cursor

Device produces, driver consumes - the mirror of TX. green DD=1 = a received frame (driver-owned, hand it up); amber DD=0 = a posted-empty buffer the device still owns. Here only NTC (next_to_clean) is used - it consumes a frame then refills the slot, publishing RDT = NTC at end of poll. NTU is unused on RX, because the device is the producer (it advances RDH).

3.4 · Driver init & probe

Module init

  • pci_register_driver() with an ID table ({vendor, device}).
  • The PCI core enumerates the bus; on a match it calls probe.

Probe

  • pci_enable_device + pci_request_regions
  • pci_iomap(BAR0) -> the MMIO base
  • pci_set_master (bus-master, for DMA) + dma_set_mask
  • allocate the descriptor rings; verify the ID reads back
  • alloc_etherdev + register_netdev -> the interface appears

Why this order

Enable + map before touching registers; set the DMA mask before any DMA alloc; register the netdev last, once the device is ready - so the first open can't race an unfinished setup.

register != open: after probe the interface exists but is DOWN; insmod does not bring the link up. ndo_open (on ip link set up) is what posts RX buffers, requests the IRQ, and starts the queue.

3.5 · MMIO, barriers & the DD handshake

MMIO - readl / writel

  • Register access goes through accessors, never a raw pointer dereference.
  • They guarantee a single, ordered, never-elided access of the right width + correct endianness.
  • Writing the tail (TDT/RDT) register is the doorbell.

The DD ownership handshake

  • Producer: write buffer/len, dma_wmb(), then set DD (publish).
  • Consumer: READ_ONCE(DD), dma_rmb(), then read the buffer (observe).
  • dma_* (not smp_*) because the peer is a device (L1 engine: smp_*).

Synchronization is not a lock

You cannot lock a device, so the rings are race-free by four cooperating guarantees:

  • Single-owner protocol - DD says who owns each descriptor; only the owner writes it.
  • Ordering - dma_wmb before the flag, dma_rmb after it (no "saw DD, read stale payload").
  • READ_ONCE - forces a fresh reload of the flag so the compiler can't spin on a cached value. It is not a fence - the compiler-level companion to dma_rmb.
  • Reachable memory - coherent ring + buffers dma_unmap'd before the CPU reads them.

The driver's tx_ring.lock only serialises its own xmit (process) vs clean (softirq) - not the device.

3.6 · Driver DMA

Two kinds of mapping

  • Coherent (dma_alloc_coherent) - the descriptor ring: CPU and device both see it without explicit syncs; its bus address goes into TDBAL/H, RDBAL/H.
  • Streaming (dma_map_single) - the packet buffers: mapped per transfer, with a direction; unmapped on completion (which also syncs for the CPU).

The bus-address view

  • The device DMAs to bus (DMA) addresses, not CPU pointers - that is what goes in vn_desc.addr.
  • Bus-master must be enabled; an IOMMU (if present) translates these.
  • Always check dma_mapping_error(); keep the original skb separately to free it (the token is opaque).

3.7 · Driver ISR & NAPI

Top half (the ISR)

  • Runs in hard-IRQ context - do the minimum.
  • Read ICR (read-to-clear), mask the cause (IMC), napi_schedule(). Done.

Bottom half (NAPI poll, softirq)

  • Does the work: clean TX completions, receive RX up to a budget.
  • Drained below budget -> napi_complete_done() + re-enable the interrupt (IMS).

Why IMS/IMC (and ICR/ICS) are split pairs: write-1-to-set / write-1-to-clear lets the ISR mask and the poll re-arm flip only their own bits with one atomic write - no read-modify-write race, no costly MMIO read of the mask.

Why interrupt + polling?

One interrupt wakes a poll that drains many packets, then re-arms - this avoids an interrupt-per-packet storm at high load while staying low-latency when idle (NAPI).

Interrupt moderation (ITR) is a second, device-side cap on the interrupt rate.

3.8 · Driver cleanup

Stop (interface down)

  • Stop the queue (wait out any in-flight xmit).
  • Disable the device (clear CTRL); quiesce (wait out in-flight device work / synchronize_irq).
  • Stop NAPI, mask all interrupts, free the IRQ.
  • Drain the rings: unmap + free every buffer the driver still owns.

Remove (module / device gone)

  • unregister_netdev (closes the device first if up).
  • Free the rings, unmap BAR0, release regions, disable the PCI device.
Ordering matters: never free memory the device might still be DMA-ing into - disable + quiesce first, then drain.

3.9 · TX/RX offloads — TSO/GSO (segment) · LRO/GRO (coalesce)

Off the per-packet hot path

  • TX (segment): one ~64 KB super-skb → N×MSS frames. GSO splits in software (just before the driver); TSO hands the device one skb (SG, few desc, 1 doorbell) and the device splits.
  • RX (coalesce): N×MSS frames → one skb. GRO merges in the NAPI poll (software); LRO merges in the device.
  • MSS = MTU − IP − TCP = 1460; frames stay ≤ MTU — jumbo's win without jumbo on the wire.
  • Needs checksum offload (the device must (re)checksum each segment).
  • VN1000: GRO on (napi_gro_receive); TSO / LRO / checksum = backlog A6 (the TX engine would become a segmenter).

4 · The QEMU PCIe NIC device model

How the host side is built

  • QOM type parented to TYPE_PCI_DEVICE; instance struct embeds PCIDevice.
  • realize() = the constructor: create the BAR MemoryRegion + pci_register_bar; msi_init; (real NIC) create the backend.
  • MMIO callbacks (MemoryRegionOps.read/write) = the device's register logic (the doorbell, ICR/IMS/...).
  • DMA: pci_dma_read/pci_dma_write at the driver's bus addresses.
  • Interrupt: msi_notify.

Each guest action -> a callback

Guest doesQEMU runs

writel(TDT) doorbellwalk the ring, pci_dma_read + send

A backend (-netdev tap) carries frames out: TX qemu_send_packet, RX a receive() callback.

5.1 · The vNIC project goal

Understand the Linux NIC driver working flow by building one - the same driver core, three progressively more real "devices" behind one HAL seam.

Level 1 - in-kernel engine

  • Pure kernel module: vnic_loop0.
  • "Device" = an engine kthread; fake registers; the buffer pointer is the DMA token; a simulated-MSI tasklet.
  • Internal IP-swap loopback - exercises the whole datapath with no hardware.
  • Goal: real-NIC structure (netdev + rings + NAPI + the interrupt protocol), validated by a test suite.

Level 2 - QEMU PCIe device

  • vnic0, bound to an emulated PCIe device in a self-built QEMU.
  • Real MMIO, DMA (coherent ring + streaming buffers), real MSI.
  • Default loopback; or a real tap backend -> ping the host, reach the internet.
  • Goal: the same core, now over a real PCIe/DMA/MSI path; both levels load at once.

5.2 · Design: the HAL seam — one core, two backends

Why one HAL seam

  • All of core/ is written once and reused unchanged at every level.
  • Only the HAL backend changes per level (D-12).
  • The seam is the smallest set of operations the core needs to touch the device.
  • The opaque device token lets L1 use a kernel pointer and L2 a DMA address - the core never dereferences it (real skb stays in ring->bufs[]).
  • Doorbell (TDT/RDT) and mask (IMS/IMC) are just wr32 - no extra HAL verbs.
  • Compile-time LEVEL=1|2 links exactly one backend.

5.3 · Dataflow: Level 1 — in-kernel engine (loopback)

Level 1 — one address space

  • No PCI / MMIO / DMA - a pure kernel module.
  • The engine kthread is device + wire.
  • The TDT doorbell (a wr32) wakes it.
  • The token is a kernel pointer - direct deref, no DMA.
  • An IP-swap loops the frame into the RX ring.
  • A tasklet stands in for the MSI.

5.4 · Dataflow: Level 2 — QEMU device, internal loopback

Level 2 — across the PCIe boundary

  • The same core, now over real PCIe.
  • Rings live in guest RAM (dma_alloc_coherent).
  • The device reaches them by real DMA (pci_dma_read/write).
  • Rung over MMIO: writel(TDT).
  • Completes with a real MSI → NAPI.
  • loopback=on: loops internally, no backend.

5.5 · Dataflow: Level 2 — QEMU tap backend

Level 2 — tap backend (real frames)

  • loopback=off bridges real frames guest↔host↔internet.
  • TX: pci_dma_read → qemu_send_packet → host tap.
  • RX: backend .receive() → DMA-write RX + MSI.
  • Flow control: can_receive() + flush (RDT / RXEN).
  • The device is backend-agnostic (D-30).
  • The PCIe half is identical to 5.4.

The lens: per-packet vs per-byte

Doorbell-batching, ITR and MTU all attack the per-packet fixed cost (descriptor, DMA-map, skb, stack, the MMIO exit). Checksum is per-byte - only an offload removes it. Coalescing buys throughput/CPU at the cost of latency; polling buys latency at the cost of CPU.
