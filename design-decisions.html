D-01 Goal and the three-level path

Decision: the goal is to understand the Linux NIC driver working flow end to end by building a software-simulated NIC and its driver. Path: Level 1 (kernel-only) -> Level 2 (QEMU PCI device) -> Level 3 (FPGA, bonus). Why: it exercises the core architecture shared by NIC / NVMe / GPU / accelerator drivers (queue + DMA + completion + interrupt + NAPI). Phasing keeps every step runnable and demonstrable, without getting stuck on hardware / QEMU complexity up front.

D-02 Device name and register style - VN1000

Decision: name the device VN1000 (1GbE virtual Ethernet controller); PCI 0x1234:0x11E9; registers in Intel e1000e style (CTRL/STATUS, TDBAL/TDH/TDT, ICR/IMS/IMC...), descriptor in virtio style (addr/len/flags/status). Why: e1000e is well documented and the most approachable head/tail ring driver, so what is learned maps directly onto real Linux source. Vendor 0x1234 is QEMU's test vendor, so the guest will not bind a real driver to it.

D-03 Device and driver share one spec header

Decision: register offsets and the descriptor layout live in include/vnic_regs.h, shared by the driver and the QEMU device. Why: a mismatch between the two sides is the most common bug in this kind of project; a single source avoids drift. The authoritative spec is docs/vn1000-datasheet.md; the header is derived from it.

D-04 Level 1 = single netdev (real-NIC structure), not a char device

Decision: Level 1 is "one real NIC" - a single netdev vnic0 (net_device ops + NAPI + TX/RX ring + DMA-like + simulated interrupt), with the engine acting as the "hardware + wire" behind it. Why: the goal is something that "looks like a real NIC's structure". The netdev structure is identical to Level 2, so Level 1 -> Level 2 only swaps the hardware-access layer; the core is untouched. Rejected: (a) char device + ioctl (datapath sim, but not wired to the network stack, not pingable); (b) snull-style two netdevs (looks like "two NICs", not "one").

D-05 Internal loopback by swapping src/dst IP only (keep L4 ports)

Decision: the engine loops a transmitted frame back into the RX ring with only the src/dst IP swapped (ports/payload unchanged) (reflection-on-IP). Assign vnic0 IP=A; the fictional peer B is on the same subnet. Why: - Swapping is "reordering the same 16-bit words" -> the IP and TCP/UDP checksums stay valid, no recomputation -> ICMP/TCP/UDP all testable. - After the swap dst=A (local) -> the IP layer delivers locally to L4; src=B (non-local) -> avoids the "plain loopback with src=A gets dropped as a martian source" problem. Rejected: snull's "flip a bit in the third octet" changes the address values -> breaks the TCP/UDP checksum (snull only recomputes IP, hence ICMP-only); and snull needs two interfaces.

D-06 Why not snull's two devices / why not netns

Decision: Level 1 uses a single device with IP-swap; real TCP/UDP load testing is left to Level 2 against the host. Why: snull (LDD3, 2005) uses two devices + bit-flip to teach "two hosts on one wire", and network namespaces did not exist then. netns (~2008) is the modern clean way to have "two ends" but needs two instances. We want "one NIC", so a single device fits best. Level 2 via the QEMU backend reaches the host (a genuine remote), which is the accurate place to load-test TCP/UDP.

D-07 TCP/UDP test method

Decision: server bound to 0.0.0.0 (or A); client connects to the fictional peer B; after the engine swaps the IP the packet reaches the server on A. Why: the server must bind a local address (cannot bind B, which is non-local); the client must target the non-local B (targeting A/localhost goes via lo and bypasses vnic0). Swapping the IP only (keeping ports) preserves the dst port, so it lands on the server.

D-08 ARP / L2 handling

Decision: resolve B's MAC via (a) a static neighbor (ip neigh ... permanent, zero code) or (b) the engine answering ARP itself; on loopback set the RX dst MAC = vnic0's own MAC (-> PACKET_HOST), src MAC = B's fake MAC. Why: B does not exist -> normal ARP gets no reply -> the packet cannot be sent. Keep B on vnic0's subnet so strict rp_filter passes. The point is to leave the ARP/neighbour path on (so vnic0 behaves like a real NIC and the resolution stays observable) and merely pre-seed the one entry that has no responder. Rejected: snull's (LDD3) approach - set IFF_NOARP on the netdev so the stack never emits ARP at all, plus a custom header_ops->create (snull_header) that fabricates the Ethernet header (dest MAC = the device's own MAC with the low bit of the last byte toggled, which maps to its paired sn0/sn1). That needs zero external config, but it turns ARP off and the L2 addresses are fake - fine for snull's closed two-interface bounce, but VN1000 wants to look like a real NIC, so it keeps ARP intact and pre-seeds a static neighbour instead. (See D-05 / D-06 for why VN1000 also rejects snull's two-device + third-octet bit-flip topology.)

D-09 Baseline convergence (drop MODE / RST_DONE, single MSI)

Decision: (1) drop the device-side MODE register - int/poll/adaptive is achieved by the driver's NAPI; (2) reset uses only the self-clearing CTRL.RST, drop STATUS.RST_DONE; (3) baseline uses a single MSI + explicit IMC. Why: (1) real NICs have no MODE register, and NAPI is itself an int+poll hybrid; (2) two ways to detect reset completion are redundant; (3) a single MSI is enough for a single queue. See D-13 for the advanced counterparts.

D-10 MAXFRM register + Change MTU

Decision: add MAXFRM (max frame size); ndo_change_mtu quiesces -> reallocates RX buffers -> re-enables. Why: the device must know the RX buffer / max frame (cf. Intel RCTL.BSIZE / RLPML). In the single-buffer model MAXFRM doubles as the RX buffer size; TX in the skb zero-copy model needs no buffer reallocation (only the copy model does). Multi-descriptor split and the buffer/frame register split are tracked in backlog A4.

D-11 Interrupt cause/mask design (ICR/ICS/IMS/IMC; IAM is advanced)

Decision: ICR (read-to-clear) / ICS (WO) / IMS (RW1S) / IMC (RW1C); ISR reads ICR -> IMC mask -> NAPI -> after poll IMS re-enable. IAM (auto-mask) is backlog A1. Why: modeled on Intel. Splitting the mask into IMS/IMC (set/clear write paths) makes updates atomic and avoids a read-modify-write race. IAM (atomic ack+mask, removing the "read ICR -> write IMC" window) is an advanced refinement.

D-12 Driver layout and HAL naming

Decision: driver/ = core/ (shared netdev/ring/NAPI/datapath/stats) + hal/ (hardware-access abstraction with the L1/L2 implementations) + level1/, level2/ (level-specific: L1 engine, L2 PCI glue). The access layer is named hal; register read/write helpers are vnic_hal_rd32 / vnic_hal_wr32; a build-time LEVEL=1|2 produces vnic-l1.ko / vnic-l2.ko. Why: the real L1/L2 difference is the "hardware-access layer (reg/DMA/IRQ/doorbell)"; factoring out core/ avoids duplication. The doorbell (writing TDT/RDT) and interrupt mask (IMS/IMC) are just vnic_hal_wr32, so the HAL surface stays small. hal covers reg+dma+irq better than regops. The register helpers take the vnic_hal_ prefix (not a short vn_): they are public, non-static, backend-specific functions (different L1/L2 implementations) called from level-independent core/, so the prefix marks every HAL crossing. The shared device spec keeps the VN_* / struct vn_desc naming, and level-independent core logic stays vnic_*. Rejected: scattering the access impls across level1//level2/ (defeats "access functions in one place"); a runtime ops table (only needed if one module must support both levels at once; compile-time selection is simpler here).

D-13 Advanced backlog

Decision: A1 IAM (auto-mask); A2 forced mode comparison (int/poll/adaptive + MODE register); A3 MSI-X / multi-queue / RSS; A4 jumbo multi-descriptor RX + buffer/frame register split. Why: keep the baseline main path minimal and runnable; these bring the project closer to a production NIC and deepen understanding of real-NIC features.

D-14 Development environment - Option B (QEMU guest)

Decision: run QEMU + an Ubuntu guest under WSL2; do all kernel-module development inside the guest. Why: the WSL2 custom kernel has no matching linux-headers in apt, so out-of-tree modules cannot be built there; the guest has headers, kgdb works, and a module panic only kills the guest; Level 2 needs a QEMU guest anyway, so this unifies the environment. Rejected: (A) building the WSL2 kernel to get a build tree (stays in WSL2 but a panic kills the whole WSL2 VM).

D-15 Documentation and diagram conventions

Decision: the block diagram is a hand-authored SVG (docs/diagrams/host-address-space.svg - a CPU / RAM-rings / BAR0-MMIO / DMA-engine / transceiver address-space view with the full CSR map, shared with the slide deck; it replaced the earlier app->stack->driver layering SVG); docs/architecture.md's register/descriptor sections point to the datasheet (the authoritative source) to avoid drift; the config-space doc lists the full Type 0 header + capabilities. Document language: see D-17. Why: Mermaid's auto-layout (dagre) cannot fix coordinates and repeatedly failed to reproduce the requested layout; an SVG gives exact positions and renders on GitHub.

D-16 API reference format - kernel-doc

Decision: function / struct comments use kernel-doc (/** func() - ... @param: ... Return: */); scripts/gen-docs.sh calls the kernel build tree's scripts/kernel-doc to generate man pages (and HTML when mandoc is present). The existing include/vnic_regs.h, driver/vnic.h, driver/hal/vnic_hal.h were annotated; new files follow the same convention. Why: kernel-doc is the de-facto standard for Linux kernel code (docs.kernel.org uses it), so it matches kernel documentation conventions; the comments read well as plain text. The man/mandoc path has few dependencies (only the kernel build tree), avoiding a Sphinx setup. Rejected: Doxygen (general C/C++ standard, can draw call graphs, but not a kernel convention and incompatible with kernel-doc); Sphinx + Breathe / Hawkmoth (nicer HTML but heavier out-of-tree setup, kept as an optional future upgrade). Whether to use Doxygen for the user-space tools/ is to be evaluated later.

D-18 RX buffer model - pre-posted skb (Model A)

Decision: RX buffers are posted to the ring before any frame arrives (mandatory - the device needs a destination to DMA into). The baseline uses Model A: pre-allocate one sk_buff per RX descriptor and store its buffer address in the descriptor. On completion the skb is already populated, so the driver only does skb_put(len) + eth_type_trans + napi_gro_receive, then allocates a fresh skb to refill the slot and advances RDT. In Level 1 the loopback engine copies the TX frame (with src/dst IP swapped) into the pre-posted RX skb buffer; in Level 2 the device DMA-writes into it - the driver side is identical. Why: simplest and clearest for the learning goal, matches the classic e1000 design and our single-buffer descriptor model (MAXFRM = RX buffer size); zero-copy on the completion side (the skb is allocated at refill, not on the hot path). Keeping the buffer pre-posted also preserves the device/driver boundary unchanged across L1/L2. Rejected (-> backlog A5): Model B - post plain buffers / pages and build the skb at completion (build_skb / napi_build_skb, page_pool recycling, copy-break for small frames, frags for jumbo). This is what modern high-speed drivers (ixgbe, virtio-net) do and is more memory-efficient, but it is more complex and not needed for the baseline.

D-19 Loopback RX-full handling - backpressure, not drop

Decision: when the Level 1 loopback engine finds the RX ring full, it stalls (stops consuming TX, leaves the descriptor un-DD) instead of dropping the frame. The TX ring then fills and netif_stop_queue throttles the sender; NAPI refilling RX rings the RDT doorbell, which wakes the stalled engine. RX ring stays small (256). Why: measured. With drops, a 256 ring caused 745 TCP retransmits (4.41 Gbit/s) and a 4096 ring removed them but cost throughput (3.62 Gbit/s, ~6 MB cache-thrashing working set). Backpressure with a 256 ring gave 0 retransmits at 6.03 Gbit/s - best on both axes: no drops (cwnd grows, no retransmit overhead) and a cache-friendly ring. The right fix for overflow was "do not drop" (convert to flow control), not "enlarge the ring". See lessons-learned L-02 and results/loopback-iperf3.md. Note: the resume path (RX refill doorbell wakes the engine) plus the napi_schedule after every stall guarantees a poll always runs to drain RX, so the stall cannot deadlock. A real NIC does not couple TX completion to RX this way - the coupling is inherent to the loopback model (a TX frame becomes an RX frame), so this models a congested link that stops accepting frames.

D-20 Phase 4 simulated MSI - tasklet, not a kthread/workqueue

Decision: in Phase 4 the engine stops calling napi_schedule() directly. Instead the engine (the device, still a kthread) sets the interrupt cause (ICS -> ICR bit) and "fires the MSI" only if unmasked (IMS); the simulated MSI handler runs as a tasklet and does the real ISR sequence - read ICR (read-to-clear), mask the cause via IMC, then napi_schedule(). After the poll, napi_complete_done() re-enables the cause via IMS. Why: a tasklet runs in softirq context - it cannot sleep, runs deferred, and is separate from the engine - which is the closest software analogue to a hardware interrupt handler, so it models the MSI -> ISR -> NAPI timing and the IMS/IMC masking faithfully. The mask makes the interrupt + polling hybrid real: completions accumulate in ICR while masked and do not re-fire until the poll re-enables IMS. Rejected: (a) keep calling napi_schedule() inline from the engine (Phase 3's shortcut - no register protocol, does not teach the ISR/mask flow); (b) a workqueue (the original TODO wording) - runs in process context like another kthread, can sleep, higher latency, least like a real interrupt. At Level 2 the tasklet is replaced by the real MSI hardirq handler via request_irq(); the vnic_poll half is unchanged.

D-21 Descriptor-ring barriers - dma_ in core, smp_ in the L1 engine

Decision: the shared driver core (core/vnic_netdev.c) uses dma_wmb()/dma_rmb() for the TX/RX descriptor DD-bit handshake; the Level 1 engine (hal/vnic_hal_l1.c) keeps smp_wmb()/smp_rmb(). Why: from the driver's side the peer is a device - the L1 engine today, the L2 QEMU device (real DMA) later - so dma_* is the correct family (on ARM/PowerPC dma_* covers the device's coherence domain, where smp_* would not). The L1 engine is a CPU (a kthread), so its side is CPU-to-CPU and smp_* is the precise tool. The pair still holds because dma_* is at least as strong as smp_*. Using dma_* in core makes it correct for both levels, so moving to Level 2 needs no datapath/barrier change in core. See background-knowledge.md topic 3.

D-22 HAL map-error contract - check every vnic_hal_map()

Decision: vnic_hal_map() is paired with a predicate vnic_hal_mapping_error(priv, token) (mirroring dma_mapping_error()), and the core checks every map result. On failure: TX (ndo_start_xmit) drops the frame (tx_dropped, free the skb) and does not advance the ring or ring the doorbell, so the device never sees the half-filled descriptor; RX (vnic_rx_post and the vnic_rx_clean refill) frees the new skb, keeps the old buffer mapped in the slot and drops the frame - the RX invariant "every slot always holds a mapped buffer" is preserved. The refill maps the replacement before delivering the received skb so a map failure cannot leave an empty slot. The same recycle path also covers the earlier allocation failure (napi_alloc_skb returning NULL, not only a map error): either trigger keeps the old buffer in the slot and drops the frame (rx_dropped), so the "every slot always holds a mapped buffer" invariant holds whichever step fails. The general "recycle beats leak" reasoning is in background-knowledge.md topic 1.1.

Cost / impact of the two options (why recycle, not deliver-and-leak). On a failed replacement the slot holds one buffer and no spare, forcing a choice:

Option Immediate cost Downstream impact
Deliver the old frame, do not refill the slot (rejected) none for this packet the slot has no buffer, so it cannot be published - advancing RDT over a buffer-less slot would make the device DMA into freed memory. RDT therefore stops keeping up with RDH, and the pool of posted RX buffers [RDH, RDT) drains by one per failure until it reaches zero: the device hits RXEMPTY and RX stalls. The descriptor array itself does not resize - it is the count of posted buffers that drains. In this single-next_to_clean-cursor Model A there is no recovery point: the cursor has advanced past the slot and nothing records that it still owes a buffer, so it is never revisited.
Keep the old buffer, drop the received frame (chosen) one packet (rx_dropped++), recovered by TCP retransmit or tolerated by UDP the slot stays published (RDT advances normally), so the posted pool stays full and the path is self-healing: the next successful alloc refills normally and RX continues at full depth. rx_errors stays 0 - a buffer-pressure drop is backpressure, not a hardware / protocol error.

The decision trades a bounded, recoverable cost (one dropped packet, fixed up by TCP) for avoiding an unbounded failure (posted-pool drain -> RX stall) that this single-cursor design cannot undo. It is also the lowest-cost, simplest option: recycling keeps the ring full with zero extra bookkeeping - no deficit counter, no second RX cursor, no separate refill path - and a transient allocation failure heals itself on the next poll.

Could we compensate later instead? Yes - deferred / lazy refill (e.g. ixgbe) lets the slot stay un-posted, records the deficit with a second RX cursor (a next_to_use / next_to_alloc lagging next_to_clean), and retries the allocation on a subsequent poll, advancing RDT only once it succeeds. That does recover the capacity at a later point, but adds the second cursor + separate refill-path bookkeeping and buys nothing in this single-buffer-per-packet model - so VN1000 keeps the simpler recycle (deferred refill is tracked as an advanced TODO). Either way the principle is the same as the loopback engine's RX-full backpressure (D-19): never lose the ring's ability to receive. Why: streaming DMA (dma_map_single) can fail on Level 2, and a bad token handed to the device is a DMA fault / corruption. A companion predicate keeps the token opaque (D-12) and lets each level answer correctly - Level 1 only fails a NULL token; Level 2 returns dma_mapping_error(). Recording it now (while only L1 exists) means the L2 backend just implements the predicate, with no datapath change. See Phase 5.2 and background-knowledge.md topic 1 (E1/E2 lifecycle).

D-23 Ring locking discipline - TX lock, RX lockless (NAPI-serialised)

Decision: the TX ring is guarded by tx_ring.lock - ndo_start_xmit takes it with spin_lock_bh(), vnic_tx_clean (NAPI softirq) with plain spin_lock(). The RX ring takes no driver lock: vnic_rx_clean is its only accessor and NAPI serialises polls. rx_ring's next_to_use is unused (the device is the RX producer); only next_to_clean tracks the driver. The queue stop (xmit, on full) and wake (tx_clean, after freeing) decisions both run under tx_ring.lock, so there is no missed wakeup; ndo_start_xmit also stops the queue proactively after a full enqueue and returns NETDEV_TX_BUSY only as the rare-race safety net. Why: the TX producer (process ctx) and consumer (softirq) are not serialised against each other by the stack, so they need a lock; spin_lock_bh() in xmit also fences the local NAPI softirq. Putting stop and wake under the same lock makes them atomic w.r.t. the index updates, removing the classic SPSC missed-wakeup race without an explicit smp_mb() + re-check (the lockless alternative). RX has a single serialised consumer, so a lock would be pure overhead. The DD-bit ownership handshake is ordered by dma_wmb()/dma_rmb() (D-21), independent of these locks. Rejected: one lock shared by both rings (RX contends for nothing); lockless TX SPSC (needs the smp_mb() + re-check dance and careful barrier review - more error-prone for no measured gain here).

D-24 Level 2 interrupts - a single MSI vector

Decision: Level 2 uses one MSI vector (the device declares msi_init(pdev, 0, 1, ...) with VN_CAP_MSI; the driver requests it with pci_alloc_irq_vectors(pdev, 1, 1, PCI_IRQ_MSI) + request_irq()). All causes (TXDONE/RXAVAIL/RXEMPTY/ERR) share that one vector and are demultiplexed by reading ICR in the ISR, exactly as the single L1 tasklet did. Why: the device is single-queue (one TX + one RX ring), so a single completion vector is sufficient and is the simplest faithful model; it keeps the L2 ISR a 1:1 replacement for the L1 simulated-MSI tasklet (D-20) and the vnic_poll half unchanged. MSI (not legacy INTx) because the whole interrupt design - ICR/ICS/IMS/IMC, raise-by-memory-write - is modelled on MSI. Rejected: MSI-X / multi-vector / multi-queue (per-queue or per-cause vectors) - real value only with multiple queues and RSS, which is explicitly backlog A3; legacy INTx - level- triggered shared-line semantics are a worse fit and add no learning over MSI.

D-25 Level 2 datapath - device-internal IP-swap loopback first (tap later)

Decision: Phase 9's device datapath is a device-internal loopback identical to the Level 1 engine - the QEMU device DMA-reads each TX frame and writes it back into the RX ring with the Ethernet MAC and IPv4 src/dst swapped (D-05), raising TXDONE+RXAVAIL. RX-full uses backpressure (D-19), resumed by the RDT doorbell. Only in Phase 10 does a real tap backend replace the loopback (kept as a selectable device mode). Why: it lets the entire existing Level 1 test suite run unchanged on Level 2, so the new DMA/MMIO/MSI plumbing is validated against a known-good datapath before any external networking - a Phase 9 failure is then unambiguously plumbing, not the backend. Reusing the proven IP-swap transform keeps TCP/UDP checksums valid (D-05/D-07). Rejected: jumping straight to tap (couples plumbing bring-up with host tap/bridge setup - two failure domains at once); a different loopback transform (the L1 one is already test-validated).

D-26 Level 2 DMA model - coherent ring, streaming buffers, device pci_dma_*

Decision: descriptor rings use dma_alloc_coherent() (CPU and device see them with no explicit sync; desc_dma is programmed into TDBAL/H, RDBAL/H); packet buffers use streaming dma_map_single()/dma_unmap_single() (vnic_hal_mapping_error = dma_mapping_error, D-22); and the QEMU device reaches guest memory with pci_dma_read/pci_dma_write at the driver's bus addresses. The L2 ISR is a real request_irq handler that masks via IMC and napi_schedule()s; vnic_poll re-arms IMS (unchanged from L1). The shared core/ is untouched (D-12). Why: coherent memory for the small, frequently-walked descriptor ring avoids per-descriptor sync; streaming maps for packet buffers are the normal NIC pattern (the device DMAs the skb data). The DD-bit handshake + dma_wmb()/dma_rmb() in core (D-21) already orders ownership for a device peer, so the datapath logic was unchanged - moving the vnic_hal_* backend was enough. Portable RX completion: core's vnic_rx_clean unmaps the old RX buffer before delivering it (vnic_hal_unmap = dma_unmap_single, which also syncs it for the CPU), then installs the freshly mapped replacement; the error paths keep the old still-mapped buffer in place and drop. This is the one HAL call added to core for Level 2 - a no-op on Level 1 - and it makes the RX path correct even under a vIOMMU / non-coherent host (no elided unmap, no dma-debug splat), not just on the cache-coherent x86 q35 target. TX already unmaps in vnic_tx_clean. Rejected: coherent buffers for packets too (wasteful, unlike real NICs); leaving the per-frame RX unmap out and relying on the x86-no-vIOMMU no-op (works on the current target but leaks the mapping under an IOMMU and is not how a real driver behaves - rejected in favour of the portable form).

D-27 Level 2 device build - compiled into a self-built QEMU

Decision: the QEMU device (qemu-device/vnic.c) is compiled into a checked-out QEMU source tree (hw/net/vnic.c + a meson/Kconfig hook, sharing include/vnic_regs.h) and run as a self-built binary pinned to the guest's current QEMU version; run-qemu.sh points at it via QEMU=. Full SOP in qemu-build.md. Why: QEMU has no stable out-of-tree device-module ABI (unlike a Linux .ko), so a device must be built inside a QEMU tree. Pinning to the guest's version keeps emulated behaviour matched to what the guest already runs. This is a WSL2-host task, distinct from the in-guest driver work (D-14). Rejected: an out-of-tree QEMU plugin/module (no supported ABI); patching the distro QEMU binary (not reproducible); a separate userspace process speaking a custom protocol (would not exercise the real PCI/MMIO/DMA/MSI path, which is the point of Level 2).

D-28 Phase 8 bring-up - PCI/MMIO/MSI keystone before the datapath

Decision: Phase 8 brings up only the bus/register/interrupt plumbing and stops there. The device exposes a BAR0 register file (read-only ID/VERSION/CAP/STATUS/TDH/RDH/stats, read-to-clear ICR, RW1S/RW1C IMS/IMC, CTRL with self-clearing RST); the TDT/RDT doorbells only store their value. The driver's vnic_probe() enables the device, maps BAR0, checks the ID magic, sets up the MSI, and runs a probe-time self-test (raise TXDONE via ICS, wait for the ISR) - but does not register_netdev() or open the datapath. The HAL gets the pci_dev from to_pci_dev(netdev->dev.parent) (set by SET_NETDEV_DEV), so vnic_hal_init(priv) keeps the shared single-arg signature; vnic_hal_ring_alloc/map/unmap are Phase-9 stubs. Why: proving MMIO + MSI + ABI in isolation makes the keystone debuggable on its own (lspci, ID read-back, one MSI to the ISR) before layering DMA on top in Phase 9 - a failure then is unambiguous. Not registering the netdev avoids exposing a half-built datapath (ndo_open would call the ring/DMA stubs); discovering pdev via dev.parent avoids widening the shared vnic_priv/HAL just for Level 2. The self-test lives in the L2 HAL (with the ISR) and is deleted in Phase 9 when real completions drive the interrupt and the ISR calls napi_schedule(). Rejected: registering the netdev in Phase 8 with stubbed open (a loaded-but-broken vnic0 invites ip link set up -> NULL ring deref); building the DMA datapath in the same step (couples two hard-to-localise failure domains - the explicit Phase 8/9 split is the debuggability win); adding a pci_dev * to the shared vnic_priv (Level-1 dead weight, breaks the "device specifics live behind the HAL" rule, D-12).

D-29 Distinct netdev names per level (vnic_loop0 / vnic0) for coexistence

Decision: Level 1 (the in-kernel loopback engine) registers its netdev as vnic_loop0; Level 2 (the QEMU device) keeps the canonical vnic0. With the already-distinct module names (vnic-l1.ko / vnic-l2.ko), both drivers can be insmod'd in the same guest at once and targeted separately. The test harness derives IFACE from the module level (tests/lib.sh: *l2* -> vnic0, else vnic_loop0; still overridable) and scripts/setup-vnic.sh is interface/IP-parameterized, so the two levels can even run on separate subnets simultaneously. Why: both previously used vnic%d, so co-loading made the names load-order-dependent (vnic0/vnic1) - you could not tell which level was which. Giving the "real" device (L2) the clean vnic0 and the engine (L1) the self-describing vnic_loop0 makes each deterministic, and lets one guest A/B-test L1 vs L2 side by side. L1's rename is the only ripple and it is contained because the scripts/tests were already IFACE-parameterized. Rejected: keep both vnic%d (non-deterministic which-is-which when co-loaded); symmetric vnl1/vnl2 (renames L2 too and reads worse than keeping the canonical vnic0 for the real device); fixed vnic0/vnic1 by level (L2-alone would surprise by skipping to vnic1).

D-30 Level 2 backend - selectable tap vs internal loopback (default loopback)

Decision: the QEMU device carries a loopback property (default on). loopback=on keeps the Phase 9 internal IP-swap loopback (no backend). loopback=off + a -netdev (tap) makes it a real NIC backend: TX frames go out via qemu_send_packet, RX arrives via the receive() callback (pci_dma_write into the RX ring + RXAVAIL), with can_receive() + qemu_flush_queued_packets() (on the RDT refill doorbell) as the RX-full backpressure - the tap analogue of D-19. scripts/run-qemu.sh selects it with VNIC_TAP=<ifname> (-netdev tap,... -device vnic,loopback=off). Why: keeping loopback the default means a bare -device vnic stays in the validated, backend-free mode the L2 test suite needs - no host tap to set up just to run tests - while tap is opt-in for real guest<->host traffic. It is one device with one datapath core: the TX frame has two dispositions (loop into RX vs send to the peer) and RX has two sources (the engine vs the receive() callback), chosen by a single bool, so the Phase 9 path is byte-for-byte unchanged when loopback=on. Rejected: auto-detect by "is a peer attached" (fragile - qemu_new_nic always creates a client, so "real peer" is murky); default loopback=off (would break the bare-device test invocation and force a host tap for every test run); a separate device type (duplicate QOM/PCI/DMA/MSI code for no gain). Backend = tap specifically (not user-mode/socket) so the host can ping the guest and see frames with tcpdump -i tapN directly (D-25).

D-31 Tap host-side plumbing - routing+NAT on WSL2, bridge on a real host

Decision: how the host connects the tap (vnic-tap0) to the outside is a host-environment choice, separate from the device model (the QEMU/device side is identical either way - only what the host does with the tap interface differs). On the WSL2 host we use L3 routing + NAT: the tap gets its own subnet (192.168.20.0/24), and scripts/tap-nat.sh enables ip_forward + an outbound MASQUERADE out the WAN iface (D-30 / qemu-build.md 8.1). On a real Linux host with a physical NIC on a real LAN, the more correct approach is L2 bridging: create a bridge (br0), enslave both the physical NIC and the tap (-netdev bridge or ip link ... master br0); the guest then gets a real LAN IP via the LAN's DHCP, is directly reachable from the LAN, and needs no NAT. Why: WSL2's eth0 is itself a NAT'd virtual interface behind Windows, not a port onto a physical LAN - bridging the tap to it reaches no real LAN, so routing+NAT is the only sensible path there (at the cost of double-NAT and one-way reachability - inbound needs a DNAT, D-30). On a real host, bridging is cleaner and more faithful to "a real NIC": the guest is a first-class L2 member (own IP, pingable from any LAN host, no NAT asymmetry, accurate L2/broadcast behaviour), which is the natural production/demo setup. Both sit on the same tap - the difference is L3 (give the tap an IP, route/NAT) vs L2 (tap is a bridge port, no IP). Rejected / alternatives: bridging on WSL2 (pointless - the bridge would not reach the physical LAN, since WSL2's network is NAT'd); macvtap on a real host (a lighter bridge alternative, but host<->guest cannot talk directly by default - a worse fit when you also want host-side testing); slirp/user-mode (no real L2 at all - D-25/D-30). The repo automates only the WSL2 routing+NAT path (tap-nat.sh); a bridged real-host setup is documented as the alternative, not scripted here.

D-32 TX doorbell moderation - batch the doorbell over an xmit_more burst

Decision: vnic_start_xmit defers the VN_REG_TDT doorbell when netdev_xmit_more() reports more skbs queued behind this one, ringing it once per burst (or when the ring fills) instead of once per packet. A tx_coalesce module param (default on, runtime-writable) toggles it for A/B measurement. The change is in the shared core/ and is level-independent (Phase 12 perf work). Why: each writel(TDT) is an MMIO VM-exit on Level 2 (~thousands of cycles, vs a posted write on real hardware); a per-packet doorbell capped TX at ~350 Mbit/s while NAPI-batched RX reached ~3.77 Gbit/s (L-09). Batching amortizes the exit over a burst - the standard real-driver technique (background-knowledge topic 7.2). It is correct for both levels: the kernel guarantees the last skb of a dequeue burst clears xmit_more (so the deferred doorbell always fires), and a full ring forces a doorbell (so the device never stalls with un-rung descriptors); on Level 1 it just wakes the engine once per burst instead of per packet. Rejected: a count/timer-based deferral (adds latency + a flush timer; xmit_more is the kernel-native, zero-added-latency batch boundary); doing it only in the L2 HAL (the doorbell call lives in core/ and is identical across levels, D-12, so the optimization belongs there - neutral on L1). Interrupt moderation (ITR) is a separate lever - interrupt rate, not doorbell exits - left as backlog / Phase 12 (b).

D-33 Interrupt moderation (ITR) - device coalesces the MSI with a QEMU timer

Decision: the QEMU device honours VN_REG_ITR (microseconds - the moderation register it already advertises via VN_CAP_MODER). With ITR>0 it delivers at most one MSI per ITR microseconds using a QEMU_CLOCK_VIRTUAL timer: fire immediately if the gap since the last interrupt is already satisfied (low latency when idle), otherwise arm the timer to fire at the gap boundary and let further causes coalesce into it (they stay latched in ICR). ITR=0 keeps per-cause immediate delivery (the original behaviour). The driver programs ITR at open from an itr_usecs module param (default 0). This is the interrupt-side complement to D-32's TX doorbell batching (Phase 12 b). Why: under load the device raised an MSI per completion; ITR bounds the interrupt/NAPI rate, the standard real-NIC technique (background-knowledge topic 7.3). A virtual-clock timer is the natural QEMU mechanism, and the "fire-if-gap-else-schedule-and-coalesce" rule gives low latency when idle and real coalescing under load. Default off so the test suite is unchanged and Level 1 (whose simulated MSI ignores ITR) is unaffected - only the L2 device acts on it. Rejected: an "every Nth completion" counter (the rate then varies with traffic; time-based is what hardware does); on-by-default (would change test/L1 behaviour and add latency unasked); doing it in the driver (moderation is the device's job - the driver only programs the register, as on real hardware). Adaptive moderation (DIM) and MSI-X/multi-vector remain backlog (A3).
