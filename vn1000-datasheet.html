vNIC VN1000 1GbE Virtual Ethernet Controller - Software Developer Datasheet

vNIC VN1000 1GbE Virtual Ethernet Controller

Software Developer Datasheet

This document is the authoritative hardware specification for the VN1000 simulated NIC. It defines the register set, descriptor formats, and the control flow (programming sequences) for each operation. Both the device side (Level 1 engine / Level 2 QEMU model) and the driver side must conform to this spec; include/vnic_regs.h is derived from it.

0. Conventions

  • Byte order: little-endian (matching the x86 host)
  • Access width: all registers are 32-bit and must be accessed on a 4-byte aligned boundary
  • Access types:
  • RO read-only
  • RW read/write
  • WO write-only
  • RC read-to-clear (reading clears the register)
  • RW1S write-1-to-set (bits written as 1 are set; writing 0 has no effect)
  • RW1C write-1-to-clear (bits written as 1 are cleared)
  • Reserved fields: read value is undefined; must be written as 0
  • BAR0 is an MMIO region of 4 KiB (0x1000)

1. Overview

VN1000 is a single-queue, single-function virtual 1GbE Ethernet controller providing:

  • One TX descriptor ring and one RX descriptor ring (in host memory, accessed by the device via DMA)
  • An MMIO register interface (BAR0) controlling ring base / head / tail and interrupts
  • A single MSI interrupt vector with interrupt moderation (ITR)
  • Basic statistics counters

Completion is signalled by interrupts (throttled by ITR); the driver uses NAPI to achieve a hybrid of interrupt and polling. "Forced fixed-mode comparison" and "MSI-X / multi-queue" are advanced items, tracked in the TODO backlog (A2 / A3).

1.1 Block diagram

How the host sees the device: the CPU reaches RAM directly (load/store) and the NIC's registers through the BAR0 MMIO window (readl/writel - e.g. the TDT doorbell). The NIC's DMA engine is a bus master that reads/writes the descriptor rings + buffers in RAM itself; completions raise an MSI, and the transceiver (PHY) drives the wire. The register file (CSR) is shown in full; the BAR0 column is the same registers seen as an address-offset map.

2.1 Capabilities

Capabilities form a linked list starting at the Capabilities Pointer (0x34). Standard (PCI) capabilities live in 0x40-0xFF; PCIe extended capabilities live in the 0x100-0xFFF extended configuration space (PCIe only).

Standard capabilities (0x40-0xFF):

Cap ID Capability VN1000 Description
0x01 Power Management optional D0/D3 power states, PME
0x05 MSI yes (baseline) message-signalled interrupt, 1 vector
0x10 PCI Express PCIe endpoint identifies the device as PCIe; Device / Link / Slot capability, control, status
0x11 MSI-X backlog A3 table-based interrupts, multiple vectors

PCIe extended capabilities (0x100+):

Ext Cap ID Capability VN1000 Description
0x0001 AER advanced Advanced Error Reporting (correctable / uncorrectable error logging)
0x0003 Device Serial Number optional 64-bit globally-unique ID
0x0010 SR-IOV advanced Single-Root I/O Virtualization (virtual functions)

Baseline VN1000 exposes only the MSI capability. The PCI Express Capability (cap ID 0x10) — and therefore the extended config space / AER — exists only when the device is modelled as a true PCIe endpoint on a q35 PCIe bus. MSI-X is tracked in TODO backlog A3; the PCIe-endpoint upgrade (PCI Express Capability + MSI-X + extended caps) is the natural Level 2+ enhancement.

¹ TDH/RDH are device-owned while the device is enabled - the device advances them and the driver must treat them as read-only. They are writable only while the device is disabled (CTRL.EN = 0): the driver resets the head to 0 during ring setup, before it sets CTRL.EN. Writing a head while the device is enabled is undefined. (Same as Intel TDH/RDH: R/W, but software may write them only while the queue / DMA is disabled.) In this driver the head writes are vnic_up → TDH = 0 and vnic_rx_post → RDH = 0, both ordered before the single CTRL = EN | TXEN | RXEN write that enables the device; teardown writes CTRL = 0 first, so the next bring-up again sets the heads while disabled.

4. Register Definitions

4.1 ID (0x0000, RO)

Always returns 0x564E3130. The driver reads and validates this during probe; on mismatch it declines to bind.

4.2 VERSION (0x0004, RO)

bits[31:16] = major, bits[15:0] = minor. This revision = 0x0001_0000.

4.3 CAP (0x0008, RO)

Bit Name Description
0 CAP_MSI MSI supported
1 CAP_MODER Interrupt moderation (ITR) supported
2 CAP_POLL Forced polling mode (reserved, see advanced A2; this revision = 0)
3 CAP_CSUM Checksum offload (reserved; this revision = 0)
31:4 reserved

4.4 CTRL (0x0010, RW)

Bit Name Description
0 EN Device enable
1 RST Software reset; cleared by the device when complete (self-clear = done)
2 TXEN Transmit enable
3 RXEN Receive enable
4 PROMISC Promiscuous mode
31:5 reserved write 0

4.5 STATUS (0x0014, RO)

Bit Name Description
0 RUNNING Device enabled and initialized
1 LINK_UP Link is up
31:2 reserved

STATUS is driven by the device in response to CTRL - the driver never writes it (it is RO). Writing CTRL.EN = 1 makes the device assert RUNNING and LINK_UP; clearing EN deasserts both. This is the control/status handshake: the driver writes CTRL.EN, then reads back STATUS.RUNNING/LINK_UP to confirm the device (and "link") came up (bring-up step 9). CTRL.RST additionally restores the power-on register state.

Reset completion has no separate status bit; the self-clearing CTRL.RST is the sole completion signal.

4.6 MAC_LO / MAC_HI (0x0018 / 0x001C, RW)

  • MAC_LO bits[31:0] = MAC[0..3]
  • MAC_HI bits[15:0] = MAC[4..5], bit31 = AV (address valid)
  • Recommended write order: MAC_LO first, then MAC_HI, set AV last

4.7 MAXFRM (0x0020, RW)

Maximum Ethernet frame size in bytes the device will transmit and receive, including the 14-byte header and 4-byte FCS. Default 1518 (MTU 1500); the driver sets MAXFRM = MTU + 18. Posted RX buffers must be at least MAXFRM bytes. The device drops received frames larger than MAXFRM (increments STAT_RXDROP, may set ICR.ERR). See section 6.7 for the change-MTU sequence.

In this single-buffer-per-packet model MAXFRM also serves as the RX buffer size (cf. Intel RCTL.BSIZE). Splitting it into a separate RX buffer size, max-frame length, and long-packet enable (cf. Intel RLPML / LPE) together with multi-descriptor (scatter) receive is tracked as advanced item A4 in TODO.

4.8 TX/RX Ring registers

  • TDBAL/TDBAH, RDBAL/RDBAH: ring base physical address, must be 256-byte aligned
  • TDLEN/RDLEN: ring total length in bytes, not the length of a single descriptor. A single descriptor is fixed at 16 bytes (struct vn_desc), so the number of descriptors in the ring is N = TDLEN / 16 — both the device and the driver derive the ring size this way; there is no separate "count" register (same convention as Intel TDLEN/RDLEN). N must be a power of two, range 8..4096
  • TDH/RDH: the device's head indices - device-owned while enabled (the device advances them; the driver then treats them as read-only). Writable by the driver only while the device is disabled (CTRL.EN = 0): it resets the head to 0 during ring setup, before CTRL.EN (vnic_up/vnic_rx_post). Writing a head while enabled is undefined; see the ¹ note under the register map
  • TDT/RDT: maintained by the driver; a write acts as a doorbell, notifying the device of new work or newly available buffers
  • empty: HEAD == TAIL; full: (TAIL + 1) % N == HEAD

4.9 Interrupt cause bits (shared by ICR / ICS / IMS / IMC)

Bit Name Description
0 TXDONE One or more TX descriptors completed
1 RXAVAIL One or more RX packets available
2 RXEMPTY No free RX buffers (overrun / drop)
3 ERR Error condition
31:4 reserved
  • ICR (RC): reading returns the currently pending causes and clears them
  • ICS (WO): bits written as 1 raise the corresponding cause - the software counterpart of ICR (its set/write path). A 1 here has the same effect as the hardware event and asserts MSI if that cause is also enabled in IMS
  • IMS (RW1S): bits written as 1 enable the corresponding interrupt cause
  • IMC (RW1C): bits written as 1 disable the corresponding interrupt cause
  • MSI is asserted only for causes that are both enabled (set in IMS) and pending
  • The interrupt mask is internally a single logical register; IMS/IMC are its set/clear write paths, giving atomic updates and avoiding read-modify-write races

More on ICS. It lets software trigger an interrupt without a real event. On physical (Intel-style) NICs ICS is used to self-test the MSI/ISR wiring at probe, to force a cause (e.g. a link-status interrupt), and for netpoll / ndo_poll_controller (netconsole, kgdb-over- net); the MSI-X form is EICS. It is not part of the steady TX/RX datapath - there, the device raises completions by setting ICR itself. VN1000 splits this by level:

  • Level 2 (QEMU device): the device sets ICR directly on TX/RX completion, exactly like real hardware; the ICS register is implemented but its write path is not used by the datapath.
  • Level 1 (in-kernel engine): the engine talks to its fake device through the same MMIO accessors, so it deliberately raises completions by writing ICS - this keeps the L1 interrupt path faithful (raise cause -> MSI -> ISR reads ICR) instead of shortcutting to napi_schedule.

ISR acknowledge sequence (read-to-clear). When the MSI fires, the driver's ISR (top half) acknowledges the interrupt by reading ICR: because ICR is read-to-clear (RC), that single read returns the currently pending cause bits and atomically clears them to 0. The read itself is the ack - no write is needed, and nothing is ever written to ICR to clear it (ICS is the set/raise path, not a clear path). The ISR then masks the just-taken cause(s) with a write to IMC so a busy device cannot re-interrupt while the poll runs, and calls napi_schedule(). The NAPI poll drains the ring and, on napi_complete_done(), re-arms the cause(s) with a write to IMS. The steady-state sequence:

  1. MSI asserted - a cause is both pending in ICR and enabled in IMS
  2. ISR: cause = read(ICR) - read-to-clear, this read is the ack (clears the latched cause)
  3. ISR: write(IMC, cause) - mask that cause while polling, then napi_schedule()
  4. poll: drain descriptors; on napi_complete_done() -> write(IMS, cause) to unmask / re-arm

The ack (ICR read) and the mask (IMC write) are two distinct register actions: the ICR read clears only the pending cause, never IMS; masking is the separate explicit IMC write. The TX/RX flow diagrams collapse the pair into one label, "ack + mask IRQ in CSR" (steps D2 / C2). Both levels follow this: the Level 1 simulated MSI handler vnic_l1_msi() in driver/hal/vnic_hal_l1.c reads ICR, writes IMC, then schedules NAPI; the Level 2 real MSI ISR does the same against the QEMU device, whose ICR read path is implemented as read-to-clear.

4.10 ITR (0x0310, RW)

bits[15:0] = minimum interval between interrupts, in microseconds. 0 = no moderation. Has no effect when CAP_MODER is 0.

4.11 Statistics registers (0x0400..0x0414, RO)

32-bit counters, cleared on software reset (CTRL.RST).

5. Descriptor Format

TX and RX share a 16-byte structure; the field semantics differ by direction.

struct vn_desc {
    u64 addr;    /* 0x00  buffer address (Level 2 DMA / Level 1 token) */
    u32 len;     /* 0x08  TX: packet length; RX (on completion): bytes received */
    u16 flags;   /* 0x0C  TX: command; RX (on completion): result */
    u16 status;  /* 0x0E  DD ownership bit: driver clears -> 0, device sets -> 1 */
};               /* sizeof = 16 */

5.1 TX flags (driver-written, VN_TXF_*)

Bit Name Description
0 EOP End of packet (always set for a single-buffer packet)
1 RS Report status; device sets DD on completion and may raise an interrupt
2 IC Insert checksum (reserved; ignored in this revision)

5.2 RX flags (device-written, VN_RXF_*)

Bit Name Description
0 EOP End of packet
1 ERR Receive error

5.3 status - the DD ownership / done bit (VN_DESC_*)

Bit 0 is DD (Descriptor Done); bits 31:1 are reserved.

DD Owner Meaning
0 device submitted (TX) / posted (RX) - the device may process the descriptor; the driver must not touch it or its buffer
1 driver done - the device has finished (TX sent, or RX frame written with len valid); the driver may reclaim / receive it

DD is written by both sides (not device-only): the driver clears it to 0 when handing a descriptor to the device (TX submit; RX post / refill), and the device sets it to 1 when handing it back (TX completion; RX delivery). That two-way write is the ownership handshake - see 5.4 and the paired memory barriers in background-knowledge topic 4.

5.4 Ownership model

  • Descriptor ownership is expressed by status.DD
  • The driver clears DD before submitting; the device sets DD on completion
  • The driver scans from its own next_to_clean, reclaiming descriptors with DD = 1

5.5 Ring buffers and control flow (TX and RX)

Both rings are contiguous, circular arrays of N = *DLEN / 16 descriptors based at *DBAL/*DBAH, with a head (*DH) and tail (*DT). Each descriptor's status.DD carries ownership:

DD owner meaning
0 device submitted (TX) or posted-empty (RX); in progress - the driver must not touch it
1 driver completed - a sent TX descriptor, or an RX descriptor now holding a received frame

The only asymmetry between the rings is who produces: on TX the driver fills buffers for the device to send; on RX the device fills buffers (that the driver pre-posted) with received frames. Each descriptor points to a DMA buffer in host memory:

  desc[i] = [ addr(u64) | len | flags | status(DD) ]  -->  [ buffer in host RAM ]

Hardware pointers vs the driver's cursors. *DH / *DT are the device's ring pointers (MMIO registers). The driver also keeps two software cursors per ring in struct vnic_ring (in RAM - the device never sees them):

  • next_to_use - the driver's producer cursor: the next descriptor it will fill (TX) or post (RX).
  • next_to_clean - the driver's consumer / reclaim cursor: the next descriptor it will clean. "Clean" means - on TX, the device has finished sending, so free the completed skb; on RX, a frame has arrived, so take it up the stack and refill the slot. The driver advances next_to_clean only across descriptors whose DD == 1, scanning the in-memory DD bits rather than reading *DH over MMIO on every completion (a cheap RAM read vs an expensive register read). It is the index the cleanup loops below start from.

Why a separate cursor, not just *DH/*DT? *DH/*DT describe the device's current physical occupancy of the ring (how far it has consumed/produced) - they are the hardware's view. But when the driver takes a completion interrupt it does not necessarily drain all pending descriptors in one pass: NAPI runs under a budget (it may stop part-way and reschedule), and TX cleanup likewise processes only what is done so far. So the driver needs its own variables to remember how far it has got - next_to_clean (where to resume reclaiming/receiving) and next_to_use (where it is filling, which may run ahead of the last *DT it published so it can batch the doorbell). There is no hardware register for "how far the driver has processed", and reading *DH each time would cost an MMIO read; tracking it in software (advanced by the in-memory DD bit) is both necessary and cheaper.

TX ring - driver produces, device consumes

TDH (device consumer) and TDT (driver producer) split the ring. At open the ring is empty: next_to_use = next_to_clean = 0 and TDH = TDT = 0 - nothing is in flight; the driver fills forward from next_to_use (NTU) and the device follows at TDH. Example: the driver has filled 0..4 (next_to_use = TDT = 5); the device has sent 0..1 (it set their DD = 1 and advanced TDH = 2) and is now sending 2..4; the driver has not reclaimed yet (next_to_clean = 0):

    0    1    2    3    4    5    6    7      (index; N = TDLEN / 16, wraps 7->0)
  +----+----+----+----+----+----+----+----+
  | 1  | 1  | 0  | 0  | 0  | -  | -  | -  |   <- DD bit per slot
  +----+----+----+----+----+----+----+----+
    ^         ^              ^
   NTC       TDH          TDT/NTU
  [NTC, TDH) DD=1 "SENT":    completed - driver to reclaim (free the skb)
  [TDH, TDT) DD=0 "SENDING": submitted - device-owned, being sent on the wire
  rest             "free":   DD don't-care - driver fills here, clearing DD as it goes
  NTU (next_to_use): driver producer cursor = next slot to fill; here = TDT, but
    may run ahead of the published TDT when the doorbell is batched (xmit_more / D-32)
  NTC (next_to_clean): driver reclaim cursor = next DD = 1 slot to free

TX control flow (note where DD is cleared / set / checked): 1. Driver submit (per packet): fill desc[next_to_use] (not desc[TDT]) - clear its DD, write addr/len/flags (EOP, RS) - then dma_wmb() and advance next_to_use = (next_to_use + 1) % N. The doorbell writes the TDT register = next_to_use; with tx_coalesce (D-32) it is batched - rung once at the end of an xmit_more burst (or when the ring fills), not per packet - so next_to_use may run ahead of the last published TDT. 2. Device: while TDH != TDT - read desc[TDH], DMA-read the buffer and transmit, write status and set DD = 1 last, TDH++, raise TXDONE (if RS). 3. Driver reclaim (NAPI tx_clean): from next_to_clean, read desc[ntc].DD - if 0, stop (not done yet); if 1, dma_rmb() (this barrier pairs with that DD-flag read - it orders "saw DD == 1" before the following reads, so they cannot be hoisted ahead of the flag), then unmap/free bufs[ntc], next_to_clean++, wake the queue if it was stopped. - empty: TDH == TDT full: (TDT + 1) % N == TDH (one slot kept unused)

RX ring - device produces, driver consumes

At open the driver pre-posts an empty buffer into every usable slot (Model A) with DD = 0 and writes RDT to publish them. Concretely at open: next_to_clean = 0, RDH = 0, RDT = count - 1 (every usable slot posted, DD = 0, one slot kept unused); next_to_use is unused on RX, because the device is the producer (it advances RDH, the driver does not fill these slots). RDH is the device's producer index (device-owned at runtime; the driver writes it only to reset it to 0 here at open). Example: the device wrote 3 frames (so RDH = 3) and the driver has not polled yet (next_to_clean = 0):

    0    1    2    3    4    5    6    7      (index; N = RDLEN / 16, wraps 7->0)
  +----+----+----+----+----+----+----+----+
  | 1  | 1  | 1  | 0  | 0  | 0  | 0  | -  |   <- DD bit per slot
  +----+----+----+----+----+----+----+----+
    ^              ^                   ^
   NTC            RDH                 RDT
  [NTC, RDH) DD=1 "FULL":  a received frame - driver to process (up the stack)
  [RDH, RDT) DD=0 "EMPTY": a posted buffer - device-owned, awaiting a frame
  last slot       "unused": kept unused to tell empty from full

RX control flow. Note that next_to_use is unused on RX - next_to_clean is the only driver cursor, and since each consumed slot is refilled immediately it also doubles as the refill point written to RDT: 1. Driver post / refill (at open via vnic_rx_post; and in rx_clean right after consuming a slot): write desc[i].addr = a fresh empty buffer and len = 0, dma_wmb(), then clear DD (status = 0) to hand the slot back to the device - publish the buffer before the flag. The RDT doorbell is written once at the end of the poll, = next_to_clean (the refill point) - not per slot; it tells the device how far buffers are available and resumes it if it had stalled. 2. Device on receive: DMA-write the frame into desc[RDH].addr, write len (+ RX flags), set DD = 1 last, RDH++, raise RXAVAIL. With no posted buffer (RDH would reach RDT) it raises RXEMPTY and drops (backpressure). 3. Driver consume (NAPI rx_clean): from next_to_clean, read desc[ntc].DD - if 0, stop; if 1, dma_rmb() (pairs with that DD read, so the next reads see the freshly DMA-written data, not stale bytes), then read len, hand bufs[ntc] up the stack (napi_gro_receive), refill the slot (step 1), and next_to_clean++ (up to the NAPI budget). - nothing to process: next_to_clean == RDH

The DD handshake (both directions; D-21)

DD is the ownership flag and the buffer/len is the payload; the two must not be observed out of order. The side that hands ownership over by setting DD writes the payload first, then a write barrier, then DD; the side that takes ownership waits for DD == 1, then a read barrier, then reads the payload:

  producer:  write addr/len/data  ->  dma_wmb()  ->  set DD = 1
  consumer:  read DD == 1          ->  dma_rmb()  ->  read len/data

On TX the device is the producer of the completion; on RX the device is the producer of the frame. (On Level 1 the in-kernel engine and the driver share one address space, so these are smp_wmb / smp_rmb rather than the DMA barriers - see background-knowledge topic 4.)

6. Theory of Operation (control method per operation)

6.1 Initialization

  1. Read ID, verify = 0x564E3130; read VERSION / CAP
  2. Software reset: write CTRL.RST = 1, poll until CTRL.RST self-clears (indicating reset complete)
  3. Set MAC: write MAC_LO, MAC_HI, then set MAC_HI.AV
  4. Allocate TX/RX rings (contiguous DMA memory, 256-byte aligned, zeroed)
  5. TX: write TDBAL/TDBAH/TDLEN, set TDH = TDT = 0
  6. RX: fill each descriptor with an available buffer, write RDBAL/RDBAH/RDLEN, RDH = 0, RDT = N-1
  7. Configure interrupts: IMS = TXDONE | RXAVAIL | ERR; set ITR if desired
  8. Enable: CTRL = EN | TXEN | RXEN
  9. Confirm STATUS.RUNNING = 1 and LINK_UP = 1

6.2 Transmit

  1. The driver places the packet in a buffer and fills desc[TDT]: addr, len, flags = EOP | RS, clears status.DD
  2. Write barrier (wmb) so descriptor contents are visible before the doorbell
  3. Advance TDT and write TDT (doorbell)
  4. Device: consumes from TDH to TDT, DMA-reads the buffer and transmits, sets status.DD, advances TDH; if RS is set, sets ICR.TXDONE and raises MSI (subject to IMS / ITR)
  5. The driver cleans TX in NAPI / ISR: scans DD = 1 from next_to_clean, frees the buffer / skb, advances next_to_clean
  6. When the ring is full the driver stops the queue (netif_stop_queue) and resumes after completions

6.3 Receive

  1. The driver posts available buffers: fill desc[i].addr, clear DD; set RDT to the last available index
  2. On packet arrival the device writes into the buffer at RDH, sets len, flags = EOP, status.DD, advances RDH; sets ICR.RXAVAIL and raises MSI (subject to IMS / ITR)
  3. The driver processes (by polling / after interrupt): scans DD = 1 from next_to_clean, builds an skb and hands it to the network stack, allocates a new buffer, clears DD, advances RDT (doorbell to repost)
  4. With no free buffers the device sets ICR.RXEMPTY, drops the packet, and increments STAT_RXDROP

6.4 Interrupt handling

  1. After MSI fires, the ISR reads ICR (clear-on-read) to obtain the pending causes
  2. For RXAVAIL or TXDONE: mask those causes with IMC and call napi_schedule()
  3. NAPI poll cleans TX / receives RX; when work done is below budget it calls napi_complete_done() and re-enables interrupts via IMS; after re-enabling, re-check the ring and reschedule if work remains (to avoid the NAPI-completion lost-interrupt race)
  4. ITR bounds the minimum inter-interrupt interval to reduce overhead at high packet rates

6.5 Completion notification

  • Baseline: the device signals completion by interrupt (throttled by ITR); the driver uses NAPI to switch between interrupt and polling (interrupt arrives -> IMC mask and poll -> IMS re-enable when drained), which is itself a hybrid of interrupt and polling.
  • "Forced fixed mode (pure interrupt / pure polling / adaptive) + three-mode benchmark comparison" is an advanced item (TODO backlog A2, which introduces the MODE register).

6.6 Teardown / Reset

  • Pause: CTRL.EN = 0 (stops TX/RX)
  • Full reset: CTRL.RST = 1, resets ring pointers and statistics

6.7 Change MTU

Changing the MTU changes the maximum frame size and therefore the required RX buffer size, so the RX path must be reconfigured. Recommended driver sequence (ndo_change_mtu): 1. Validate the requested MTU is within [VN_MTU_MIN, VN_MTU_MAX] (68 .. 9000) 2. If the interface is down, record the new MTU and return; it takes effect on the next open 3. If up, quiesce both sides (they are complementary, not alternatives): - Engine side (stops DMA): clear CTRL.RXEN and CTRL.TXEN. This halts the NIC's RX/TX DMA engines and makes the buffers safe to free - Driver/stack side (stops concurrency): stop the TX queue with netif_tx_disable() (the stack then issues no new ndo_start_xmit), call napi_disable() to stop the poll, mask interrupts via IMC, and synchronize_irq() to let any in-flight ISR finish - Note: stopping the queue and napi_disable() alone do not stop RX DMA — the RX engine keeps writing posted buffers until RXEN is cleared 4. Reclaim and free the existing RX buffers and reset the RX descriptors 5. Compute MAXFRM = MTU + 18 and write MAXFRM 6. Reallocate RX buffers of at least MAXFRM bytes, repost them, and reset RDH = 0, RDT = N-1 7. Re-enable: set CTRL.RXEN / CTRL.TXEN, napi_enable, re-enable interrupts via IMS, and wake the queue 8. Update the interface MTU

Notes: - TX must also be able to send frames up to the new MAXFRM. Whether TX buffers are reallocated depends on the TX model: - Zero-copy / skb model (Level 2 netdev): TX descriptors point directly at each skb's own data (DMA-mapped per packet), so there is no pre-allocated TX buffer pool to resize; the stack simply produces larger skbs. No TX buffer reallocation is required. - Copy model (Level 1 char device): if the driver copies payloads into pre-allocated fixed-size TX buffers, those are sized to the MTU and must be reallocated together with the RX buffers (steps 4 and 6). - RX always requires reallocation, because buffers are posted before packets arrive and must be pre-sized for the new maximum frame - Jumbo frames (MTU up to 9000) require buffers up to ~9 KiB; ensure the buffer allocator can satisfy this - A new MTU only takes effect after this reconfiguration; in-flight RX descriptors sized for the old MTU must be drained or discarded first

7. Programming Flow Pseudocode

init():
    assert read32(ID) == 0x564E3130
    write32(CTRL, RST); wait until (read32(CTRL) & RST) == 0
    write32(MAC_LO, mac_lo); write32(MAC_HI, mac_hi | AV)
    alloc tx_ring/rx_ring (256-byte aligned, N power-of-two)
    write32(TDBAL/TDBAH/TDLEN); write32(TDT, 0)
    for i in 0..N-1: rx_desc[i].addr = buf[i]; rx_desc[i].status = 0
    write32(RDBAL/RDBAH/RDLEN); write32(RDT, N-1)
    write32(IMS, TXDONE|RXAVAIL|ERR); write32(ITR, itr_us)
    write32(CTRL, EN|TXEN|RXEN)

xmit(pkt):
    if ring_full(): stop_queue(); return BUSY
    d = tx_desc[tdt]
    d.addr = dma(pkt); d.len = pkt.len; d.flags = EOP|RS; d.status = 0
    wmb()
    tdt = (tdt + 1) % N; write32(TDT, tdt)

isr():
    cause = read32(ICR)            # read-to-clear
    if cause & (RXAVAIL|TXDONE):
        write32(IMC, RXAVAIL|TXDONE)   # mask while polling
        napi_schedule()

poll(budget):
    clean_tx()                     # free desc with DD set
    n = clean_rx(budget)           # build skb for desc with DD set, refill, bump RDT
    if n < budget:
        napi_complete()
        write32(IMS, RXAVAIL|TXDONE)   # re-enable, then re-check ring
    return n

8. Level 1 vs Level 2 Implementation Mapping

Level 1 and Level 2 expose the same netdev driver — a single vnic0 interface with the full real-NIC structure (net_device ops + NAPI + TX/RX descriptor rings + interrupt). Only the "device" behind the rings differs: an in-kernel engine (Level 1) vs the QEMU model (Level 2). Moving from Level 1 to Level 2 is therefore mostly swapping the engine for QEMU, not rewriting the driver.

Spec element Level 1 (kernel-only engine) Level 2 (QEMU PCI device)
Upper interface netdev vnic0 (real-NIC structure) netdev vnic0 (same driver)
Register access module struct fields (fake registers) real MMIO, readl/writel <-> QEMU read/write callback
addr kernel buffer pointer / token guest physical address (DMA)
Doorbell (TDT/RDT write) wakes the engine kthread MMIO write triggers QEMU handling
Descriptor DMA direct memory access pci_dma_read / pci_dma_write
Interrupt workqueue simulation MSI (msi_notify)
TX path ndo_start_xmit -> TX ring -> engine ndo_start_xmit -> TX ring -> QEMU
RX source engine loops the frame back into the RX ring QEMU receive() from backend / host
Reachability ICMP + TCP/UDP via internal loopback real ping + TCP/UDP to the host

The single VN1000 engine acts as an internal loopback (a mode real NICs provide for self-test): a transmitted frame is looped back into the RX ring with its source/destination IP swapped (L4 ports kept) — snull-inspired — and completed via the simulated interrupt + NAPI, so vnic0 is reachable without external hardware or a second interface.

  • Swapping only the IP addresses (not the ports) re-orders the same 16-bit words, so the IP and TCP/UDP checksums stay valid — no recomputation needed.
  • Setup: give vnic0 a local IP A; the fictional peer B is another address on the same subnet, not assigned to any interface. Traffic to B is non-local so it routes out vnic0; the engine swaps it to arrive at A.
  • ICMP: ping B works (the kernel answers).
  • TCP/UDP: run the server bound to 0.0.0.0 (or A) and point the client at B (e.g. iperf -s then iperf -c B). The IP-swap delivers the client's B-targeted packets to the server on A, and replies back. The client must target B, not A / localhost (those route via lo and bypass vnic0).
  • L2 / ARP: the kernel must resolve B's MAC before sending. Since B does not exist, either (a) add a static neighbor entry (ip neigh replace B lladdr <fake-mac> dev vnic0 nud permanent) so no ARP is sent, or (b) have the engine answer ARP requests for B (fabricate the reply). When looping a frame back, the engine sets the RX frame's destination MAC to vnic0's own MAC (so eth_type_trans marks it PACKET_HOST and delivers it) and the source MAC to B's (fake) MAC.
  • rp_filter: keep B on vnic0's subnet so strict reverse-path filtering passes (otherwise set vnic0's rp_filter to loose / 0). The IP swap also keeps the looped-back source non-local, avoiding a martian-source drop.

Both the device and the driver share include/vnic_regs.h, ensuring offset / bit definitions stay consistent.
