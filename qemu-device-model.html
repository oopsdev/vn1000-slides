QEMU Device Model (background for Level 2)

Level 2 replaces the in-kernel fake engine with a real emulated PCIe device inside QEMU (qemu-device/vnic.c) talking to the same netdev driver over real MMIO / DMA / MSI. This note answers: how QEMU emulates the device, and for each guest action (config access, MMIO, DMA, interrupt) which QEMU callback runs and how the guest side is wired. Every QEMU hook here has a Level 1 analogue we already built, so the mental model carries over.

API names match QEMU's device API (the version pinned to the guest's current QEMU - currently 6.2; see qemu-build.md step 1). Guest side is the in-tree Linux PCI/DMA/IRQ API.

How to construct a QEMU PCIe NIC device

The anatomy of a QEMU emulated PCIe NIC, in the order you build it. Each step points to the deep-dive section below and to the real implementation in qemu-device/vnic.c (built up across Phases 8-10). The minimum for "appears in lspci" is steps 1-2; a working datapath adds 3-5; a real backend adds 6.

  1. Define the QOM type (section 1). An instance struct whose first member is PCIDevice, a TypeInfo with .parent = TYPE_PCI_DEVICE, and a class_init that sets the PCI identity (vendor_id/device_id/class_id = PCI_CLASS_NETWORK_ETHERNET) and the realize/exit hooks. Register with type_init(). From the IDs QEMU synthesises the PCI config space, so the guest can enumerate it. c struct VNICState { PCIDevice pdev; MemoryRegion mmio; uint32_t regs[...]; /* + backend */ }; static void vnic_class_init(ObjectClass *klass, void *data) { PCIDeviceClass *k = PCI_DEVICE_CLASS(klass); k->realize = vnic_realize; k->exit = vnic_exit; k->vendor_id = ...; k->device_id = ...; k->class_id = PCI_CLASS_NETWORK_ETHERNET; }
  2. realize() is the constructor (sections 1, 3, 5). It runs when the device is plugged in: create the register BAR as a MemoryRegion and pci_register_bar() it; msi_init() the interrupt; (real NIC) create the backend; then reset the register file. exit() undoes it.
  3. BAR MMIO registers (section 3). Give the BAR MemoryRegionOps { .read, .write } with a fixed access size; the callbacks are the device's register logic - read-only IDs, RW config, the TDT/RDT doorbells, read-to-clear ICR, RW1S/RW1C IMS/IMC. A guest readl/writel traps here at the physical BAR address.
  4. DMA to/from guest memory (section 4). The device autonomously reaches the buffers the driver prepared with pci_dma_read() / pci_dma_write() at the bus addresses the driver programmed (descriptor ring bases, vn_desc.addr); needs the guest to have set bus-master. No guest callback fires for DMA - the handshake is the descriptor DD bit (+ barriers).
  5. Interrupt (section 5). msi_init() in realize; raise a completion with msi_notify() (gated by the device's own ICR & IMS mask). Physically an MSI is just a memory write to the LAPIC address, which the APIC turns into a vCPU vector.
  6. NIC backend - only for a real NIC (section 6 + D-30). To move frames off the box, add NICConf conf + NICState *nic, a NetClientInfo { .can_receive, .receive }, and create it with qemu_new_nic() in realize. TX: send a frame out with qemu_send_packet(). RX: the backend calls your .receive() - DMA the frame into the RX ring and raise the interrupt; .can_receive() + qemu_flush_queued_packets() are the RX-flow-control. DEFINE_NIC_PROPERTIES exposes mac=/netdev=. (The VN1000 keeps an internal-loopback mode too - loopback=on, the default - so it needs no backend to run the test suite.)
  7. Properties + reset. DEFINE_PROP_* + device_class_set_props() for -device vnic,<opt>=...; a reset hook (dc->reset) to restore the power-on register state on machine reset / CTRL.RST.
  8. Build it into QEMU. Drop the file in hw/net/, add a one-line meson + Kconfig hook, and build a self-built QEMU (no out-of-tree device ABI) - see qemu-build.md.

The rest of this note is the per-action reference: which callback each guest operation hits, and its Level 1 analogue.

1. How QEMU emulates a PCIe device, and how the guest probes it

QEMU side. A device is a QOM (QEMU Object Model) type whose parent is TYPE_PCI_DEVICE and whose instance struct embeds a PCIDevice:

struct VNICState { PCIDevice pdev; MemoryRegion mmio; ... };

static void vnic_class_init(ObjectClass *klass, void *data)
{
    PCIDeviceClass *k = PCI_DEVICE_CLASS(klass);
    k->realize   = vnic_realize;          /* called when the device is plugged in */
    k->vendor_id = PCI_VENDOR_ID_...;     /* shows up at config offset 0x00 */
    k->device_id = 0x....;
    k->class_id  = PCI_CLASS_NETWORK_ETHERNET;
}
static const TypeInfo vnic_info = {
    .name = "vnic", .parent = TYPE_PCI_DEVICE,
    .instance_size = sizeof(VNICState), .class_init = vnic_class_init,
    .interfaces = (InterfaceInfo[]){ { INTERFACE_PCIE_DEVICE }, { } },  /* PCIe endpoint */
};

vnic_realize() sets up the BARs (section 3), MSI (section 5) and the NIC backend. From the ID/BAR/capability registrations QEMU synthesises the device's PCI config space (256 B legacy, 4 KiB extended for PCIe). Plugging it onto a q35 machine's PCIe bus makes it a PCIe endpoint.

Guest side / probe. On boot the guest's PCI subsystem enumerates the bus by reading config space: vendor/device ID at offset 0, then it sizes each BAR (write all-ones, read back the size mask), assigns it a guest-physical address, and sets the Command register bits (memory decode, bus-master). A driver registers an ID table; when it matches, .probe runs:

static const struct pci_device_id vnic_ids[] = { { PCI_DEVICE(VENDOR, DEVICE) }, { 0 } };
static struct pci_driver vnic_pci = { .name="vnic", .id_table=vnic_ids, .probe=vnic_probe, ... };
/* probe: pci_enable_device -> pci_request_regions -> pci_iomap(BAR0) -> dma_set_mask
 *        -> pci_alloc_irq_vectors(MSI) + request_irq -> alloc_etherdev/register_netdev */

L1 analogue: vnic_hal_init() allocated the fake register block and "appeared"; here QEMU + the PCI bus make the device appear and the kernel calls probe().

1.1 What this looks like on the host (one process, vCPU threads)

A useful mental check: on the host you see one QEMU process (a PID), but there is no separate "guest PID" - the guest is not another host process, it is the QEMU process. Each guest vCPU is a thread inside that process, not a process of its own.

ps aux | grep qemu-system-x86_64     # one PID, e.g. 12345
ps -T -p 12345                       # its threads (or: top -H -p 12345, htop thread view)
ls /proc/12345/task/                 # one sub-dir per thread (TID)

ps -T shows roughly:

  PID  SPID  CMD
12345 12345  qemu-system-x86_64      <- QEMU main loop (event loop / BQL)
12345 12346  CPU 0/KVM               <- vCPU 0 thread
12345 12347  CPU 1/KVM               <- vCPU 1 thread   (we boot -smp 2)
12345 12350  ...                     <- worker threads (AIO / virtio / etc.)

Why it is shaped this way:

  • One process, vCPUs as threads. Each vCPU thread runs ioctl(KVM_RUN) on its vCPU fd and enters guest code on a host CPU (VMX non-root). To the host scheduler it is just a thread burning CPU - so QEMU's CPU% rises with the guest's load.
  • Guest RAM is host memory. The guest's -m 2048 is an anonymous mmap in QEMU's address space - visible as one large mapping in /proc/<pid>/maps (smaps).
  • Our device runs in that process. A guest writel(TDT) traps (VM-exit) -> the host KVM module -> back out to QEMU userspace on a vCPU thread, which runs vnic_mmio_write (under the BQL). pci_dma_* then reads/writes the guest-RAM mapping directly, and msi_notify injects the vector back via KVM.

So the slide/architecture bands map onto real OS objects: "HOST userspace - QEMU process" is that one PID; "GUEST (VM)" is not another PID but the vCPU threads + the guest-RAM mapping of the same process.

WSL2 note: WSL2 is itself a lightweight Hyper-V VM, so "host" here means the Linux inside WSL2; QEMU runs there and uses nested KVM to run the guest. The process/thread picture above is unchanged.

2. Config-space access -> which QEMU callback

Guest side. The kernel reads/writes config space with pci_read_config_dword() etc. On x86 those go through the host-bridge mechanism - legacy CF8/CFC I/O ports, or MMCONFIG/ECAM for PCIe extended config. Most config writes are done by the PCI core, not our driver: BAR sizing, the Command register, and enabling MSI.

QEMU callback. The host bridge routes the trapped access to the device's config accessors in PCIDeviceClass:

uint32_t (*config_read)(PCIDevice *d, uint32_t addr, int len);
void     (*config_write)(PCIDevice *d, uint32_t addr, uint32_t val, int len);

The defaults are pci_default_read_config / pci_default_write_config, which operate on d->config[] honouring the wmask/w1cmask tables. pci_default_write_config already implements the important side effects: BAR programming and Command-register changes call pci_update_mappings() to map/unmap the BAR MemoryRegions into guest address space, and MSI/MSI-X capability writes are handled by msi_write_config() (this is where the guest's "enable MSI + here is my address/data" lands). A device overrides config_write only to react to its own config side effects (then usually calls pci_default_write_config first).

Takeaway: for VN1000 we likely do not override config access - the defaults handle BAR mapping and MSI-enable. Our registers live in the BAR (MMIO), not config space.

3. MMIO (BAR) access -> which QEMU callback

QEMU side. In realize the device creates a MemoryRegion with I/O ops and registers it as a BAR:

static const MemoryRegionOps vnic_mmio_ops = {
    .read  = vnic_mmio_read,            /* <- guest load from the BAR lands here */
    .write = vnic_mmio_write,           /* <- guest store to the BAR lands here  */
    .endianness = DEVICE_LITTLE_ENDIAN,
    .valid.min_access_size = 4, .valid.max_access_size = 4,
};
memory_region_init_io(&s->mmio, OBJECT(s), &vnic_mmio_ops, s, "vnic-mmio", VN_BAR0_SIZE);
pci_register_bar(&s->pdev, 0, PCI_BASE_ADDRESS_SPACE_MEMORY, &s->mmio);

After the guest assigns the BAR and enables memory decoding, that BAR is mapped into guest-physical space but is not backed by RAM, so every load/store traps and the memory subsystem dispatches to vnic_mmio_read(opaque, addr, size) / vnic_mmio_write(opaque, addr, value, size), where addr is the offset within the BAR (i.e. our VN_REG_*).

Guest side. The driver does regs = pci_iomap(pdev, 0, 0) then readl(regs + off) / writel(val, regs + off). So vnic_hal_rd32/wr32 (which on L1 were a switch over the fake array) become readl/writel, and the matching switch now lives in vnic_mmio_read/write on the QEMU side.

L1 analogue: vnic_mmio_write is the Level-1 vn_wr32() switch - the TDT/RDT doorbell cases, ICS/IMS/IMC semantics, etc. - only now triggered by a real guest MMIO store instead of a direct function call.

FAQ - does the callback fire if the driver dereferences the pci_iomap pointer directly instead of using readl/writel? Yes - the trap is at the physical MMIO address, not at the accessor. KVM maps the BAR region to fault out to QEMU (TCG routes it via softmmu), so any CPU load/store that actually reaches that address dispatches to vnic_mmio_read/write, however the C was written; readl/writel do not "notify" QEMU, they just emit a CPU load/store. But the driver must still use readl/writel: a raw dereference may be elided / merged / reordered by the compiler (then no access is emitted, so the callback never runs - a silent failure), may use the wrong access width (the device rejects sizes outside valid.min/max_access_size), and lacks the ordering / endianness guarantees the datapath relies on. See background-knowledge.md topic 5.

3.1 What the device actually does on a special-behaviour write

vnic_mmio_write is the device's register logic. Most registers are plain RW (stored verbatim - vnic_reg_is_rw) or RO (writes ignored); a few have side effects. Here is what those writes make the VN1000 model do (qemu-device/vnic.c):

CTRL (0x10) write - and how the device answers through STATUS. Writing CTRL is the driver's only lever at init / teardown; the device reflects the outcome back through the read-only STATUS register (the driver never writes STATUS), which is the control/status split of a real datasheet (section 4.4 / 4.5): - CTRL.RST -> a self-clearing software reset (vnic_reset): the register file returns to its power-on defaults and the bit reads back 0 (mirrors a real NIC's "write RST, poll until it clears"). - CTRL.EN set -> the device comes up and asserts STATUS.RUNNING | STATUS.LINK_UP; clearing EN tears them down. So the init handshake is literally driver writes CTRL.EN, then the device drives STATUS.RUNNING/LINK_UP - the driver can poll STATUS to confirm the device (and "link") is up. - CTRL.RXEN newly set (tap mode) -> the device re-arms the backend's RX delivery (qemu_flush_queued_packets); while RXEN was clear can_receive() returned false (lessons-learned L-08).

So yes - STATUS is the device's response channel: it is RO to the driver and entirely device-driven (RUNNING/LINK_UP follow CTRL.EN). There is no ack written back into CTRL itself beyond RST self-clearing.

TDT write = the TX doorbell -> it kicks the engine. Storing TDT then calls vnic_engine_run(), which walks every pending descriptor in TDH..TDT: 1. pci_dma_read the descriptor, then the packet at vn_desc.addr; 2. loopback mode: IP-swap it and pci_dma_write it into the RX slot at RDH (data before DD, D-21), set that RX descriptor's DD, advance RDH; tap mode: qemu_send_packet it to the backend; 3. set the TX descriptor's DD and pci_dma_write it back, advance TDH; 4. after the loop, raise TXDONE (plus RXAVAIL in loopback) in ICR and msi_notify() if unmasked.

So one writel(TDT) is what makes the device do all the DMA and the completion interrupt - there is no per-descriptor cost, the single doorbell amortises the whole burst (which is why the driver batches the TDT write, D-32).

RDT write = the RX refill doorbell. The driver writes RDT after posting / refilling RX buffers to publish how far buffers are available: - loopback mode: if a prior run stalled on a full RX ring (tx_stalled, backpressure D-19), the fresh buffers let it continue, so RDT re-runs vnic_engine_run(); - tap mode: the freed RX slots let the net core re-deliver frames can_receive() had paused - qemu_flush_queued_packets.

(ICR/IMS/IMC have their own side effects - read-to-clear ICR, RW1S IMS, RW1C IMC - see background-knowledge topic 10.)

4. Device DMA -> guest memory (and the guest-side setup)

QEMU side. The device reaches guest RAM with the PCI DMA helpers - it autonomously reads/writes memory the driver prepared:

pci_dma_read (PCIDevice *d, dma_addr_t addr, void *buf, dma_addr_t len);  /* read guest mem  */
pci_dma_write(PCIDevice *d, dma_addr_t addr, const void *buf, dma_addr_t len); /* write guest mem */

addr is the bus/DMA address the driver programmed into the descriptor (vn_desc.addr). These go through the device's PCI address space (pci_get_address_space(d)), respect the IOMMU if any, and only work once the guest set the bus-master bit (so the descriptor ring base TDBAL/RDBAL and buffer addresses must be DMA addresses, not CPU pointers - unlike L1).

There is no guest callback for DMA itself - DMA is the device touching memory directly. The guest learns "the device finished" through the descriptor ownership handshake (the DD bit + barriers, D-21) and/or the completion MSI (section 5).

Guest side (the setup that makes those dma_addrs valid). - Ring (coherent): dma_alloc_coherent(dev, size, &ring->desc_dma, GFP_KERNEL) returns a CPU pointer + a dma_addr_t (desc_dma) the driver writes to TDBAL/TDBAH (RDBAL/RDBAH). Coherent = no explicit sync needed. - Packet buffers (streaming): dma_map_single(dev, data, len, dir) returns a dma_addr_t; check dma_mapping_error() (D-22). Before the device reads TX, and after it writes RX, the driver uses dma_sync_single_for_device / dma_sync_single_for_cpu to hand the buffer over / take it back. On x86 with no IOMMU these are no-ops (the dma_addr is ~the physical address), but the calls are required for portability / IOMMU / bounce buffers. - pci_set_master(pdev) enables bus-master so the device may DMA at all.

L1 analogue: vnic_hal_map() returned the kernel pointer and the engine memcpy'd it; on L2 it returns a dma_addr_t and the device pci_dma_read/writes it. The core driver treats the token as opaque (D-12), so this swap needs no core change.

5. MSI -> the guest interrupt

QEMU side. The device declares MSI in realize and fires it on completion:

msi_init(&s->pdev, /*cap offset*/ 0, /*nr_vectors*/ 1, /*msi64bit*/ true,
         /*per_vector_mask*/ false, errp);     /* once */
...
if (msi_enabled(&s->pdev))                      /* guest turned MSI on */
    msi_notify(&s->pdev, /*vector*/ 0);         /* raise the interrupt */

An MSI is, physically, just a memory write to a special address: msi_notify() writes the guest-programmed MSI data to the guest-programmed MSI address (the vCPU's LAPIC region, 0xFEEx_xxxx); QEMU's APIC decodes that write and injects the vector into the vCPU. The address/data were set by the guest when it enabled MSI - those config writes were handled by msi_write_config() (section 2), so msi_notify already knows where to send.

Guest side. The driver asked for vectors and registered a handler:

pci_alloc_irq_vectors(pdev, 1, 1, PCI_IRQ_MSI);
request_irq(pci_irq_vector(pdev, 0), vnic_isr, 0, "vnic", priv);

When QEMU delivers the MSI, the guest's IRQ subsystem runs vnic_isr (the top half), which - as on L1 - reads ICR, masks via IMC, and napi_schedule()s the poll.

L1 analogue: the tasklet that was our simulated MSI (D-20) becomes a real request_irq handler; vn_wr32(ICS) raising the cause becomes the device calling msi_notify(). The vnic_poll re-arm of IMS (which re-fires a pending cause) maps to msi_notify checking enable/mask state.

5.1 When the device raises the MSI (the timing points)

Every interrupt goes through one gate - vnic_update_irq() -> vnic_deliver_irq() -> msi_notify() - which fires only if (a) a cause is pending and unmasked (ICR & IMS), and (b) the guest has enabled MSI (msi_enabled). Otherwise nothing fires and the cause stays latched in ICR, to be delivered later when it is unmasked (mirrors the L1 ICS/IMS logic, D-20). On top of that, ITR moderation (D-33) paces delivery: ITR = 0 fires immediately; ITR > 0 enforces a minimum gap and lets further causes coalesce into one interrupt (they accumulate in ICR, so none is lost).

Within that gate, a cause is set and vnic_update_irq() is called at exactly these points (qemu-device/vnic.c):

When Cause raised Where
End of a TX run - the engine drained a TDT-doorbell burst TXDONE (+ RXAVAIL in loopback, since looped frames also land in the RX ring) vnic_engine_run() - one MSI for the whole TDH..TDT batch, not per descriptor
A frame arrived from the backend (tap mode) RXAVAIL vnic_receive(), after pci_dma_write of the frame + DD
Driver wrote ICS (software-raise) whatever bits it wrote vnic_mmio_write ICS case - lets the driver self-trigger
Driver re-armed IMS (RW1S) after a NAPI poll (no new cause) - re-delivers a cause that latched while masked vnic_mmio_write IMS case - the NAPI "mask -> poll -> re-arm" path
The ITR timer expired (no new cause) - delivers the coalesced interrupt deferred earlier vnic_itr_expire()

Two things to take away. First, the device coalesces by design: the TX engine raises a single MSI after draining the whole submitted burst, and ITR coalesces further across events - so the interrupt count is far below the packet count. Second, the mask is the driver's throttle: the ISR masks with IMC and napi_schedule()s; while masked, TXDONE/RXAVAIL keep latching in ICR but raise no MSI; when the poll finishes and re-arms IMS, any still-pending cause fires once more - which is exactly what stops NAPI from missing a completion that arrived mid-poll.

6. The NIC backend - choosing it, and how the device calls it

Sections 1-5 are about the guest <-> device edge. This section is the other edge: device <-> the outside world. A QEMU NIC is two separate objects - a device front-end (-device vnic) and a backend (-netdev <type>,id=n0), wired together with netdev=n0. The device is backend-agnostic: it talks to whatever backend through one small API, so switching tap / user / bridge is a command-line change, not a device change (D-30). The VN1000's loopback=on (default) uses no backend at all - it loops internally (Phase 9); loopback=off attaches to the -netdev.

6.1 How the device calls the backend (the NIC client API)

QEMU side. In realize the device creates a NIC client and gives it a NetClientInfo vtable:

static NetClientInfo net_vnic_info = {
    .type        = NET_CLIENT_DRIVER_NIC,
    .size        = sizeof(NICState),
    .can_receive = vnic_can_receive,   /* may we accept a frame right now?     */
    .receive     = vnic_receive,       /* a frame arrived from the backend     */
};
/* in vnic_realize(): */
qemu_macaddr_default_if_unset(&s->conf.macaddr);
s->nic = qemu_new_nic(&net_vnic_info, &s->conf, object_get_typename(OBJECT(s)), dev->id, s);
  • TX (guest -> world): the TX engine hands a frame to the backend with qemu_send_packet(qemu_get_queue(s->nic), buf, len).
  • RX (world -> guest): the backend calls our vnic_receive(); we pci_dma_write the frame into the RX ring, set DD, advance RDH and raise RXAVAIL (then msi_notify).
  • Flow control / backpressure: can_receive() returns false when we cannot take a frame; the backend then pauses delivery and resumes only after the device calls qemu_flush_queued_packets(qemu_get_queue(s->nic)). Two flush points are needed: the RDT refill doorbell (ring had filled), and when RX is first enabled (CTRL.RXEN) - omitting the latter leaves the backend paused forever (the Phase 10 bug, lessons-learned L-08).
  • DEFINE_NIC_PROPERTIES(VNICState, conf) adds the mac= and netdev= device properties.

L1 analogue: none - Level 1 had no external backend; its "network" was the internal loopback engine. The VN1000 keeps that as loopback=on, so qemu_send_packet/receive() are used only in loopback=off (tap) mode; the descriptor-ring + MSI half (sections 3-5) is identical either way.

The device model is the same for all of these; only the -netdev differs.

6.3 How user (slirp) works - the contrast with tap

tap hands raw Ethernet frames to /dev/net/tun, so they enter the host kernel network stack like a real NIC's would; the host then routes/NATs/bridges them. slirp instead runs a userspace TCP/IP stack + virtual router inside the QEMU process (the libslirp library): it terminates the guest's L2/L3/L4, presents a virtual network (guest 10.0.2.15, gateway/DNS 10.0.2.2/10.0.2.3, its own DHCP), and when the guest opens a connection it opens an ordinary host socket() to the real destination and relays bytes between the two - a socket-level NAT. That is why slirp needs no host config or root (it only borrows the host's ability to make outbound connections), but also why the host cannot initiate to the guest (except hostfwd=) and there is no real L2 interface to tcpdump. ("Slirp" is the name of a mid-1990s program that emulated a SLIP/PPP link to give shell users IP connectivity - a play on SLIP (Serial Line IP), not an acronym.)

Bottom line for Level 2: qemu-device/vnic.c re-implements, on the host side, the exact register/descriptor/interrupt behaviour the Level 1 fake engine already validated - vn_wr32 becomes vnic_mmio_write, the engine's memcpy becomes pci_dma_read/write, and vn_wr32(ICS) becomes msi_notify. The guest driver's core/ is unchanged; only hal/vnic_hal_l2.c swaps fake-register access + pointer tokens for readl/writel + dma_* (D-12).
