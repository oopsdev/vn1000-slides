L-13 The single-stream smoke "Retr" is a spurious Tail Loss Probe, not loss

  • Date / commit: 2026-06-18. Found while verifying the Phase 5.2 priv->up fix on Level 2.
  • Trigger: 01-smoke.sh A2 (single-stream iperf3 -t 3 over the L2 device loopback) failed once with Retr = 1 while every device counter was clean. L-04 had assumed A2 would not stall at low load and kept a strict Retr == 0 - so was this real loss, or spurious again?
  • Measured (Level 2 loopback, 10 single-stream rounds, TCP MIB via nstat; the one round that retransmitted):
signal value source
Tcp:RetransSegs 1 nstat (/proc/net/snmp)
TcpExt:TCPLossProbes 1 nstat - it was a Tail Loss Probe
TcpExt:DelayedACKLost 1 a delayed ACK overran its timer
TcpExt:TCPDSACKRecv / ...RecvSegs 1 / 1 receiver got the segment twice
TcpExt:TCPDSACKIgnoredNoUndo 1 sender confirmed the retransmit was needless
TcpExt:TCPFastRetrans / TCPLostRetransmit / SackReorder 0 nstat - no loss-driven retrans
tx_dropped/rx_dropped/tx_errors/rx_errors (delta, all 10 rounds) 0 sysfs
  • What we learned: 1. Timeout-class, not a drop. No TCPFastRetrans/TCPLostRetransmit (not 3-dup-ACK loss recovery) and no TCPTimeouts (not a full RTO) - it is a Tail Loss Probe (TCPLossProbes), the Linux timer that probes the tail of a flow before a full RTO. 2. It was spurious. TCPDSACKRecv == RetransSegs (the receiver DSACK'd the duplicate) plus DSACKIgnoredNoUndo (the sender agreed) prove the original tail segment did arrive - and the device dropped nothing. A short -t 3 flow ends with a tail whose last ACK is late under loopback backpressure, so TLP fires once, harmlessly. 3. L-04 generalises to single-stream. L-04 saw spurious Retr under -P8; this shows even one stream at low load can trip one spurious TLP at the tail, so A2's strict Retr == 0 was too optimistic. 4. RetransSegs >= DSACKRecv - read it as a ratio, not an equality. A later -P8 C1 run showed RetransSegs=58 vs TCPDSACKRecv=55 (TCPLossProbes=121) with device drops still 0. The 3-segment excess is expected, not 3 lost packets: a DSACK returns ~1 RTT after its retransmit so the last few fall outside the nstat window; DSACK is best-effort (delayed-ACK coalesced / lost / GRO-merged); a few retransmits genuinely fill a host-queue micro-drop (qdisc overflow under saturation) or a TLP-recovered tail loss - real, but above the driver, not a device/wire drop. So "spurious" = DSACKRecv close to RetransSegs and zero device drops - never require strict equality (background-knowledge topic 13.1).
  • Outcome: 01-smoke.sh A2 now tolerates a small retransmit count (<= 5) and prints the TCP-MIB delta (tcp_reset/tcp_report) so the cause is auditable inline; rx_dropped/tx_dropped (A3) remain the authoritative real-loss gate - consistent with L-04 and 03-concurrency.
  • Follow-up: none - expected TCP behaviour on a backpressured loopback, not a driver bug.

L-12 MTU scales throughput (~2.5x at 9000); the interrupt rate is NAPI-driven, not packet-count

  • Date / commit: 2026-06-15. scripts/perf-sweep.sh. Raw: results/phase12-itr-mtu.md.
  • Measurement: guest, itr=0, two 30 s runs per MTU, each direction.
  • RX (iperf3 -R, host -> guest), run1/run2: 1500 3.0/2.5, 3000 4.3/4.0, 6000 5.8/6.7, 9000 7.4/7.4 Gbit/s; 0 drop.
  • TX (iperf3, guest -> host, doorbell batching on): 1500 3.2/3.3, 3000 4.6/4.2, 6000 6.6/6.2, 9000 8.6/8.2 Gbit/s; 0 drop.
  • Interrupts/s were non-reproducible in both (e.g. RX 6000: 2823/1604; TX 9000: 1024/1299).
  • What we learned: 1. Throughput rises monotonically and reproducibly with MTU in both directions - ~2.5x at 9000 vs 1500 - because the same bytes ride ~6x fewer packets, so the per-packet cost (descriptor DMA, the doorbell/interrupt handling, the stack) is amortized over far more payload (background topic 7). TX is the faster direction at jumbo (9000: ~8.4 vs RX ~7.4 Gbit/s): TX batches the doorbell (D-32) and the engine sweeps a whole burst per kick, vs RX's per-frame receive() callback in the iothread. 2. The interrupt rate does NOT track MTU - it swings 2-3x run-to-run at the same MTU and is non-monotonic. It is governed by NAPI completion timing (how often a poll drains below budget and re-arms IMS), which is sensitive to load/scheduling, not by packet count. "Bigger MTU = fewer interrupts" is not a reliable rule here. 3. Methodology: single 10 s runs were too noisy - a spurious 9000 throughput dip and wild interrupt swings - and two 30 s runs were needed to separate the throughput signal from noise. On KVM/WSL2, measure long and repeat before concluding.

L-11 ITR moderation: ~60x fewer interrupts at flat throughput - but NAPI is the primary moderator

  • Date / commit: 2026-06-15. D-33; scripts/itr-sweep.sh. Raw: results/phase12-itr-mtu.md.
  • Measurement: guest, iperf3 -R (high RX), sweeping itr_usecs: 0 -> 849/s, 50 -> 389/s, 500 -> 116/s, 2000 -> 14/s, 5000 -> 18/s; throughput ~2.6-3.1 Gbit/s (flat, run variance), 0 drop throughout.
  • What we learned: 1. ITR cuts the interrupt rate hard (849 -> 14/s, ~60x) with throughput essentially unchanged and 0 drops - the classic moderation win: interrupt/CPU cost falls, throughput is unaffected because NAPI already batches the data (topic 7.3). 2. NAPI is the primary moderator. Even itr=0 is only ~849/s (not per-packet): the ISR masks via IMC and NAPI stays in poll under load, re-arming IMS only when it drains below budget. ITR trims further by delaying the re-trigger so more packets accumulate per interrupt. (This corrects my earlier guess that a sub-gap ITR "wouldn't bite" - it does, because the effect is per-interrupt batching, not a hard 1/ITR cap.) 3. No drops even at ~14 ints/s (~16k packets between interrupts) because under sustained load NAPI keeps polling in softirq without a fresh interrupt; the interrupt only restarts NAPI after it idles.
  • Cost side (not yet measured): ITR trades latency for fewer interrupts (the RX completion is delayed up to ITR us -> higher RTT). A ping-RTT-per-ITR sweep is the missing half; adaptive DIM is also backlog.

L-10 TX doorbell batching (xmit_more) ~4.5x'd TX - the VM-exit was the bottleneck

  • Date / commit: 2026-06-15. Change 394b2d5 (D-32). Raw: results/phase12-tx-doorbell.md.
  • Measurement: same-session A/B, guest -> host iperf3 over the tap, toggling the new tx_coalesce param at runtime (no reload, nothing else changes):
  • off (per-packet doorbell, the old behaviour): 414 Mbit/s, 0 Retr.
  • on (batch TDT over an xmit_more burst): 1.86 Gbit/s, 0 Retr.
  • ~4.5x, no added loss. The off number matches the L-09 per-packet baseline (~350-414 Mbit/s).
  • What it confirms: the dominant Level 2 TX cost was the per-packet writel(TDT) = one MMIO VM-exit per packet. Deferring it across the stack's xmit_more burst and ringing once per burst (D-32, background topic 7.2) removes most of those exits. It also closes most of the TX/RX gap from L-09: TX 0.35 -> 1.86 Gbit/s vs RX ~3.77 Gbit/s (RX was already NAPI-batched; the residual gap is the remaining per-burst doorbell + the synchronous vCPU-thread engine run).
  • cwnd is independent confirmation: across the run the TCP congestion window grew ~6x with batching on (to 1.51 MB) but stayed small off (~130-350 KB) - with 0 Retr in both arms. So the small off cwnd is not congestion/loss; the doorbell exits throttled the send path and TCP simply read the link as slow. Removing the per-packet exit let the realized rate (and hence cwnd ~= rate x RTT) climb. Per-second cwnd table in results/phase12-tx-doorbell.md.
  • Why it is the right fix: it is exactly what real drivers do (igb/ixgbe/mlx5), the change is in the shared core/ and is level-independent (the kernel clears xmit_more on the last skb of a burst, and a full ring forces the doorbell, so no descriptor is left un-rung; on Level 1 it just wakes the engine once per burst). 0 Retr in both arms shows correctness held.
  • Caveat / follow-up: the gain scales with burst size (GSO/qdisc), measured on QEMU 6.2 + KVM, tap mode. The next lever is the interrupt side - device ITR (Phase 12 b) - which targets IRQ rate, not doorbell exits.

L-09 Tap throughput: TX (doorbell VM-exit) is ~10x slower than RX (NAPI batch)

  • Date / commit: 2026-06-15. Measured on Level 2 tap (loopback=off), QEMU 6.2 + KVM. Raw: results/phase10-tap.md.
  • Measurement: iperf3 guest <-> host over the tap, 10 s each way:
  • TX (guest -> host, iperf3 -c): ~350 Mbit/s, 0 Retr.
  • RX (host -> guest, iperf3 -c -R): ~3.77 Gbit/s, 79 Retr (~0.002% of 4.4 GB).
  • Device counters after both: 0 errors, 0 dropped in each direction (3.42M RX / 342K TX pkts).
  • The lesson - the asymmetry is structural, not a bug:
  • TX is gated by the MMIO doorbell. Each transmit batch ends with writel(TDT), which is an MMIO trap = a VM-exit from the guest vCPU into QEMU (the engine then runs synchronously in the vCPU thread). VM-exits are expensive, so TX throughput is bounded by the doorbell/exit rate.
  • RX is delivered by the iothread and drained by NAPI. The host/iothread pci_dma_writes frames into the RX ring and raises one MSI; the guest's NAPI poll batches many packets per interrupt with few guest exits. So RX amortizes far better -> ~10x TX here.
  • This is the emulated-NIC face of a real principle: MMIO exits are costly; interrupt coalescing / NAPI batching is what makes high throughput possible (background-knowledge topics 2, 5). A real optimization (and our backlog) would be to moderate/batch the TX doorbell (e.g. ITR / fewer writel(TDT)), the TX analogue of NAPI.
  • Robustness confirmed: the 79 Retr were TCP-layer spurious RTOs, not device drops (rx_dropped/errors = 0); the Phase 10 RX flow-control fix (L-08) held at sustained multi-Gbit/s.
  • Known limitations surfaced: 1. STAT_RXBYTES/STAT_TXBYTES are 32-bit device registers that wrap past 4 GiB - packet counts are reliable, byte totals above 4 GiB are not. 64-bit counters / driver-side accumulation is backlog. 2. The L2 netdev shows altname enp0s3 - NET_NAME_USER (36fcc54) keeps the primary name vnic0; udev attaches its predictable name as an alternative name (both resolve). The pin works as intended.
  • Follow-up: this is the seed of the Phase 12 performance analysis (real DMA/exits vs the L1 in-process memcpy).

L-08 Phase 10 tap RX wedged: a NIC must flush its backend when RX is enabled

  • Date / commit: 2026-06-12. Tap backend b7dc2c0; RX fix a8850d1. Raw: results/phase10-tap.md.
  • Trigger: first guest<->host ping over the real tap backend (loopback=off). The guest's TX reached the host (echo requests on vnic-tap0) and the host replied, but the guest got 0% of the replies.
  • What happened: ip -s link show vnic0 had RX packets = 0 (TX fine), and /proc/interrupts showed 3 TXDONE MSIs but 0 RXAVAIL - vnic_receive() was never called. Yet tcpdump -ni vnic-tap0 on the host clearly showed both the echo request and the reply.
  • Root cause - QEMU net flow control. While RXEN was clear at boot, vnic_can_receive() returned false, so QEMU disabled the tap fd read handler. The driver enables RXEN after posting RX buffers, so the RDT-doorbell flush fired while RXEN was still clear (too early); then RXEN went up with nothing to flush the backend, leaving the tap read handler stuck off. The host's reply sat in the kernel tap queue (so tcpdump still saw it) but QEMU never read it -> vnic_receive() never ran -> no RX, no RXAVAIL.
  • What we learned: 1. A QEMU NIC must qemu_flush_queued_packets() whenever it becomes able to receive (RX turned on), not only on ring-refill. A can_receive() that returned false earlier disables backend delivery until an explicit flush re-arms it. 2. tcpdump on the host tap is kernel-side: it shows frames at the tap interface, before and independent of whether the emulated device reads them. "tcpdump sees the reply" did not mean the guest device received it - the decisive signals were the device RX stats and the RXAVAIL interrupt count (both zero). 3. Keep can_receive() cheap and side-effect-free (no per-frame guest-memory reads from the net/iothread path); let receive() return 0 for ring-full and re-arm via flush. 4. The driver was unchanged and correct throughout (Phase 10 touched only the QEMU device) - diagnose backend bring-up on the device side.
  • Fix (a8850d1): qemu_flush_queued_packets() on CTRL.RXEN enable; vnic_can_receive() gates only on tap-mode + RXEN. Outcome: guest<->host ping 0% loss, vnic0 RX packets > 0, RXAVAIL interrupts now fire. Phase 10 done. Later also validated guest -> public internet through vnic0 (ICMP/DNS/TCP/HTTP, ip route get confirms egress via vnic0) once the host is made a router + NAT (scripts/tap-nat.sh) - the full datapath carries real external traffic, not just loopback. Details in results/phase10-tap.md.

L-07 Phase 9 datapath works on L2; a PCI netdev gets udev-renamed by default

  • Date / commit: 2026-06-12. Phase 9 datapath f634651; name-pin fix 36fcc54. Raw run: results/phase9-l2-tests.md.
  • Trigger: first full tests/run-all.sh on Level 2 (MODULE=.../vnic-l2.ko), over real MMIO/DMA/MSI against the self-built QEMU device.
  • What worked: the datapath is solid - iperf3 ~574 Mbps, 0 Retr-as-loss / 0 drops / 0 errors, lossless ping including jumbo (-s 8000) and IP-fragmented (-s 65000) frames, ndo_change_mtu resize, RX alloc-failure injection (recycle + rx_dropped), -P8 concurrency, the 40x down/up flood storm, and rmmod-under-live-traffic - all pass. The shared core/ + the IP-swap loopback behaved identically to Level 1, now exercised through dma_alloc_coherent rings, dma_map_single buffers and a real MSI (D-25/D-26). Retr>0 with 0 drops is the same backpressure-latency artefact as L-04, not loss.
  • The gotcha: 9 failures (01-smoke "present", all 8 02-lifecycle "missing on load") were not datapath bugs - systemd-udev renamed vnic0 -> enp0s3. dmesg showed it plainly: vnic 0000:00:03.0 enp0s3: renamed from vnic0. The tests that ran after the rename settled all passed; the failures were a rename race - the "interface present" checks fired in the millisecond between register_netdev (still vnic0) and udev finishing the rename.
  • Why it renames: the L2 netdev has a PCI parent (SET_NETDEV_DEV) and a kernel-enumerated name (NET_NAME_ENUM, from register_netdev("vnic%d")). systemd's net_setup_link default NamePolicy=keep kernel database onboard slot path only keeps user/renamed names, so for an ENUM name it computes a predictable slot name: en + p0s3 (PCI 0000:00:03.0) = enp0s3. This is realistic real-NIC behaviour (eth0 -> enp3s0). Level 1 is untouched because it has no PCI parent, so udev cannot derive a predictable name and leaves vnic_loop0.
  • What we learned: 1. A PCI netdev's registered name is not its final name - udev may rename it asynchronously. A test/setup must either pin the name or discover the iface by driver/MAC, never assume the name it registered with, and never check "present" immediately after load (rename race). 2. Diagnose by dmesg first: renamed from X tells you instantly it is naming, not datapath.
  • Fix: register a fixed vnic0 and set name_assign_type = NET_NAME_USER so udev's keep policy retains it (36fcc54). Single-device only (fixed name); multi-device is backlog A3. Alternatives considered: a /etc/systemd/network/*.link file (admin-side, not in-repo) or net.ifnames=0 (global). Confirmed: with the name pinned the rename race is gone and the full suite is all-green (6/6, overall PASS) at the default IFACE=vnic0 (iperf3 ~611 Mbps, Retr 3, 0 drops).

L-05 "Destination Host Unreachable" is an L2/neigh failure, not a datapath bug

  • Date / commit: 2026-06-11, 16feb35.
  • Trigger: the C2 concurrency test (40x down/up under a flood ping) reported 100% post-storm ping loss, which looked like rapid down/up wedging the RX datapath.
  • What happened: the ping output was Destination Host Unreachable + +10 errors (pipe 10), not Request timeout. That means the packet never left - the kernel had no L2 address for the peer. ip link set <dev> down flushes the per-device neighbour table, including our static nud permanent peer entry (D-08), so after the storm there was no MAC for 192.168.10.2.
  • What we learned: 1. Diagnose by the ping error first. Destination Host Unreachable / +errors = ARP/neigh (L2) resolution failed, the frame never entered the datapath; Request timeout / no reply = the frame went out but nothing came back (a real RX/datapath issue). Check which before blaming the driver. 2. The driver was fine. RX == TX, 0 dropped, 0 errors, dmesg clean throughout - the datapath survived the storm; only the host neighbour table was disturbed (standard kernel behaviour on link-down, not driver state). 3. A static-neigh loopback test must re-establish config (setup-vnic.sh) after any link bounce; persistence of nud permanent across down/up cannot be assumed.
  • Outcome: tests/03-concurrency.sh C2 re-runs vnic_setup after the storm and diag() now dumps ip neigh; the test then went green (post-storm loss 0%).

L-04 With backpressure, iperf "Retr" is latency, not loss

  • Date / commit: 2026-06-11, 18877fc (test instrumentation).
  • Trigger: the new concurrency test C1 (iperf3 -P8) reported Retr = 8 over the Level 1 loopback. With D-19 backpressure (L-02) the device never drops on queue-full - so is something actually dropping?
  • Measured (Level 1, RX ring 256, backpressure; one C1 run):
signal value source
iperf Retr / Tcp:RetransSegs 8 iperf JSON / /proc/net/snmp
rx_dropped / tx_dropped (delta) 0 / 0 /sys/class/net/vnic0/statistics
TcpExt:TCPDSACKRecv 8 (== Retr) /proc/net/netstat
TcpExt:TCPLossProbes 41 /proc/net/netstat
  • What we learned: 1. Nonzero Retr was not packet loss. rx_dropped == tx_dropped == 0 proves the device dropped nothing, and TCPDSACKRecv == RetransSegs proves the receiver got every "retransmitted" segment a second time - i.e. the originals did arrive. The retransmits were spurious. 2. Mechanism: backpressure turns a would-be drop into latency. When the RX ring stalls, a segment / ACK can sit queued past TCP's TLP / RTO timer, so TCP retransmits a segment that was only late, not lost. This is the actual mechanism behind L-01's "deeper ring -> fewer Retr": a deeper ring stalls less often and for shorter, so it trips fewer spurious retransmits (at the cache cost L-01 measured). 3. Measure loss, not Retr. Once backpressure exists, iperf Retr is a misleading loss proxy. Gate correctness on rx_dropped / tx_dropped (real device loss, sysfs) and attribute retransmits via the TCP MIB in procfs - /proc/net/netstat (TCPDSACKRecv, TCPSpuriousRtxHostQueues, TCPSpuriousRTOs, TCPLossProbes) and /proc/net/snmp (RetransSegs). TCP stats are not under /sys/class/net (that is link-layer only).
  • Outcome: tests/03-concurrency.sh C1 now gates on rx_dropped / tx_dropped and prints the TCP-MIB delta. (The single-stream smoke 01-smoke.sh A2 originally kept a strict Retr == 0, on the assumption it would not stall at low load - L-13 later showed even one stream trips an occasional spurious Tail Loss Probe at the tail, so A2 was relaxed to tolerate a small count + print the MIB.)
  • Follow-up: a residual real RX drop under burst would be a Model A GFP_ATOMIC refill failure (rx_dropped > 0); page_pool / Model B (backlog A5) would remove it.

L-03 The full interrupt protocol (tasklet MSI) costs nothing here

  • Date / commit: 2026-06-10, 11544ed.
  • Trigger: Phase 4 replaced the engine's direct napi_schedule() with the real ICR/ICS/IMS/IMC protocol and a tasklet "MSI" (an extra softirq hop). Did the added fidelity cost throughput?
  • Measured (RX 256, backpressure, same setup):
Completion signal Throughput Retr
engine napi_schedule (Phase 3) 6.03 Gbit/s 0
tasklet MSI (Phase 4) 6.33 Gbit/s 0

ip -s link after the Phase 4 run: TX packets == RX packets (2,762,908 each), 0 dropped, 0 errors - the loopback is exactly 1:1 and lossless. - What we learned: 1. No regression: 6.33 vs 6.03 Gbit/s is within run-to-run variance (intervals ranged 5.2-7.3). The extra tasklet hop and the irq_lock-protected register dance are negligible at this rate, so modelling the real interrupt protocol was effectively free - we kept the behaviour and gained fidelity (and Level-2 readiness: the tasklet becomes a real MSI ISR). 2. TX == RX with 0 dropped is the clean proof that backpressure (D-19) holds under load: no frame is ever dropped, every TX loops back to exactly one RX. - Outcome: Phase 4 adopted; tasklet/MSI is the completion path going forward. - Data: results/loopback-iperf3.md.

L-02 Backpressure beats both dropping and a bigger ring

  • Date / commit: 2026-06-10, 49dcd48.
  • Trigger: L-01 showed a dilemma - a small RX ring drops (retransmits), a big one thrashes cache (lower throughput). Tried a third option: don't drop, apply backpressure.
  • Change tested: on RX-full, the engine stalls (stops consuming TX, leaves the descriptor un-DD) instead of dropping; the TX ring fills, netif_stop_queue throttles the sender; NAPI refilling RX rings the RDT doorbell to resume the engine. RX ring 256.
RX ring RX-full policy Throughput Retr
256 drop 4.41 Gbit/s 745
4096 drop 3.62 Gbit/s 0
256 backpressure 6.03 Gbit/s 0
  • What we learned: 1. Backpressure won both axes: 0 retransmits and the highest throughput. It removes drops (cwnd grows to ~4 MB, no retransmit overhead) while keeping the small ring's hot cache (~0.4 MB working set vs the 4096 ring's ~6 MB thrash). 2. The real fix for overflow was "don't drop", not "make the ring bigger". Flow control (netif_stop_queue) turns loss into a slowdown the sender (TCP) reacts to cleanly. 3. Resume design matters: a stalled producer needs a wake path. Here the consumer's refill doorbell (RDT) wakes the engine, and the napi_schedule after every stall guarantees a poll runs to drain RX - so it cannot deadlock.
  • Outcome: adopted as the default; recorded as design-decision D-19.
  • Data: results/loopback-iperf3.md.

L-01 RX ring depth is a trade-off (overflow drops vs cache thrash)

  • Date / commit: 2026-06-10, RX ring bump 9684e5b; baseline driver 95e67eb.
  • Trigger: after Phase 3, iperf3 TCP over the Level 1 internal loopback ran at ~4.4 Gbit/s but reported 745 retransmissions - over a software loopback with no real wire, so the driver itself must be dropping frames.
  • Change tested: VNIC_RX_RING_DEFAULT 256 -> 4096.
RX ring Throughput (goodput) Retr max cwnd
256 4.41 Gbit/s 745 ~0.47 MB
4096 3.62 Gbit/s 0 ~4.0 MB
  • What we learned: 1. The retransmits were RX-ring overflow: the loopback engine (producer) outran the single NAPI vnic_poll (consumer); the 256-entry ring filled and extra frames were dropped. Deepening the ring took Retr 745 -> 0. Hypothesis confirmed. 2. Bigger is not free: goodput dropped (4.41 -> 3.62 Gbit/s, receiver 2.57 -> 2.11 GB). The loopback is CPU-bound, and a 256-deep ring reuses each buffer every 256 packets (stays hot in cache, ~0.4 MB working set) while a 4096-deep ring spreads ~6 MB that is reused only every 4096 packets, so the engine's memcpy and the poll's header reads miss cache far more often. 3. Ring depth is a trade-off with a sweet spot: too small -> overflow drops
    • retransmits; too large -> cache thrash + lower throughput.
  • Follow-up: sweep 256 / 512 / 1024 / 2048 / 4096 to find the smallest depth that reaches Retr = 0 with the least throughput loss. Best done with ring depth as a module parameter (no rebuild per value). Tracked under Phase 7.
  • Data: results/loopback-iperf3.md.
