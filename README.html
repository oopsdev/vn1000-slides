Linux NIC Simulation (vnic)

A learning project to understand the Linux NIC driver working flow end to end — the device/driver hardware-software interface (registers, DMA descriptor rings, interrupts, NAPI) and netdev integration — by building a software-simulated NIC (VN1000) and its Linux driver. It runs first as an in-kernel engine (Level 1) and finally as a QEMU PCI device that behaves like a real NIC inside the guest (Level 2, pingable to the host).

Scope: a Linux-kernel simulation framework that exercises the core architecture shared by NIC / NVMe / GPU / accelerator drivers — queue submission, DMA buffers, completion handling, interrupt moderation, and datapath performance analysis.

Levels (two)

Each level exposes its own netdev — Level 1 is vnic_loop0 (the in-kernel loopback engine), Level 2 is vnic0 (the QEMU device) — so both drivers can be loaded at once (distinct module and interface names). The ring / DMA / interrupt engine and the shared spec header are identical across them; Level 2 only swaps the "device" behind the rings from an in-kernel engine to a QEMU PCI device (real MMIO / DMA / MSI) — it is not a rewrite.

Level What Acceptance
Level 1 — kernel-only netdev pure kernel module: single netdev vnic_loop0 (real-NIC structure: net_device ops + NAPI + TX/RX ring + DMA-like buffers + simulated interrupt); the engine does internal loopback (swap src/dst IP, keep ports); no PCI bus vnic_loop0 appears in ip link; after assigning an IP, ping the fictional peer; server on 0.0.0.0 + client to the peer can test TCP/UDP (iperf)
Level 2 — QEMU PCI device + netdev custom QEMU PCI device (BAR / MMIO / MSI / DMA); the guest driver registers net_device vnic0; packets flow through the QEMU backend to the host vnic0 appears in ip link; after assigning an IP, ping the host; visible in host-side tcpdump
Level 3 — FPGA / PCIe endpoint (bonus) real hardware; out of the main scope —

The device and the driver share one register / descriptor definition (include/vnic_regs.h) so the two sides never drift apart — the most common source of bugs in this kind of project.

Development environment

  • Edit: Windows (this directory)
  • Build / run: QEMU + an Ubuntu guest under WSL2 — kernel-module development happens inside the guest (the WSL2 custom kernel has no matching headers; the guest has apt install linux-headers, kgdb works, and a module panic only kills the guest). Full SOP: docs/dev-setup.md.
  • Level 2 needs: building QEMU from source (there is no stable out-of-tree mechanism to load a custom device dynamically).
  • Debugging:
  • guest kernel: QEMU -s -S + gdb single-step
  • QEMU device model: host gdb attached to the QEMU process

References (reading order)

  1. Intel e1000e Linux driver — the most approachable head/tail ring register model
  2. Intel 82574 datasheet — to cross-check the e1000e registers (TDBAL/TDH/TDT, etc.)
  3. Intel 82599 datasheet + Linux ixgbe — production 10G NIC, NAPI, multi-queue
  4. ixy (educational user-space driver) — understand the 82599 datapath in minimal code
  5. virtio-net / virtio_ring — preparation for the QEMU version (descriptor ring essence)
