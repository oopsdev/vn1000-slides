Debug-Kernel Test Gate

The Phase 5 acceptance gate ("passes concurrency / edge tests") is meant to run tests/run-all.sh on a guest kernel built with the runtime debug checkers - KASAN (use-after-free / out-of-bounds), lockdep (lock ordering / misuse) and kmemleak (leaks). The functional suite already passes on a stock kernel; this gate catches the races / UAF / leaks that functional tests cannot see.

These are build-time kernel options. Stock Ubuntu -generic does not enable them (so /sys/kernel/debug/kmemleak is absent and the suite's kml_check SKIPs), and there is no prebuilt Ubuntu kernel that has them. So either use a boot parameter for partial coverage (Stage 0) or build a debug kernel (Stage 1). All steps run inside the guest (D-14: all kernel work in the guest).

Status: documented SOP, not yet executed. The suite is green on a stock kernel (results/); running it under Stage 1 is the remaining Phase 5 item.

Stage 0 - slub_debug boot parameter (no rebuild, partial)

Stock Ubuntu usually has CONFIG_SLUB_DEBUG=y (compiled in, off by default). Enabling it at boot turns on slab red-zoning / poisoning / use-after-free / double-free / sanity checks - a large slice of what KASAN finds, for free.

# confirm the options are compiled in
grep -E "SLUB_DEBUG=|PAGE_POISONING=" /boot/config-$(uname -r)

# add the boot params and reboot
sudo sed -i 's/GRUB_CMDLINE_LINUX_DEFAULT="\(.*\)"/GRUB_CMDLINE_LINUX_DEFAULT="\1 slub_debug=FZPU page_poison=1"/' /etc/default/grub
sudo update-grub && sudo reboot

After reboot:

cat /proc/cmdline                 # expect ... slub_debug=FZPU ...
cd ~/vnic && sudo bash tests/run-all.sh

slub_debug flags: F sanity, Z red-zone, P poison, U user tracking (records alloc/free stacks - makes a splat point at the culprit). Any slab corruption / UAF / double-free in our RX/TX buffer or skb handling now prints to dmesg, which the suite's dmesg_clean catches. lockdep, kmemleak and full KASAN still need a config build (Stage 1).

Stage 1 - build a debug kernel (the full gate)

1. Give the guest more resources

KASAN roughly doubles memory and slows the build/run 2-3x. A 2 vCPU / 2 GiB guest is too small. Bump scripts/run-qemu.sh for the debug runs, e.g. -smp 4 -m 6144, and restart the guest.

2. Build deps + kernel source

A vanilla stable kernel avoids the signing-key hassle of the Ubuntu source tree.

sudo apt install -y build-essential libncurses-dev bison flex libssl-dev \
                    libelf-dev bc dwarves cpio

ver=6.8.12     # pick one close to your guest's `uname -r`
curl -LO https://cdn.kernel.org/pub/linux/kernel/v6.x/linux-$ver.tar.xz
tar xf linux-$ver.tar.xz && cd linux-$ver

3. Configure: stock config + the debug checkers

cp /boot/config-$(uname -r) .config
scripts/config -d SYSTEM_TRUSTED_KEYS -d SYSTEM_REVOCATION_KEYS   # skip signing keys

scripts/config -e KASAN -e KASAN_GENERIC \
               -e PROVE_LOCKING -e DEBUG_LOCK_ALLOC -e DEBUG_SPINLOCK \
               -e DEBUG_ATOMIC_SLEEP -e PROVE_RCU \
               -e DEBUG_KMEMLEAK -e DEBUG_OBJECTS -e DEBUG_OBJECTS_FREE \
               -e DEBUG_INFO_DWARF_TOOLCHAIN_DEFAULT
make olddefconfig
Option Catches
KASAN / KASAN_GENERIC use-after-free, out-of-bounds on heap/stack/globals
PROVE_LOCKING (lockdep) + DEBUG_LOCK_ALLOC + DEBUG_SPINLOCK lock-ordering inversions, wrong-context locking, bad spinlock use
DEBUG_ATOMIC_SLEEP sleeping while atomic (e.g. in the spinlock/softirq paths)
PROVE_RCU RCU misuse
DEBUG_KMEMLEAK unreferenced allocations (skb / ring / buffer leaks)
DEBUG_OBJECTS (+_FREE) use of freed / wrong-state objects (timers, etc.)

4. Build, install, reboot

make -j"$(nproc)"                  # slow (KASAN build is large)
sudo make modules_install install  # installs the kernel + rebuilds initramfs
sudo update-grub                   # add it to the menu
sudo reboot                        # boot the new kernel (select it if not default)

If kmemleak is not set to auto-scan, add kmemleak=on to the kernel cmdline (same GRUB_CMDLINE_LINUX_DEFAULT edit as Stage 0).

5. Rebuild the module against the debug kernel + run

uname -r                           # confirm the debug kernel is running
cd ~/vnic/driver && make           # builds against the new /lib/modules/.../build
cd .. && sudo bash tests/run-all.sh

What changes vs a stock run

  • kmemleak: /sys/kernel/debug/kmemleak now exists, so the suite's kml_check (run at the end of each test, after kml_clear at the start) actually scans; an unreferenced skb / ring buffer fails the test.
  • lockdep / KASAN / atomic-sleep: a violation prints a splat to dmesg, which dmesg_clean (run after every step) catches and fails.
  • Everything is much slower. If a test times out or the storm/soak loops take too long, shrink them with ITERS=... (e.g. ITERS=10 sudo -E bash tests/03-concurrency.sh).

A fully green run-all.sh under this kernel is the Phase 5 sign-off.
