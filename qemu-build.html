Self-built QEMU with the VN1000 Device

Level 2 needs a QEMU that contains our emulated device (qemu-device/vnic.c). QEMU has no stable out-of-tree device-module ABI, so the device is compiled into a QEMU source tree and we run a self-built binary. This is a WSL2-host task (QEMU is the hypervisor that runs the guest) - distinct from the driver work, which stays inside the guest (D-14). See TODO Phase 8 and decision D-27.

The device source lives in qemu-device/vnic.c in this repo (the source of truth); it is copied into a checked-out QEMU tree plus a two-line meson/Kconfig hook. Pin the QEMU version to the one already running the guest so behaviour matches.

1. Pin the version (match the guest's current QEMU)

On the WSL2 host:

qemu-system-x86_64 --version      # e.g. "QEMU emulator version 8.2.2"

Use that as the upstream tag (8.2.2 -> v8.2.2).

2. Build dependencies (WSL2 host, Ubuntu)

sudo apt install -y git python3-venv ninja-build pkg-config \
                    libglib2.0-dev libpixman-1-dev flex bison
# or, if deb-src is enabled:  sudo apt build-dep qemu

3. Get the QEMU source at that tag

git clone https://gitlab.com/qemu-project/qemu.git ~/qemu
cd ~/qemu
git checkout v8.2.2                       # <- the version from step 1

4. Add the VN1000 device to the tree

REPO is a checkout on the WSL2 host, not the one inside the guest. QEMU is built on the host, so step 4's cp can only pull from a repo that lives on the host. A clone sitting inside the guest VM is invisible to the host build. Keep git as the single channel (per D-14, driver work happens in the guest, device work on the host - both sync through the same remote):

```bash

one-time, on the WSL2 host:

git clone https://github.com/oopsdev/Linux_NIC_Simulation.git ~/vnic # -> REPO=~/vnic

after editing qemu-device/vnic.c inside the guest, publish then pull on the host:

guest: git commit -am "..." && git push

host: cd ~/vnic && git pull

```

(Alternatives if you must move uncommitted edits: scp -P 2222 <guest-user>@localhost:... over the run-qemu.sh SSH forward, or point REPO at the Windows checkout via /mnt/c/.... Both are manual copies that drift out of sync - prefer the git path above.)

Drop our device source in and register it with meson + Kconfig (idempotent - skip if already there):

REPO=~/vnic                                # this repo on the host
cp "$REPO/qemu-device/vnic.c"     hw/net/vnic.c
cp "$REPO/include/vnic_regs.h"    hw/net/vnic_regs.h   # shared spec; vnic.c #includes it

# meson: compile vnic.c when CONFIG_VNIC is set
grep -q "CONFIG_VNIC" hw/net/meson.build || \
  echo "system_ss.add(when: 'CONFIG_VNIC', if_true: files('vnic.c'))" >> hw/net/meson.build

# Kconfig: enable it for machines that have PCI (q35 does)
grep -q "config VNIC" hw/net/Kconfig || cat >> hw/net/Kconfig <<'EOF'

config VNIC
    bool
    default y if PCI_DEVICES
    depends on PCI
EOF

(hw/net/ is the natural home for a NIC model. The device's QOM type name is vnic, so the guest sees it via -device vnic.)

5. Configure + build

./configure --target-list=x86_64-softmmu     # KVM is enabled by default on Linux
make -j"$(nproc)"                            # binary: build/qemu-system-x86_64

A rebuild after editing qemu-device/vnic.c is: re-cp it to hw/net/vnic.c (step 4), then make -j -C ~/qemu (or ninja -C ~/qemu/build).

6. Run the guest under the self-built QEMU + attach the device

scripts/run-qemu.sh honours two env vars:

  • QEMU= - path to the QEMU binary (default qemu-system-x86_64)
  • VNIC_DEVICE=1 - append -device vnic to the command line
sudo env QEMU="$HOME/qemu/build/qemu-system-x86_64" VNIC_DEVICE=1 VMDIR="$HOME/vnic-vm" ./scripts/run-qemu.sh

Use sudo env VAR=... (not sudo VAR=...): under sudo's default env_reset, plain VAR=value arguments are dropped, so run-qemu.sh would silently fall back to the system qemu-system-x86_64 with no -device vnic (the device then never appears in lspci). env sets the variables in the command's own environment instead. VMDIR= is passed explicitly because sudo may reset $HOME (to root's), which would otherwise point the default image dir at the wrong place. If your host user is in the kvm group you can drop sudo entirely - then the plain QEMU=... VNIC_DEVICE=1 ./scripts/run-qemu.sh prefix works as a normal shell assignment.

By default -device vnic runs the internal loopback (Phase 9), which is what the L2 test suite uses. For a real tap backend (guest <-> host traffic), see section 8.

7. Verify (loopback)

In the guest:

lspci -nn | grep 1234:11e9          # the device (not in pci.ids, so grep by ID, not by name)
sudo insmod driver/vnic-l2.ko       # probe -> vnic0 (name pinned, NET_NAME_USER)
dmesg | tail                        # ID read back as 0x564E3130; "MSI" path exercised by the suite
sudo env MODULE="$PWD/driver/vnic-l2.ko" bash tests/run-all.sh   # full Level 1 suite over real DMA

8. Phase 10 - run with a real tap backend (guest <-> host)

loopback=off + a -netdev tap makes the device bridge frames to a host tap instead of looping internally (D-30). run-qemu.sh builds this when VNIC_TAP=<ifname> is set.

Host (WSL2) - create + address the tap once (root; the tap is a point-to-point link to the guest, so give the host side an IP and let the guest take another on the same /24):

sudo ip tuntap add dev vnic-tap0 mode tap user "$USER"   # create a persistent tap you own
sudo ip addr replace 192.168.20.1/24 dev vnic-tap0
sudo ip link set vnic-tap0 up

Boot the guest on that tap (loopback=off is added by the script for the tap path):

sudo env QEMU="$HOME/qemu/build/qemu-system-x86_64" VNIC_TAP=vnic-tap0 VMDIR="$HOME/vnic-vm" \
     ./scripts/run-qemu.sh

Guest - bring vnic0 up on the tap subnet (no fictional peer / no IP-swap here - the peer is the real host):

sudo insmod driver/vnic-l2.ko
sudo ip addr replace 192.168.20.2/24 dev vnic0
sudo ip link set vnic0 up

Verify both ways:

# in the guest:   ping the host
ping -c3 192.168.20.1
# on the host:    ping the guest, and watch frames on the tap
ping -c3 192.168.20.2
sudo tcpdump -ni vnic-tap0          # should show the ICMP echo/reply in both directions

Notes: script=no,downscript=no means QEMU does not run a helper to configure the tap, so the tap must exist first (the ip tuntap step). user "$USER" lets the persistent tap be opened without root for the QEMU side, but opening /dev/net/tun + KVM still typically needs sudo here. WSL2's own NAT is separate; this tap is a host<->guest link inside WSL2. Tear down with sudo ip tuntap del dev vnic-tap0 mode tap.

8.1 Reach external servers from the guest (host as router + NAT)

By itself the tap is only a host<->guest link - the guest reaches 192.168.20.1 and nothing more, because the host's net.ipv4.ip_forward defaults to 0 (it acts as an end host, not a router) and the guest's 192.168.20.x is a private, non-routable source. To let the guest reach the outside through vnic0, the host must forward + NAT. scripts/tap-nat.sh does the host side, idempotently (re-run up after a wsl --shutdown - WSL2 does not persist this):

# host: ensure tap + ip_forward=1 + outbound MASQUERADE out the WAN iface
sudo ./scripts/tap-nat.sh up
# guest: send external traffic out vnic0, set a DNS
sudo ip route add default via 192.168.20.1 dev vnic0 metric 50
echo nameserver 1.1.1.1 | sudo tee /etc/resolv.conf
ping -c3 1.1.1.1 ; curl -sI http://example.com
# host: revert when done
sudo ./scripts/tap-nat.sh down

Caveats: this is outbound only and double-NAT (vnic0 -> tap -> host MASQUERADE -> WSL2's own NAT -> Windows -> internet) - good for driving the datapath with real, varied traffic, not for accurate throughput numbers.

Inbound is NOT automatic (MASQUERADE is one-way). For an external client to reach a service on the guest you need a per-service DNAT on the host - e.g. host :8080 -> guest :8080:

sudo iptables -t nat -A PREROUTING -i eth0 -p tcp --dport 8080 -j DNAT --to-destination 192.168.20.2:8080
sudo iptables -A FORWARD -i eth0 -o vnic-tap0 -p tcp -d 192.168.20.2 --dport 8080 -j ACCEPT

and, for a client beyond the WSL2 host (on Windows or the LAN), a Windows-side netsh interface portproxy as well, because WSL2 is itself NAT'd behind Windows. (This is the manual equivalent of what slirp's hostfwd= does for the user-mode NIC; see qemu-device-model.md section 6.)

Keeping it reproducible

  • Source of truth is qemu-device/vnic.c in this repo; the QEMU tree is a build area, not edited directly. Re-run step 4's cp (+ the meson/Kconfig hooks, which are idempotent) after each device change, then rebuild.
  • A small scripts/qemu-integrate.sh <qemu-src> can wrap step 4 (copy + ensure the meson/Kconfig lines) once the device file exists - a Phase 8 follow-up.
  • Version bumps: if you later move the guest to a different QEMU, re-pin (step 1), git checkout the new tag, re-apply step 4, rebuild. The meson/Kconfig hook rarely changes across versions, but the device API (QOM/PCI/DMA/MSI helpers) can - check against the qemu-device-model.md APIs after a bump.
  • KVM still works with a self-built QEMU (it uses /dev/kvm the same way); run-qemu.sh already picks accel=kvm when /dev/kvm is present, else falls back to TCG.
