NIC Driver Working Flow - a guided walkthrough

This is the start-here narrative for the project: how a Linux NIC driver actually works, end to end, told through VN1000. It links out to the detailed docs rather than repeating them - read it first, then follow the pointers. The goal of the whole repo is to understand the NIC driver working flow, and this walkthrough is that flow in prose.

If you want the one-paragraph version: a NIC driver is a net_device whose datapath is a pair of descriptor rings in memory shared with the device; the driver and device hand buffers back and forth using a DD ownership bit + barriers, the driver kicks the device with an MMIO doorbell, the device signals completion with an MSI, and the driver does the actual work in a NAPI poll (softirq), not in the interrupt. Everything else is detail on top of that loop.

2. The driver lifecycle: init -> probe -> open -> close -> remove

Before any packet moves, the driver goes through a fixed lifecycle. The TX/RX datapath (sections 3 and 4) is only what runs between open and close; the rest is bring-up and teardown.

module init (module_init, at insmod) - register the driver with its subsystem. - L2: pci_register_driver(&vnic_pci_driver) with an ID table ({0x1234, 0x11E9}); the PCI core then calls vnic_probe for each matching device. - L1: there is no bus, so the module builds the netdev directly in its init - there is no probe.

Why L2 has no hand-written module_init. Because init's only job is pci_register_driver, L2 uses the one-line macro module_pci_driver(vnic_pci_driver) (level2/vnic_l2_main.c). It expands to the __init/__exit entries for you - load calls pci_register_driver, unload calls pci_unregister_driver - so there is no custom init function; all device bring-up lives in vnic_probe (the PCI core calls it on a vendor/device-ID match). L1 has no bus to drive a probe, so it keeps an explicit __init (level1/vnic_l1_main.c) that does the probe's work itself - builds the netdev and starts the in-kernel engine. Same lifecycle, different trigger.

probe (vnic_probe, level2/vnic_l2_main.c) - claim the device and make the interface exist (DOWN). alloc_etherdev + SET_NETDEV_DEV + pci_set_drvdata; pin the name vnic0 (NET_NAME_USER, D-29); wire netdev_ops + netif_napi_add; set min/max MTU. Then vnic_hal_init - enable PCI, pci_iomap BAR0, pci_set_master + dma_set_mask, and dma_alloc_coherent the rings. Verify ID == 0x564E3130, then register_netdev - now ip link shows vnic0, administratively DOWN. No IRQ is requested here.

open (ndo_open -> vnic_open -> vnic_up) - triggered by ip link set up, this brings the datapath live: reset TX indices; pre-post RX buffers (vnic_rx_post, Model A, D-18); vnic_hal_irq_request (the MSI / simulated-MSI ISR); program ITR; enable the device (CTRL = EN|TXEN|RXEN); unmask interrupts (IMS); netif_start_queue. From here TX/RX run.

close (ndo_stop -> vnic_stop -> vnic_down) - triggered by ip link set down; the exact mirror of open, ordered for safety (datasheet 6.7): netif_tx_disable (stop the queue, wait out an in-flight xmit) -> CTRL = 0 (the device stops consuming) -> vnic_hal_quiesce (wait out an in-flight engine run) -> napi_disable (flush the poll) -> IMC (mask all causes) -> vnic_hal_irq_free -> drain the rings (vnic_tx_drain/vnic_rx_drain: unmap + free every buffer the driver still owns). Order matters: disable + quiesce before freeing, so nothing is still DMA-ing into memory you are about to release.

remove / exit (vnic_remove, module_exit) - the device is gone or the module is unloaded: unregister_netdev (which closes the device first if it is still up), vnic_hal_exit (free rings, unmap BAR0, release regions, disable PCI), netif_napi_del, free_netdev; module_exit -> pci_unregister_driver.

register != open. Loading the module only makes the interface exist (DOWN); ip link set up is what allocates buffers, hooks the IRQ and starts the queue (background-knowledge topic 9). vnic_up/vnic_down are also reused by ndo_change_mtu (down -> resize RX pool -> up).

3. TX, end to end (follow tx-path.svg)

  1. The stack calls ndo_start_xmit (vnic_start_xmit, core/vnic_netdev.c) with an skb.
  2. The driver makes the buffer reachable by the device - vnic_hal_map() (L2: dma_map_single; L1: the pointer itself) - and writes a TX descriptor (addr/len, status=0), keeping the skb in ring->bufs[] for later freeing (D-12 explains why the token is opaque).
  3. A dma_wmb() publishes the descriptor, then the driver rings the doorbell - vnic_hal_wr32(VN_REG_TDT, ...) (L2: a real MMIO write = a VM-exit). To amortize that exit it batches the doorbell over an xmit_more burst (D-32, the ~4.5x TX win in L-10).
  4. The device consumes TDH..TDT: it DMA-reads the descriptor and packet (L2: pci_dma_read; L1: the engine reads memory directly), then either loops it back (loopback mode, D-25) or sends it out via qemu_send_packet to the tap (D-30).
  5. The device marks the descriptor done (the DD bit) and raises a completion interrupt (L2: msi_notify, D-24; L1: a simulated-MSI tasklet, D-20), optionally rate-limited by ITR (D-33, L-11).
  6. The driver's ISR (top half) does the minimum - read ICR, mask via IMC, napi_schedule().
  7. NAPI (vnic_poll -> vnic_tx_clean, softirq, bottom half) reaps every DD descriptor: dma_rmb(), vnic_hal_unmap(), free the skb, wake the queue if it was stopped.

4. RX, end to end (follow rx-path.svg)

  1. At open, the driver pre-posts an skb into every RX descriptor (Model A, D-18; vnic_rx_post) so the device always has somewhere to write.
  2. The device DMA-writes an arriving frame into the slot at RDH (L2: pci_dma_write, from the receive() callback in tap mode), sets len + DD, advances RDH, raises RXAVAIL. If the ring is full it applies backpressure (D-19).
  3. ISR -> mask -> napi_schedule(), then vnic_poll -> vnic_rx_clean (softirq): on each DD slot (after dma_rmb()) it vnic_hal_unmap()s the buffer, skb_put + eth_type_trans, napi_gro_receive() up the stack (GRO is already in play - background topic 7.8), then refills the slot with a fresh mapped skb and rings the RDT doorbell.
  4. When the poll drains below budget, napi_complete_done() re-enables the interrupt (IMS); a cause that arrived while masked re-fires (the interrupt + polling hybrid, D-11).

5. The three levels, and the one seam (D-12)

The shared core/ (netdev ops, the TX/RX datapath, NAPI) is identical across all levels. The only thing that changes is the HAL backend behind hal/vnic_hal.h:

Level 1 (vnic-l1.ko -> vnic_loop0) Level 2 (vnic-l2.ko -> vnic0)
"the device" an in-kernel engine kthread (hal/vnic_hal_l1.c) a QEMU PCI device (qemu-device/vnic.c)
registers a fake in-memory regs[] array real MMIO on pci_iomap'd BAR0 (readl/writel)
buffers / ring the kernel pointer is the token streaming dma_map_single + coherent dma_alloc_coherent
completion IRQ a simulated-MSI tasklet (D-20) a real MSI + request_irq (D-24)
network internal IP-swap loopback (D-05) loopback (default) or a real tap backend (D-30)

Both register a single netdev with distinct names so they can be loaded and tested at the same time (D-29). The core never changed when moving L1 -> L2 except one portable RX unmap (D-26) - which is the whole point of the HAL design.

The phases (TODO.md) went: kernel-only netdev + engine + loopback datapath + hardening (L1, Phases 1-5) -> QEMU device keystone (Phase 8) -> DMA datapath (Phase 9) -> real tap (Phase 10) -> coexistence + test variant (Phase 11) -> performance + polish (Phase 12). The most instructive write-ups are the lessons (docs/lessons-learned.md), several of which are debugging detective stories:

  • L-05 - "Destination Host Unreachable" is an L2/ARP failure, not a datapath bug.
  • L-06 - Phase 8 keystone + three bring-up footguns (sudo dropping env, a stale run-qemu.sh, host DNS).
  • L-08 - tap RX wedged: a NIC must flush its backend when RX is enabled (and host-tap tcpdump is a red herring).
  • L-10/L-11/L-12 - the performance trio: doorbell batching (~4.5x TX), ITR (~60x fewer interrupts), MTU (~2.5x).
