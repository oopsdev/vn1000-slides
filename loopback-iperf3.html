Level 1 loopback - iperf3 TCP

Throughput of the Level 1 internal loopback (the engine swaps src/dst IP+MAC and feeds each TX frame back into the RX ring). Both the iperf3 client and server run on the same guest, so the number reflects client + server + engine + NAPI poll all sharing the same CPU(s) over a software loopback - it is not a NIC line-rate figure. The interesting metric here is Retr (TCP retransmissions), which exposes in-driver packet drops.

Environment

  • WSL2 + QEMU Ubuntu 24.04 guest (kernel 6.8), Level 1 driver vnic-l1.ko
  • single TX/RX queue, single NAPI, no GRO, no interrupt coalescing
  • TX ring depth: 256
  • accel (KVM/TCG), vCPUs, CPU model: to fill (lscpu, nproc, check /dev/kvm)

Method

sudo insmod vnic-l1.ko
sudo bash scripts/setup-vnic.sh        # vnic0 = 192.168.10.1/24, peer B = .2
iperf3 -s &
iperf3 -c 192.168.10.2 -t 5
ip -s link show vnic0                  # RX/TX packets + dropped

Results

RX ring RX-full policy Completion Throughput Retr Notes
256 drop engine napi_schedule 4.41 Gbit/s 745 overflow retransmits; cwnd capped by loss
4096 drop engine napi_schedule 3.62 Gbit/s 0 no drops; cache thrash (~6 MB working set)
256 backpressure engine napi_schedule (Phase 3) 6.03 Gbit/s 0 no drops + cache-friendly (commit 49dcd48)
256 backpressure tasklet MSI (Phase 4) 6.33 Gbit/s 0 full ICR/IMS protocol; no regression vs Phase 3 (commit 11544ed)

For the Phase 4 backpressure run, ip -s link show vnic0 confirmed the loopback is lossless: TX packets == RX packets (2,762,908 each), 0 dropped, 0 errors both directions - every transmitted frame looped back and nothing overflowed. (Per-cause counters like STAT_RXDROP are still worth exposing via ethtool -S / debugfs in Phase 6.)

Analysis

  • 745 retransmissions over a software loopback with no real wire meant the driver dropped frames. Order is preserved (the engine produces RX in TX order and the poll consumes in order), so the cause was loss, not reordering.
  • Confirmed: deepening the RX ring 256 -> 4096 took Retr from 745 to 0. The retransmits were RX-ring overflow: the loopback engine (producer) outran the single NAPI vnic_poll (consumer), the 256-entry ring filled and further loopback frames were dropped (STAT_RXDROP); TCP recovered by retransmitting. With losses gone, cwnd grew from ~0.47 MB (capped by loss) to ~4 MB.
  • But goodput genuinely dropped (receiver 2.57 -> 2.11 GB; 4.41 -> 3.62 Gbit/s) - this is real, not just variance (the 4096 run was steady at 3.25-4.13). Leading hypothesis: cache footprint / buffer reuse frequency. The loopback is CPU-bound (iperf client + server + engine memcpy + NAPI poll on the same vCPUs); with a 256-deep ring each RX buffer is reused every 256 packets and stays hot in cache (~0.4 MB working set), whereas a 4096-deep ring spreads ~6 MB of buffers that are reused only every 4096 packets, so the engine's memcpy and the poll's header reads miss cache far more often.
  • Backpressure wins both axes (RX 256, stall instead of drop): 6.03 Gbit/s, 0 retransmits - faster than both drop-based runs. It removes drops (so cwnd grows to ~4 MB, no retransmit overhead) and keeps the small cache-friendly ring (~0.4 MB working set, unlike the 4096 ring's ~6 MB thrash). So the answer was not "size the ring bigger" but "do not drop": convert overflow into flow control (netif_stop_queue) and keep the ring small. Adopted as the default (design-decision D-19).
  • Takeaway - ring depth is a trade-off, not "bigger is better": too small + drop -> overflow retransmits; too large -> cache thrash. Backpressure sidesteps the trade-off by never dropping, letting us keep the small (fast) ring. A real driver also leans on interrupt coalescing, GRO, and multiple queues.

This is informal data toward Phase 7 (stress + profiling). It is not a real NIC benchmark - both endpoints and the "wire" are the same kernel.
