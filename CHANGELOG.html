Fixed

  • napi_disable double-call on a failed ndo_change_mtu re-init. If vnic_up() failed after vnic_down() (e.g. jumbo RX buffer alloc failed), the interface stayed admin-up with NAPI left disabled, so the next ndo_stop called napi_disable() a second time and could hang. vnic_down() is now idempotent via a priv->up flag (set only on a fully successful vnic_up()). Found by the Phase 5.2 error/teardown audit.

  • TDH/RDH access corrected to RW-at-init across spec, device and docs. The datasheet had marked the ring heads read-only, but the driver resets them to 0 during ring setup (vnic_up writes TDH=0, vnic_rx_post writes RDH=0) before CTRL.EN. Documented them as device-owned while enabled but writable only while disabled (datasheet 4.8 / 5.5, footnote), matching Intel TDH/RDH; and the QEMU device (qemu-device/vnic.c) now enforces it - a head write while CTRL.EN is set is ignored with a LOG_GUEST_ERROR, so the engine's head can't be moved underneath it. The device also now logs LOG_GUEST_ERROR for any stray write to a read-only / reserved register (it was already silently dropped via the RW allowlist; now it is flagged for driver-bug debugging). Verified on a self-built QEMU 6.2.0 against the full Level 2 suite (tests/run-all.sh): all 6 groups pass, dmesg clean over real DMA/MSI, and the new guards stay silent (the driver never writes a head while enabled, nor a read-only register).

Changed

  • Explained why the driver keeps next_to_use/next_to_clean instead of using the hardware head/tail (datasheet 5.5 + new background-knowledge 1.3): *DH/*DT are the device's physical ring occupancy, but a completion IRQ is drained incrementally (NAPI budget), so the driver tracks its own resume point; there is no register for the driver's progress, scanning the in-memory DD avoids a per-completion MMIO read, and a separate next_to_use lets the doorbell be batched (D-32).

Changed

  • Interrupt moderation (ITR) (Phase 12, D-33): the QEMU device now honours VN_REG_ITR (the VN_CAP_MODER it already advertised) - with ITR>0 it coalesces completion MSIs to at most one per ITR microseconds via a QEMU_CLOCK_VIRTUAL timer (fire-if-gap-else-schedule); ITR=0 keeps per-cause delivery. The driver programs it at open from an itr_usecs module param (default 0 = off, so the test suite and Level 1 are unchanged; Level 1's simulated MSI ignores ITR). The interrupt-side complement to the TX doorbell batching.
  • TX doorbell moderation (Phase 12, D-32): core/vnic_netdev.c's vnic_start_xmit now batches the VN_REG_TDT doorbell over an xmit_more burst (ring once per burst / on a full ring) instead of once per packet, cutting the per-packet MMIO VM-exit that capped Level 2 TX (~350 Mbit/s, L-09). A tx_coalesce module param (default on, runtime-writable) toggles it for A/B measurement. Level-independent and correct for both levels (the kernel clears xmit_more on the last skb of a burst; a full ring forces the doorbell). No behaviour change on Level 1 beyond fewer engine wakeups.
  • Per-level test subnet so both drivers can be tested at once (Phase 11): tests/lib.sh now derives LOCAL_IP/PEER_IP from the level (L1 192.168.10.x, L2 192.168.11.x) and exports them to setup-vnic.sh, so vnic_loop0 (L1) and vnic0 (L2-loopback) can be brought up and tested simultaneously without an IP/route clash. All still env-overridable. tests/README documents the "both at once" workflow.
  • Distinct netdev names so Level 1 and Level 2 can be loaded at once: Level 1 (the in-kernel loopback engine) now registers vnic_loop0; Level 2 (the QEMU device) keeps vnic0. The module names were already distinct (vnic-l1.ko / vnic-l2.ko), so both drivers can coexist in one guest. L2 marks its name NET_NAME_USER so systemd-udev's predictable-naming keep policy does not rename the PCI netdev to a slot name (e.g. enp0s3). The test harness derives the interface from the level (tests/lib.sh: L1 -> vnic_loop0, L2 -> vnic0, still overridable) and scripts/setup-vnic.sh is interface-parameterized (run both on separate subnets via IFACE=/LOCAL_IP=/PEER_IP=). README / tests/README / architecture updated.
  • Convention sweep (D-16 / MSG_* rule): the Phase 8 source (qemu-device/vnic.c, hal/vnic_hal_l2.c, level2/vnic_l2_main.c) now documents its structs/functions in kernel-doc style (/** for the struct, /* func() - ... @param ... Return: */ for static functions, matching core/vnic_netdev.c); level1/vnic_l1_main.c's last hardcoded log string moved to a MSG_LOADED constant. No behaviour change.

The shared device spec keeps VN_* / struct vn_desc; level-independent core logic stays vnic_*.
