Linux NIC Simulation - System Architecture

1. Goal and scope

Simulate a NIC and its Linux driver in software, ending up as a real NIC inside QEMU. The goal is to build out the core concepts of a PCIe device driver - queue, DMA, completion, interrupt, NAPI, and netdev integration - in a phased, demonstrable way.

Core mental model: the driver uses the hardware, the device plays the hardware, and the two mirror each other across the MMIO / DMA / IRQ boundary. Every driver action has a matching device reaction.

Four layers, do not conflate them:

  1. Guest vnic driver - the .ko this project writes, driving the virtual NIC. <- the real driver
  2. Device model - Level 1 is an engine kthread inside the same module; Level 2 is the QEMU PCI device. <- plays the hardware
  3. QEMU netdev backend - tap / user, moves packets to the host. Off-the-shelf.
  4. Host's real NIC driver - unrelated to this project.

Level 1 is the focus of this project's learning goal. It builds the entire driver working flow - netdev registration, the TX/RX descriptor rings, NAPI, DMA-like buffer handling, a simulated completion/interrupt, and an internal loopback - as a pure kernel module with no PCI bus and no QEMU. The engine kthread plays both the "device" and the "wire": it consumes each transmitted frame and loops it back as a receive (swapping MAC + IPv4 addresses), so one host exercises the full TX -> RX path. The local address A and the fictional peer B are the same machine; the IP swap makes a frame sent to B return as a receive from B. Level 2 then swaps the in-kernel engine for a real QEMU PCI device (real MMIO / DMA / MSI) without touching the driver core. Section 6 details the Level 1 datapath, section 7 the Level 2 datapath.

2.1 The HAL seam: layering the driver so most of it is reused across levels

Intent. The plan is one driver that grows progressively more real - Level 1 (kernel-only fake engine) -> Level 2 (QEMU PCI device, real MMIO/DMA/MSI) -> Level 3 (FPGA, bonus). The learning goal is to understand the driver working flow, not to rewrite it per level. So the design isolates everything hardware-specific behind a single seam - the HAL (hardware-access layer, driver/hal/vnic_hal.h) - and writes the part worth learning (netdev, rings, the DD handshake, NAPI) once, reused unchanged at every level (decision D-12).

How the split was decided. Sort everything the driver does by one question: does it change when the "device" goes from a fake in-kernel engine to a real PCI device?

  • No -> driver/core/ (level-independent). net_device_ops (open/stop/xmit/stats/change_mtu), ring management and the software cursors, the DD ownership handshake and its barriers, the NAPI poll, TX/RX cleanup. This logic only ever reads/writes a register or hands a buffer to the device - both abstract operations - so it does not care how either is actually done.
  • Yes -> driver/hal/vnic_hal_l{1,2}.c (level-specific). How a register access happens (a fake array vs readl/writel), how a buffer is made device-visible (a pointer vs dma_map_single), how ring memory is obtained (kcalloc vs dma_alloc_coherent), how a completion is delivered (a tasklet vs a real MSI), and bring-up/teardown.

The seam is drawn at exactly the smallest set of operations through which the core touches the device - nothing else crosses it, so the backend stays thin and the core stays whole.

The interface (12 functions, five groups) - the figure above shows this as a table; in text, driver/hal/vnic_hal.h:

Group HAL functions Level 1 backend Level 2 backend
Register access vnic_hal_rd32 / vnic_hal_wr32 read/write a fake register array; wr32(TDT/RDT) wakes the engine kthread readl / writel on BAR0
Ring memory vnic_hal_ring_alloc / vnic_hal_ring_free kcalloc (desc_dma unused) dma_alloc_coherent (sets desc_dma)
Buffer mapping vnic_hal_map / vnic_hal_unmap / vnic_hal_mapping_error token = the kernel pointer (never fails) token = a dma_map_single bus address (can fail)
Interrupt vnic_hal_irq_request / vnic_hal_irq_free start/stop the simulated-MSI tasklet pci_alloc_irq_vectors + request_irq
Lifecycle vnic_hal_init / vnic_hal_exit / vnic_hal_quiesce alloc fake regs + start engine kthread; quiesce = wait out the kthread enable PCI + ioremap BAR0 + set DMA mask; quiesce = synchronize_irq

Two deliberate choices keep this surface tiny:

  • No separate doorbell or mask functions. Ringing TDT/RDT and masking via IMS/IMC are just vnic_hal_wr32() of those offsets - the register write is the doorbell - so the datapath needs no extra HAL verbs.
  • The device token is opaque. vnic_hal_map() returns a u64 the core stores in vn_desc.addr and later passes back to vnic_hal_unmap(); the core never dereferences it. That single rule is what lets the token be a raw pointer on L1 and a DMA address on L2 without the datapath knowing the difference (the real skb is kept in ring->bufs[] for completion). See vnic_hal.h and D-12.

The payoff. Because the seam is exactly right, all of core/ is byte-for-byte identical across levels - vnic_start_xmit, vnic_tx_clean, vnic_rx_clean, vnic_poll, the ring code, vnic_netdev_ops. What differs per level is only the two hal/vnic_hal_l{1,2}.c files (the same 12 functions implemented two ways) plus a thin level{1,2}/ entry (L1 hand-builds the netdev in module_init; L2 is a module_pci_driver whose vnic_probe does it - section 2's lifecycle). Selection is compile-time (Makefile LEVEL=1|2 links exactly one backend); a runtime ops table was rejected because a single module only ever needs one level (D-12).

3. Device / driver boundary principle

Even though Level 1 lives in the same kernel module, the "device engine" is cleanly isolated and driven through the fake-register interface (e.g. writing DOORBELL triggers the engine, not a direct function call). This makes the move to QEMU in Level 2 a "relocation", not a rewrite, and leaves the driver-side code almost untouched.

The shared spec header include/vnic_regs.h defines the register offsets and descriptor layout; both the driver and the qemu-device include the same one.

4. Register and descriptor spec

The register map, bit definitions and descriptor format are authoritatively defined in vn1000-datasheet.md (Intel-style MMIO BAR0); include/vnic_regs.h is derived from it. Only a summary here, to avoid the two docs drifting apart:

  • identity / control: ID / VERSION / CAP / CTRL / STATUS / MAC_LO / MAC_HI / MAXFRM
  • TX ring: TDBAL / TDBAH / TDLEN / TDH / TDT (writing TDT is the doorbell)
  • RX ring: RDBAL / RDBAH / RDLEN / RDH / RDT
  • interrupt: ICR (RC) / ICS / IMS (RW1S) / IMC (RW1C) / ITR
  • statistics: STAT_*

Compared with Linux e1000e: TDBAL/TDH/TDT, RDBAL/RDH/RDT, ICR/IMS/IMC.

5. Descriptor format (virtio style, no chaining for now)

struct vn_desc {            /* shared 16-byte TX/RX layout; full bit defs in the datasheet */
    u64 addr;   /* buffer DMA / guest physical address */
    u32 len;    /* TX: packet length; RX: bytes received */
    u16 flags;  /* TX: command (EOP/RS); RX: result (EOP/ERR) */
    u16 status; /* device-written: DD (descriptor done) */
};

Bit definitions for flags / status are in vn1000-datasheet.md section 5.

Ring head / tail semantics (Intel-style; authoritative detail in the datasheet):

  • The driver owns the tail (TDT/RDT) and advances it to hand descriptors to the device (writing the tail is the doorbell); the device owns the head (TDH/RDH) and advances it as it processes. The device works on the descriptors in [head, tail); head == tail means it has nothing to do (TX empty / RX has no free buffers).
  • TX: the driver is the producer (fills frames, advances TDT); the device is the consumer (transmits, advances TDH).
  • RX: the driver produces empty buffers (posts them, advances RDT); the device fills them (writes received data, advances RDH).
  • Per-descriptor ownership is carried by the DD bit in status (status == 0 = device-owned, to process; DD = driver-owned, done). This is what the driver and engine actually test; head / tail are the cursors.

6. Level 1 datapath (in-kernel engine, internal loopback)

Level 1 has no PCI bus and no wire. The engine kthread plays both the device and the wire: it consumes each transmitted frame and loops it back into the RX ring (swapping the Ethernet MAC and IPv4 src/dst), so one host drives the full TX + RX + NAPI flow. TX and RX are therefore coupled - one engine pass does both - so the engine is described first (6.1), then the TX (6.2) and RX (6.3) halves it drives.

6.1 Level 1 engine (loopback + backpressure)

The engine is a kthread. It sleeps in wait_event_interruptible(&h->wq, ...) and is woken by a doorbell - it is not a busy loop and uses no CPU when idle. One run walks the pending TX descriptors and, per frame, loops it into RX or stalls when RX is full:

flowchart TB
  wake["woken (TDT / RDT)"]
  loop["next TX desc"]
  full{"RX slot free?"}
  deliver["loopback into RX, RX DD"]
  txdone["mark TX DD"]
  stall["stall (backpressure)"]
  sched["napi_schedule"]
  wake --> loop
  loop -->|more| full
  loop -->|none| sched
  full -->|yes| deliver
  deliver --> txdone
  txdone --> loop
  full -->|no| stall
  stall --> sched
  • Wakeup condition: the wake comes from wake_up_interruptible(&h->wq) inside vnic_hal_wr32(). The TX doorbell (TDT) always wakes the engine (new frames to consume); the RX refill doorbell (RDT) wakes it only when tx_stalled is set (a backpressure resume). So an RDT write during normal operation is silent, and only resumes a stalled engine.
  • Deliver: copy the frame into the RX slot at RDH, swap MAC + IPv4, set RX DD, advance RDH; then mark the TX descriptor DD and advance TDH.
  • Backpressure (D-19): if the RX slot at RDH is still DD (the driver has not drained it), the engine stalls - stops consuming TX, leaves the descriptor un-DD. The TX ring then fills and netif_stop_queue throttles the sender (TCP slows) instead of dropping and retransmitting.
  • Resume: the napi_schedule after every run guarantees a poll drains RX; rx_clean refilling RX rings the RDT doorbell, which wakes the stalled engine. A poll always follows a stall, so it cannot deadlock. Measured best on both axes - see lessons-learned L-02.

6.2 Level 1 TX flow

sequenceDiagram
  participant S as Net stack
  participant D as vnic driver
  participant E as L1 engine
  Note over D: `setup - **netif_napi_add()** registers vnic_poll, **napi_enable()** in open`
  S->>D: ndo_start_xmit skb
  D->>D: skb_linearize, ring-full check, fill TX desc (addr = skb-data token)
  D->>E: write TDT doorbell - wakes the engine kthread
  E->>E: read TDH..TDT, loopback each frame into RX (see 6.1)
  E->>D: `mark TX DD, **napi_schedule()**`
  D->>D: `**vnic_poll()** -> tx_clean - **napi_consume_skb()**, wake queue`
  D->>D: `**napi_complete_done()** when RX work lt budget`
  • TX is zero-copy at Level 1: the descriptor addr is just the skb->data pointer (the token), no DMA mapping. skb_linearize() makes the frame contiguous so the engine can copy it in one go.
  • Ring-full applies backpressure (netif_stop_queue); the queue is woken from tx_clean. A full RX ring also stalls TX - see 6.1.

6.3 Level 1 RX flow

RX buffers are pre-posted - one skb per RX descriptor, allocated in ndo_open (Model A, D-18) - because the device needs somewhere to write before a frame arrives.

sequenceDiagram
  participant E as L1 engine
  participant D as vnic driver
  participant S as Net stack
  Note over D,E: RX skbs pre-posted, one per desc (Model A, D-18)
  E->>E: loopback frame into pre-posted RX buffer, swap MAC + IPv4, set RX DD
  E->>D: `**napi_schedule()**`
  D->>D: `**vnic_poll()** -> rx_clean (up to budget)`
  D->>D: skb already posted - skb_put(len) + eth_type_trans (dst MAC = ours -> PACKET_HOST)
  D->>S: `**napi_gro_receive()** - delivered locally`
  D->>E: alloc fresh skb, refill desc, ring RDT (also resumes a stalled engine)
  Note over D: `**napi_complete_done()** when work lt budget`
  • The loopback swaps the Ethernet MAC (dst becomes our own address, so eth_type_trans marks it PACKET_HOST and the stack delivers it locally) and the IPv4 src/dst (so it looks like it came from peer B). Swapping only reorders checksummed fields, so the IP and TCP/UDP checksums stay valid (ports/payload untouched). See design-decisions D-05.
  • On completion the skb is already populated, so the poll just skb_put + eth_type_trans, then allocates a fresh skb to refill the slot.

7. Level 2 datapath (QEMU PCI device)

Level 2 (implemented, Phases 8-12) keeps the same driver core; only the HAL backend changes - real MMIO instead of fake registers, streaming DMA (dma_alloc_coherent rings + dma_map_single buffers) instead of a pointer token, a real MSI instead of the engine's napi_schedule, and the QEMU device instead of the in-kernel engine. The default device mode is the internal IP-swap loopback (D-25/ D-30, what the test suite uses); loopback=off + a -netdev tap reaches a genuine remote host. The RX buffer model (pre-posted skb, Model A, D-18) and the NAPI half are identical to Level 1. The Phase 12 performance knobs - TX doorbell batching (D-32) and ITR interrupt coalescing (D-33) - tune this same path.

8. Interrupt and polling model

  • the top half does the minimum: mask that cause + napi_schedule()
  • the bottom half / NAPI poll does the work: up to POLL_BUDGET (64) descriptors per call
  • the baseline is already an interrupt + polling hybrid (interrupt arrives -> IMC mask and poll -> IMS re-enable); a forced fixed-mode comparison is an advanced item (TODO A2)
  • interrupt moderation: cut interrupt overhead at high packet rates, trading latency for throughput

8.1 How vnic_poll gets called (NAPI trigger chain)

vnic_poll is never called directly. It runs only after something calls napi_schedule(), which hands the device off to the NET_RX_SOFTIRQ machinery:

flowchart TB
  trig["Trigger source"]
  sched["napi_schedule()"]
  soft["net_rx_action()"]
  poll["vnic_poll()"]
  work["clean TX, receive RX"]
  done["napi_complete_done()"]

  trig --> sched
  sched -->|softirq| soft
  soft --> poll
  poll --> work
  work -->|work lt budget| done
  done -->|wait| trig

Reading the chain (details kept here so the boxes stay short):

  • Trigger source - whatever calls napi_schedule(). It is not part of NAPI and differs by level (see the table below): on Level 1 it is software with no real interrupt - the engine kthread (Phase 2), later a simulated MSI handler (Phase 4); on Level 2 it is the real MSI hardirq handler registered with request_irq(), exactly like a physical NIC.
  • napi_schedule() sets the NAPI scheduled (STATE_SCHED), adds it to the per-CPU poll list, raises NET_RX_SOFTIRQ, and returns immediately (the top half stays short).
  • net_rx_action() is the softirq handler; it calls the registered vnic_poll().
  • vnic_poll() runs in softirq context and processes up to budget (per-NAPI weight, default 64) per call: it cleans TX completions first (not counted against budget), then (Phase 3+) receives RX bounded by budget.
  • The wrap-up rule (what budget is for): the poll returns work_done = the number of RX packets processed.
  • work_done < budget -> RX is drained -> call napi_complete_done() to end the poll and re-arm the interrupt (IMS), then wait for the next trigger.
  • work_done == budget -> there may be more -> do not complete, just return budget; the NAPI stays on the poll list and net_rx_action() polls it again. This is how NAPI backs off interrupts and switches to pure polling under load.
  • Why budget exists: poll runs in softirq and would otherwise monopolise the CPU under a flood. Two caps apply - the per-NAPI weight (one poll call) and the per-CPU netdev_budget (300) / netdev_budget_usecs (2 ms) that bound a whole net_rx_action round across all NAPIs - keeping device polling fair and softirq latency bounded.

napi_enable() (in ndo_open) only makes the device eligible to be scheduled - it does not itself call poll. With no trigger source, vnic_poll stays idle. The trigger source by stage:

Stage Trigger source (what calls napi_schedule) Notes
Phase 1 (L1) nothing no engine, no interrupt - vnic_poll is effectively never run
Phase 2 (L1) the in-kernel engine kthread, after marking TX descriptors DD software, no real IRQ; interim completion signal; vnic_poll cleans TX
Phase 3 (L1) the same engine kthread, after looping each frame into RX (and completing TX) vnic_poll now cleans TX and receives RX; a stalled engine is resumed via the RDT refill doorbell (backpressure, D-19)
Phase 4 (L1) a simulated MSI handler (tasklet, softirq): read ICR -> IMC mask -> napi_schedule still software, but now with real interrupt-register (ICR/IMS/IMC) semantics - see 8.2, D-20
Level 2 the real MSI hardirq handler registered with request_irq() a genuine hardware interrupt - the production path on a real NIC

Because vnic_poll runs in softirq context it must not sleep: the ring lock is taken with plain spin_lock() (not _bh, already in softirq), and skbs are freed with napi_consume_skb().

8.2 Completion signalling - Phase 3 vs Phase 4

Phase 3 took a shortcut: the engine called napi_schedule() directly after a run. There were no interrupt registers in the path - simple, but it did not model how a real device signals the CPU, and the ICR/ICS/IMS/IMC registers existed but were unused.

Phase 4 is the current behaviour: the real interrupt protocol plus a simulated MSI. The engine (still a kthread - it is the device) no longer pokes NAPI; it sets an interrupt cause and "fires the MSI", and a separate handler does the ack/mask/schedule. The handler runs as a tasklet (softirq context - cannot sleep, runs deferred, separate from the engine), the closest software analogue to a hardware interrupt's handler. (Not a workqueue/kthread, which would run in process context; see design-decisions D-20.)

sequenceDiagram
  participant E as engine (device, kthread)
  participant I as MSI handler (tasklet)
  participant P as vnic_poll (softirq)
  E->>E: work done - set cause via ICS (ICR bit)
  E->>I: fire MSI - only if not masked (IMS)
  I->>I: read ICR (read-to-clear), mask cause via IMC
  I->>P: napi_schedule
  P->>P: clean TX, receive RX (up to budget)
  P->>E: napi_complete_done, re-enable cause via IMS

Why the IMS/IMC mask matters: while the poll is draining, the cause stays masked, so engine completions accumulate in ICR without re-firing the MSI; the poll re-enables IMS only when it completes. That is exactly the real NIC interrupt + polling hybrid (and the basis of NAPI backing off interrupts under load). At Level 2 this tasklet is replaced by the real MSI hardirq handler registered with request_irq(); the vnic_poll half is unchanged.

How the tasklet is actually triggered. There is no real interrupt line; the "MSI is raised" by calling tasklet_schedule(). The single fire rule lives in the Level 1 vnic_hal_wr32():

on a write to ICS (raise a cause) or to IMS (unmask a cause), if (ICR & IMS) != 0 then tasklet_schedule(&msi).

So the tasklet has exactly two trigger sites, both inside vnic_hal_wr32() under the irq_lock:

  1. engine raises a cause - vnic_hal_wr32(ICS, TXDONE|RXAVAIL) at the end of an engine run. If that cause is unmasked (IMS set), the MSI fires immediately. If it is masked (a poll is in progress), the bit just stays pending in ICR.
  2. poll re-arms - vnic_hal_wr32(IMS, TXDONE|RXAVAIL) in vnic_poll after napi_complete_done. If a cause went pending in ICR while masked, unmasking it here re-fires the MSI, so the device gets polled again. This closes the classic NAPI race (work that arrived during the poll is not lost).

ICR is read-to-clear: the tasklet's vnic_hal_rd32(ICR) returns the pending causes and clears them in one step, so each cause is handled once. All four of ICR set, ICR read-clear, IMS set and IMS clear take irq_lock, because the engine (kthread) and the tasklet/poll (softirq) race on them. Resuming a stalled engine is a separate path (the RDT doorbell wakes the kthread) and does not go through the MSI.

9. Locking model

  • the TX ring uses tx_ring.lock: producer ndo_start_xmit (process ctx) vs consumer vnic_tx_clean (NAPI softirq) are not serialised by the stack, so the lock guards them; xmit takes it with spin_lock_bh() to also fence the local softirq, vnic_tx_clean with plain spin_lock() (already in softirq). The queue stop (xmit) and wake (tx_clean) decisions both run under it, so there is no missed wakeup.
  • the RX ring takes no driver lock: vnic_rx_clean is its only accessor and NAPI serialises polls; the device is the RX producer, so rx_ring.next_to_use is unused. (see D-23)
  • driver submission and device-engine consumption form a producer / consumer relationship; the DD-bit ownership handshake is ordered by dma_wmb()/dma_rmb() (D-21), not by these locks
  • no sleeping in atomic context (interrupt / holding a spinlock): mind GFP_ATOMIC vs GFP_KERNEL

11. netdev integration (shared by Level 1 and Level 2)

Implement net_device_ops: ndo_open / ndo_stop / ndo_start_xmit / ndo_get_stats64. Create the netdev with alloc_etherdev + register_netdev; register vnic_poll with netif_napi_add. Level 1 and Level 2 share the same netdev driver; the only difference is whether an engine or QEMU sits behind the rings. The two name their netdev differently - Level 1 vnic_loop0, Level 2 vnic0 - so both drivers can be loaded in one guest at the same time.

Packet egress (how to ping / test TCP/UDP):

  • Level 1 internal loopback: the engine loops a transmitted frame back into its own RX ring with src/dst IP swapped (ports kept). Assign vnic_loop0 a local IP A; the fictional peer B is on the same subnet (non-local). ping B, or server on 0.0.0.0 + client to B (iperf), tests ICMP / TCP / UDP - swapping the IP only (not the ports) keeps every checksum valid.
  • Level 2 real ping to host: ndo_start_xmit -> TX ring -> QEMU device -> qemu_send_packet -> backend (tap/user) -> host; the return path goes MSI -> vnic_poll -> netif_receive_skb.

Compared with e1000e: ndo_start_xmit = e1000_xmit_frame, poll = e1000e_poll, RX = e1000_clean_rx_irq.
