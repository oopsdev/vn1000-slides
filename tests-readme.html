tests/ - VN1000 automated tests (Level 1 & Level 2)

Concurrency / edge / lifecycle tests for the Phase 5 hardening gate. They drive the live netdev (internal loopback), so they run in the QEMU guest as root, after the module is built. The harness follows the level: the default targets Level 1 (vnic-l1.ko -> vnic_loop0); point MODULE= at vnic-l2.ko to run the same suite on Level 2 (vnic0, needs the self-built QEMU with -device vnic).

# in the guest:
cd driver && make                 # build vnic-l1.ko (+ vnic-l2.ko via: make l2)
sudo bash tests/run-all.sh        # Level 1 (default): vnic-l1.ko / vnic_loop0
sudo bash tests/03-concurrency.sh # or one test on its own

# Level 2 (the netdev name + iface are auto-derived from the module):
sudo env MODULE="$PWD/driver/vnic-l2.ko" bash tests/run-all.sh

run-all.sh tees the whole run to a timestamped log under results/ (results/tests-<YYYYmmdd-HHMMSS>.log, gitignored) and prints the path at the end; override with LOG=/path.log. A single test prints to the terminal only - redirect it yourself (sudo bash tests/05-edge.sh 2>&1 | tee results/edge.log).

Each script is self-contained (loads the module, runs, unloads) and exits non-zero on any failure, so run-all.sh is CI-gateable. Prereqs: iperf3 (its server auto-starts at boot in this guest), and jq or python3 for the iperf JSON checks (Retr / lost). Missing optional tools SKIP, never fail.

Script Group What it stresses
01-smoke.sh A load -> ping (lossless) -> iperf3 TCP (low Retr + TCP-MIB; 0 device drops gate) -> stats (0 drop/err) -> unload
02-lifecycle.sh B insmod/rmmod + up/down cycles -> init/exit, ring alloc/free, rx_post/drain leaks
03-concurrency.sh C parallel iperf3 (xmit vs clean; 0 device drops gate, Retr reported); down/up storm under flood; rmmod under live traffic
04-ringfull.sh D iperf3 UDP -b 0 backpressure / stop-wake; pktgen + tiny-ring are SKIP stubs
05-edge.sh E,H packet-size edges (runt..near-MTU), fragmented 65000B, MTU raise without splat; rmmod-while-up; double insmod; ping-while-down
06-faultinject.sh F RX alloc-failure paths via the rx_alloc_fail knob: F1 refill recycle + rx_dropped, F2 open -ENOMEM, F3 failed change_mtu re-init then down must not hang (the priv->up guard)

run-all.sh runs 01-06 (the loopback datapath). 07-tap.sh is separate - it is the Level 2 tap smoke test (real host peer, no IP-swap): ping the host, iperf3 both ways, optional external reach. It needs a different QEMU launch (VNIC_TAP=..., loopback=off) plus a host-side tap + iperf3 -s, so it is run on its own and SKIPs any missing prereq:

# in the guest, under a tap-mode QEMU (see docs/qemu-build.md section 8):
sudo env MODULE="$PWD/driver/vnic-l2.ko" bash tests/07-tap.sh

Knobs

  • MODULE=/path/to/vnic-l1.ko (default ../driver/vnic-l1.ko; use .../vnic-l2.ko for Level 2)
  • IFACE / LOCAL_IP / PEER_IP default from the level and are overridable: L1 -> vnic_loop0 on 192.168.10.1/24 (peer .10.2); L2 -> vnic0 on 192.168.11.1/24 (peer .11.2). Distinct subnets so both levels can be set up and tested at the same time.
  • ITERS=N to lengthen the cycle/storm tests

Both drivers at once

The two .kos and netdevs are distinct (vnic-l1.ko/vnic_loop0, vnic-l2.ko/vnic0), and the harness auto-picks a separate subnet per level, so they can be loaded together and tested independently (each test only loads/unloads its own module):

# Level 1 (always available; in-kernel engine)
sudo bash tests/run-all.sh
# Level 2 (needs the self-built QEMU + -device vnic; loopback mode)
sudo env MODULE="$PWD/driver/vnic-l2.ko" bash tests/run-all.sh

Run them back-to-back, or concurrently in two shells - they use different interfaces/subnets so there is no L3 clash. (The single auto-started iperf3 server is shared, so concurrent TCP tests serialise on it; lib.sh already retries on "server busy".)

The real acceptance gate

For "passes concurrency / edge tests" (Phase 5 Output), run the suite on a guest kernel built with lockdep + KASAN + kmemleak (and CONFIG_FAULT_INJECTION + FAILSLAB for group F). The scripts already scan dmesg for BUG/WARN/KASAN after every step and scan kmemleak where present; on a debug kernel those checks are what actually catch races / use-after-free / leaks.
