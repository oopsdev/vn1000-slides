1. The Linux NIC datapath (RX and TX, end to end)

The big picture first: how a packet moves across the three components - the NIC, the Host Memory (the descriptor ring + packet buffers) and the CPU (Driver) - and how control of each descriptor flips between NIC Owned and Driver Owned. RX (receive) and TX (transmit) are mirror images of the same ownership handshake. The later topics zoom into pieces of this: NAPI (topic 2) is the interrupt/poll steps, deferred execution (topic 3) is the ISR/poll contexts, and memory barriers (topic 4) are what make the ownership handshake safe.

Both paths are split into the same five steps (A init, B the data movement, C/D the interrupt + NAPI handling, E cleanup/recycle); the per-step labels (A1, B2, ...) match the numbered arrows in the figures.

1.1 The RX path

Figure 1 - the RX path as a component / data-flow diagram. Each descriptor's fill encodes its ownership (blue = NIC Owned, green = Driver Owned); it flips to Driver Owned at B3 and back to NIC Owned at E1. The sequence diagram below shows the same A-E steps ordered in time.

sequenceDiagram
  participant D as CPU (Driver)
  participant M as Host Memory
  participant N as NIC

  Note over D,N: Step A - driver init (set up the Rx ring)
  D->>M: A1 alloc + map Rx descriptor ring + packet buffers
  D->>M: A2 write buffer phys addr into descriptors (status = NIC Owned)
  D->>N: A3 write ring base + head/tail to NIC CSR

  Note over D,N: Step B - packet arrival + DMA
  N->>N: B1 Ethernet frame arrives, into NIC Buffer FIFO
  N->>M: B2 DMA-write raw packet into the Rx packet buffer
  N->>M: B3 update descriptor (len, status = Driver Owned)

  Note over D,N: Step C - interrupt + NAPI schedule
  N->>D: C1 MSI-X interrupt over PCIe
  D->>N: C2 ISR acks + masks this IRQ in the CSR
  D->>D: C3 schedule NAPI softirq poll (ksoftirqd)

  Note over D,N: Step D - NAPI poll + deliver upstream
  D->>M: D1 napi_poll scans descriptors for Driver-Owned valid packets
  D->>M: D2 read packet data from the Rx packet buffer
  D->>D: D3 wrap into sk_buff, netif_receive_skb to IP / TCP / UDP
  D->>N: D4 driver unmasks the IRQ after napi_complete_done (only once the ring is drained)

  Note over D,N: Step E - cleanup + recycle
  D->>M: E1 clear descriptor fields, set status = NIC Owned
  D->>N: E2 write updated Tail to NIC CSR (recycle)

Step A - driver init (ring buffer setup) - A1: the driver allocates and DMA-maps the Rx descriptor ring and the packet buffers in kernel space. - A2: it writes each packet buffer's physical (bus) address into the matching descriptor and marks the descriptor NIC Owned (control handed to the NIC). - A3: it programs the ring base/length and the head/tail pointers into the NIC's CSR (control/status registers) so the hardware knows where the free buffers are.

Step B - packet arrival + DMA - B1: an Ethernet frame arrives at the NIC and enters the NIC's internal buffer FIFO. - B2: the NIC's DMA engine reads the current descriptor and DMA-writes the raw packet data straight into the corresponding Rx packet buffer in Host Memory. - B3: when the DMA completes, the NIC updates the descriptor (writes the length, flips status to Driver Owned - control returned to the driver).

Step C - interrupt + NAPI schedule - C1: the NIC raises a hardware interrupt (MSI-X) to the CPU over the PCIe bus. - C2: the CPU enters the NIC's ISR and, to avoid an interrupt storm, acknowledges and masks this NIC's interrupt in the CSR (so a flood cannot keep re-interrupting while we poll). - C3: the ISR then schedules the NAPI softirq poll (e.g. on ksoftirqd) and returns - the heavy lifting is deferred out of hardirq context.

Step D - NAPI poll + deliver upstream - D1: the NAPI poll routine runs and reads descriptor status directly from Host Memory, looking for Driver-Owned, valid descriptors. - D2: it reads the packet data out of the corresponding Rx packet buffer. - D3: it wraps the data into the kernel's standard sk_buff and calls netif_receive_skb to hand it up the Linux protocol stack (IP / TCP / UDP), up to budget packets. - D4: once the poll has drained the ring (work_done < budget), the driver calls napi_complete_done - which ends the NAPI polling state and returns true if it is safe to re-arm - and then the driver itself re-enables (unmasks) the device interrupt by writing the NIC's interrupt-enable register (e.g. IMS). napi_complete_done is generic NAPI core and does not touch the hardware mask; the register write is the driver's. If budget was exhausted instead, the poll returns without unmasking and NAPI stays scheduled to keep polling. This mask (C2) / unmask (D4) pair is exactly what bounds the interrupt rate under load.

Step E - cleanup + recycle - E1: the driver clears the consumed descriptor fields and marks the status back to NIC Owned. - E2: it writes the updated Tail pointer into the NIC CSR, telling the NIC fresh empty slots are available - completing the ring recycle.

The recurring idea is descriptor ownership: a descriptor is either NIC Owned (the device may write a packet into its buffer) or Driver Owned (the device is done; the driver must process and then recycle it). The buffers are posted before any packet arrives, because the NIC needs a destination to DMA into.

How VN1000 maps to this: it is exactly the Level 2 path. Level 1 swaps the NIC + DMA for an in-kernel engine kthread (loopback memcpy) and the MSI-X for a tasklet, but the descriptor ownership is the same - the DD bit is the "NIC Owned (0) / Driver Owned (DD)" status. See architecture.md section 6 / 7.

When the refill allocation fails (inside step E). Step E refills each consumed slot with a fresh buffer so the device always has somewhere to DMA. The replacement must be allocated before the received frame is handed up the stack - once delivered, that skb belongs to the stack and its buffer can no longer be DMA'd into. So if the allocation (or its DMA mapping) fails, the slot has only its one buffer and no replacement, forcing a choice:

  • deliver the frame but cannot refill the slot - with no buffer, the slot cannot be published back to the device (advancing the tail RDT over a buffer-less slot would point the device at freed memory). So RDT stops keeping up with the device's head, and the pool of posted RX buffers drains by one per failure until the device runs out (RXEMPTY) and RX stalls. (The descriptor array does not resize - it is the count of posted buffers that drains.) With a single next_to_clean cursor nothing records that the slot owes a buffer, so it is never revisited - the loss does not heal.
  • keep the old buffer in the slot and drop the frame - the slot stays published, the posted pool stays full, and RX is self-healing (the next successful alloc refills normally). You lose exactly one packet (rx_dropped), recovered by TCP retransmit or tolerated by UDP.

Every mainstream NIC driver (e1000 / igb / ixgbe) takes the second: keeping the ring full beats delivering one more packet. The invariant "every RX slot always holds a mapped buffer" is what keeps RX alive, so it is never traded away for a single delivery - which is why the frame is not sent up when its replacement cannot be allocated. Recycling is also the simplest, lowest-cost option: it keeps the ring full with no extra bookkeeping (no deficit counter, no second cursor, no separate refill path), and a transient failure heals on the next poll. A driver that prefers to compensate later can use deferred / lazy refill (ixgbe): a second cursor records the deficit and a later poll retries the allocation, re-posting the slot once it succeeds - recovering the capacity at the cost of that bookkeeping. (In VN1000 this is the project decision D-18 / D-22, which keeps the simple recycle - deferred refill is an advanced TODO; the fault-injection test F1 forces 1-in-3 refill failures and asserts rx_dropped rises while rx_errors stays 0 and the datapath recovers.)

1.2 The TX path

Figure 2 - the TX path, the transmit mirror of Figure 1. On the way out the data flows CPU -> Host Memory -> NIC, and the descriptor's ownership flips the opposite way - the driver hands a filled descriptor to the NIC at B3 (the doorbell) and the NIC hands it back at C3 (DD done).

Step A - driver init (ring buffer setup) - A1: the driver allocates and DMA-maps the Tx descriptor ring; the descriptors start empty and Driver Owned (the driver is free to fill them). - A2: it programs the ring base/length and the head/tail pointers (TDH/TDT) into the NIC's CSR.

Step B - transmit request (ndo_start_xmit) - B1: the protocol stack hands a packet down to the driver as an sk_buff, calling the driver's ndo_start_xmit. - B2: the driver DMA-maps the skb's data and writes the buffer physical address + length into the next free Tx descriptor. - B3: it marks the descriptor NIC Owned and rings the doorbell - writing the updated Tail (TDT) to the CSR to tell the NIC a new frame is ready.

Step C - NIC transmit + DMA - C1: seeing TDT advanced, the NIC's DMA engine reads the descriptor and DMA-reads the packet data out of Host Memory. - C2: the NIC transmits the Ethernet frame onto the wire (via its internal FIFO). - C3: after the send, the NIC writes the descriptor back (sets DD / done, flips status to Driver Owned) and advances TDH - control returned to the driver.

Step D - TX-completion interrupt + NAPI - D1: the NIC raises a TX-done interrupt (MSI-X). TX completions are usually coalesced, so this fires far less often than once per packet. - D2: the ISR acknowledges and masks the IRQ in the CSR. - D3: it schedules the NAPI poll and returns.

Step E - cleanup + recycle - E1: the NAPI poll (napi_poll) reaps the completed (DD) descriptors, DMA-unmaps each buffer and frees the sk_buff (napi_consume_skb). - E2: if the queue had been stopped because the ring was full (netif_stop_queue), the driver now calls netif_wake_queue so the stack can resume transmitting. - E3: once the completions are drained (work_done < budget), the driver calls napi_complete_done (NAPI bookkeeping; it returns whether re-arming is safe) and then re-enables (unmasks) the IRQ itself via the NIC's interrupt-enable register - the same mask (D2) / unmask (E3) pair as on RX, bounding the TX-completion interrupt rate.

Why TX rarely livelocks: the host only sends what it chooses to, and a full Tx ring applies backpressure via netif_stop_queue, so - unlike RX - you cannot be flooded by your own transmissions. That is why TX completions are coalesced and TX is not what NAPI's budget primarily targets (expanded in topic 2).

1.3 The driver's ring cursors vs the hardware head/tail

The figures show the hardware pointers TDH/TDT (and RDH/RDT); the driver also keeps two software cursors per ring - next_to_use and next_to_clean - and it is worth being clear why they are not the same thing.

  • *DH/*DT are the device's view: the head/tail CSRs describe the ring's current physical occupancy - how far the device has consumed (TX) or produced (RX), and how far the driver has published. They live in the NIC; reading the device-updated head is an MMIO access.
  • next_to_use/next_to_clean are the driver's bookkeeping (plain RAM variables; the device never sees them): next_to_use = where the driver is filling/posting next, next_to_clean = where it will reclaim/process next.

Why TX keeps both cursors but RX keeps only next_to_clean. A ring with a separate producer and consumer inherently needs two indices (fill ahead, reap behind) - that is true even in one context. On TX the driver fills (next_to_use) when the stack queues packets and reaps (next_to_clean) later when the device signals completion: two events decoupled in time, so the positions diverge and both must be tracked. Those two actions also run in different contexts (ndo_start_xmit in process/BH context vs vnic_tx_clean in the NAPI softirq) - which is why the TX ring is locked between them (D-23); the lock is a consequence of the contexts, the two cursors a consequence of the producer/consumer split. On RX the producer is the device (its position is RDH, a register), and the driver's two actions - consume the frame and refill the slot - are done together in one NAPI pass (each consumed slot is re-posted immediately), so the refill position never diverges from the consume position: a single next_to_clean suffices (it doubles as the RDT refill point) and next_to_use is unused. (This is a design choice: drivers that batch/defer RX refill, e.g. ixgbe, keep a separate next_to_use refill cursor so they can ring RDT once per burst - the RX analogue of TX doorbell batching (D-32) - batch buffer alloc/map, and enable page reuse by decoupling "took the data" from "re-armed the slot". Cost: a second cursor + page-refcount bookkeeping; see TODO backlog A5.)

Why the driver cannot just use the registers:

  1. A completion does not mean "drain everything now". When the IRQ fires, the NAPI poll runs under a budget and may stop part-way (and reschedule); TX cleanup likewise takes only what is done. The driver must remember where it left off so the next poll resumes correctly - that is next_to_clean. There is no hardware register for "how far the driver has processed"; *DH only says how far the device got.
  2. Avoid the MMIO read. To detect completion the driver scans the in-memory DD bit from next_to_clean (a cheap cached read) instead of reading *DH over MMIO on every completion (expensive - topic 5). DD write-back is also the precise per-descriptor signal; the head pointer is coarser.
  3. Batch the doorbell. next_to_use can run ahead of the last *DT the driver published, so it can fill several descriptors and ring *DT once (xmit_more, D-32) rather than per packet.

So: the registers track the device; the cursors track the driver - needed because the driver processes pending work incrementally and wants to avoid per-completion MMIO reads. (See the datasheet 5.5 control flow for the exact set/clear/check points.)

2. NAPI

NAPI ("New API") is the Linux mechanism that lets a network driver switch between interrupt-driven and polling modes automatically, depending on load. It was introduced around Linux 2.4.20 / 2.6 and is the standard RX (and TX-completion) model for all modern NIC drivers.

2.1 Before NAPI - the pure interrupt-driven model

Without NAPI, a NIC driver handled every packet from interrupt context.

Typical RX flow (old netif_rx model):

sequenceDiagram
  participant NIC
  participant ISR as ISR hardirq
  participant BL as backlog queue
  participant SOFT as softirq
  participant STACK as net stack
  NIC->>ISR: IRQ per packet
  ISR->>ISR: ack device, build skb
  ISR->>BL: netif_rx enqueues skb
  ISR->>SOFT: raise NET_RX_SOFTIRQ
  SOFT->>BL: dequeue skb
  SOFT->>STACK: push up the stack
  1. A packet arrives, the NIC DMAs it into memory and raises an interrupt - one per packet.
  2. The hardware ISR (hardirq) acknowledges the device, wraps the packet in an sk_buff, and calls netif_rx(), which puts it on the per-CPU backlog queue and raises NET_RX_SOFTIRQ.
  3. Later the softirq drains the backlog queue and pushes packets up the protocol stack.

Typical TX flow: the stack calls ndo_start_xmit, the driver puts the frame on the TX ring and starts it; when the NIC finishes sending, it raises a TX-done interrupt per packet, and the ISR frees the completed sk_buff.

2.2 The problem - interrupt storm and receive livelock

The fatal weakness is one interrupt per packet. At high packet rates (think hundreds of thousands of packets per second) this breaks down:

  • Interrupt storm: the CPU spends nearly all its time entering/leaving the ISR. Each interrupt has fixed overhead (context save, cache/TLB disturbance) that is pure waste when multiplied by the packet rate.
  • Receive livelock: hardirqs have higher priority than the softirq that actually drains the backlog. Under a flood, the CPU is so busy taking interrupts and enqueuing onto the backlog that it never gets to run the softirq - so packets pile up and are dropped after the system already paid to receive them. Useful throughput collapses toward zero even though the CPU is at 100%. (Described in Mogul and Ramakrishnan, "Eliminating Receive Livelock in an Interrupt-Driven Kernel", 1996.)
  • No batching: handling one packet per interrupt prevents amortising costs over a batch, and hurts cache locality.

This is fundamentally an RX problem: the RX rate is set by the outside world and is unbounded - a peer or an attacker can push packets at line rate, and the host must react to each one, so the interrupt load is uncapped. TX is the opposite: the host only sends what it chooses to, and a full TX ring applies backpressure (netif_stop_queue), so you cannot be flooded by your own transmissions. That is why only RX can livelock - and why NAPI and budget target RX (expanded in section 2.3 "Why the focus is RX, not TX").

In short: the interrupt-per-packet model gives great latency at low load but does not scale and can livelock at high load.

2.3 How NAPI solves it - interrupt + polling hybrid

NAPI's idea: take an interrupt to notice that work exists, then switch to polling to do the work - and keep the interrupt disabled while polling so a flood cannot storm the CPU.

The mechanism:

  1. On packet arrival the NIC raises an interrupt. The ISR does almost nothing: it masks (disables) the device interrupt and calls napi_schedule() to queue the device for polling. No packets are processed in hardirq. (On a shared-vector NIC like our single-MSI VN1000 this one mask covers both the RX-available and the TX-completion cause; with MSI-X each vector / NAPI instance is masked independently.)
  2. The kernel runs the driver's poll() callback in softirq context. poll() reads the completed descriptors directly from the NIC's RX ring in a batch, up to budget packets, and delivers each up the stack as an sk_buff - with the device interrupt still off. (Exactly how the skb is obtained - a pre-posted skb vs one built at completion - is a separate RX buffer model choice; see section 2.6.)
  3. Wrap-up: - if poll() drained the ring (work_done < budget) -> call napi_complete_done(), then the driver re-enables (unmasks) the interrupt itself -> back to interrupt-driven mode (low latency when idle). - if the ring still has packets (work_done >= budget, i.e. the budget was used up) -> stay in polling mode, do not unmask, and get rescheduled -> keep polling without any interrupts.

This is adaptive with no mode switch in the code:

Load Behaviour Why it is good
Low one interrupt per packet, poll drains immediately minimal latency
High one interrupt, then sustained polling, interrupt stays off no interrupt storm, no livelock, batched, bounded CPU

It also moves the heavy work out of hardirq into softirq (the poll), keeping the hardirq path short, and it removes the backlog queue for NAPI devices - poll() reads the device ring directly instead of going through netif_rx().

The canonical poll() skeleton

Every NAPI driver's poll callback has the same shape - clean TX, process RX up to budget, then decide whether to complete. The whole adaptive behaviour falls out of the single work_done < budget test:

/* Runs in softirq context; net_rx_action() passes @budget (= napi->weight).
 * The return value tells the NAPI core what to do next:
 *   work_done <  budget  -> we drained the ring; core removes us and we unmask
 *   work_done >= budget  -> more may remain; core keeps us scheduled         */
static int driver_poll(struct napi_struct *napi, int budget)
{
        struct my_priv *priv = container_of(napi, struct my_priv, napi);
        int work_done = 0;

        my_clean_tx(priv);                   /* free sent skbs; not counted vs budget */

        while (work_done < budget) {          /* RX, batched (steps D1-D3) */
                if (!ring_has_owned_desc(priv))   /* ring drained for us -> stop */
                        break;
                work_done += my_rx_one(priv);     /* D2 read data + D3 netif_receive_skb */
                my_refill_and_advance(priv);      /* E1/E2 recycle the descriptor */
        }

        if (work_done < budget) {             /* did not hit the cap => ring is empty */
                if (napi_complete_done(napi, work_done))  /* leave the polling state */
                        my_irq_enable(priv);              /* DRIVER unmasks the IRQ (e.g. IMS) */
        }
        /* else work_done >= budget: stay scheduled, IRQ stays masked, polled again */

        return work_done;
}

Two things to read off the code: work_done is the count processed in this one call (range [0, budget]), and napi_complete_done() only does NAPI bookkeeping - the actual unmask is the driver's my_irq_enable() register write, guarded by it.

A worked example - 5 packets pending, budget = 3

budget caps one poll() call, not the queue, so a 5-packet burst drains over two poll calls on a single interrupt:

  1. Interrupt + schedule (once): the NIC raises one IRQ; the ISR masks it and napi_schedule()s the poll. The IRQ stays masked until the burst is drained.
  2. poll() call #1 (budget = 3): the loop runs 3 times (work_done = 3), delivering and recycling each packet. work_done == budget -> it hit the cap, so the ring may still have packets -> it skips napi_complete_done and does not unmask; it returns 3 and net_rx_action() keeps the NAPI scheduled.
  3. poll() call #2 (budget = 3): 2 packets remain, so the loop runs twice and then ring_has_owned_desc() is false -> work_done = 2. work_done < budget -> it calls napi_complete_done() and the driver unmasks the IRQ; returns 2. Back to interrupt-driven mode.

Net result: 1 hardware interrupt, 2 poll() calls (3 + 2 packets), one unmask at the very end. Two consequences worth internalising:

  • Packets that arrive while the IRQ is masked raise no new interrupt - the next poll() simply finds and processes them. That is the interrupt-storm / livelock defence in action.
  • Even an exact fit needs a "confirming" poll: 3 packets with budget = 3 take call #1 (work_done == 3, no complete) and call #2 (work_done == 0 < 3, complete + unmask) before the ring is seen as empty.

net_rx_action() also bounds the whole softirq run with an overall quota (netdev_budget, default 300 packets) and a 2-jiffy time limit; if a flood exceeds either, the remaining work is handed to ksoftirqd so one busy queue cannot starve the rest of the system.

Why the focus is RX, not TX

NAPI was built to tame RX, and budget bounds RX packets - not because RX interrupts are individually more expensive, but because of an asymmetry in who controls the rate:

  • RX is externally driven and unbounded: packets arrive from the network at a rate the host does not control (a peer or an attacker can push line rate), and if they are not serviced the RX ring fills and already-DMA'd packets are dropped. This is the only direction that can flood the CPU and cause receive livelock.
  • TX is self-paced and bounded: the host only transmits what it chooses to send, and when the TX ring fills the driver applies backpressure (netif_stop_queue). You cannot be flooded by your own transmissions, so the TX-completion interrupt rate is self-limiting.

So TX never needed NAPI's "interrupt off + budget" protection. RX is also somewhat heavier per event (allocate/refill buffers, build skb, protocol demux, push up the stack) versus TX completion (free skb, advance the ring), but that is secondary - even if they cost the same, only RX can livelock. TX-completion interrupts are additionally rare by design: drivers set the report-status (RS) bit on only some TX descriptors, and TX cleanup usually rides along in the RX poll (or the next ndo_start_xmit), so it often needs no interrupt of its own. NAPI still cleans TX inside the same poll as a convenience - it just does not count against budget.

2.4 After NAPI - the typical flow

RX with NAPI:

sequenceDiagram
  participant NIC
  participant ISR as ISR hardirq
  participant POLL as poll softirq
  participant STACK as net stack
  NIC->>ISR: IRQ on first packet
  ISR->>ISR: mask device IRQ, napi_schedule
  POLL->>NIC: pull a batch from RX ring up to budget
  POLL->>STACK: napi_gro_receive each skb
  Note over POLL,NIC: drained - napi_complete, re-enable IRQ. still full - keep polling, IRQ off

TX completion with NAPI: submission (ndo_start_xmit) is unchanged - the driver puts the skb on the TX ring and rings the doorbell. The difference is completion: instead of a per-packet TX-done interrupt freeing one skb in the ISR, the driver reclaims completed TX descriptors in batch inside the same poll() (freeing skbs with napi_consume_skb()), so TX completion is mitigated the same way as RX. A single NAPI poll typically handles both TX clean and RX receive for its queue.

2.5 Related, complementary mechanisms

  • Interrupt coalescing / moderation (e.g. Intel ITR): the hardware itself delays/merges interrupts. This is orthogonal to NAPI and stacks with it - NAPI mitigates in software, ITR in hardware.
  • GRO (Generic Receive Offload): napi_gro_receive() coalesces several small RX packets into one larger skb in the poll path, cutting per-packet stack cost.
  • Threaded NAPI: run poll() in a kernel thread instead of softirq, for better control under heavy load.

Why interrupt moderation alone does not replace NAPI

Coalescing reduces the interrupt rate, so it is tempting to think it could have fixed the interrupt-per-packet problem on its own. It cannot, for two core reasons:

  1. It is a static latency/throughput trade-off. The coalescing thresholds (packet count, timer) are fixed up front, and the two goals conflict: tune them small for low latency and a flood still produces many interrupts (the storm returns); tune them large for throughput and every packet pays extra latency at low load (waiting for the timer or for packets that never come). One fixed setting cannot serve both ends. NAPI needs no such knob - it is interrupt-driven at low load and polling at high load automatically.
  2. The device still interrupts; NAPI turns the interrupt off entirely while polling. With coalescing the NIC just interrupts less often, but under sustained load it keeps interrupting at the coalesced rate, and those hardirqs still preempt the softirq that drains packets - the priority inversion that causes receive livelock is unchanged. NAPI masks the device interrupt for the whole polling period, so the device physically cannot interrupt the CPU while it drains the ring, and budget bounds the work per round for fairness. Moderation says "interrupt me less"; NAPI says "do not interrupt me at all while I am working".

So the two are complementary and used together (coalescing tames the first interrupt's rate, NAPI handles overload), but moderation by itself is insufficient - which is exactly why NAPI was added.

3. Deferred execution: kthread vs softirq vs tasklet vs workqueue

A hardware interrupt handler (the top half, hardirq context) must be tiny - it runs with that interrupt line disabled and preempts almost everything. So the real work is deferred to a bottom half that runs later. The kernel offers several deferral mechanisms, and the right one depends on one question first: does the work need to sleep?

3.1 The mechanisms

Mechanism Context Can sleep? Concurrency Latency Typical use
hardirq (top half) hardirq (atomic) no one per IRQ lowest ack the device, mask, kick a bottom half
softirq softirq (atomic) no same type runs on many CPUs at once very low high-rate net/block bottom halves (NAPI = NET_RX_SOFTIRQ)
tasklet softirq (atomic) no serialised - one CPU at a time per tasklet very low simple driver interrupt bottom half
workqueue process (kworker kthread) yes configurable higher (scheduler) deferrable work that may block (alloc, mutex, I/O)
kthread process yes you create/own each one scheduler-dependent long-lived loops / dedicated workers that wait for events

Key relationships: - A tasklet is built on softirqs (TASKLET_SOFTIRQ). It trades a little of softirq's parallelism (a given tasklet never runs on two CPUs at once) for a simpler, race-free programming model - which is why it is the classic "interrupt bottom half" primitive. - softirq and tasklet are atomic: no sleeping, no mutex, no GFP_KERNEL, no blocking I/O. Use GFP_ATOMIC, spinlocks, and napi_consume_skb()-style helpers. - workqueue and kthread run in process context: they can sleep, at the cost of higher, scheduler-dependent latency.

3.2 Choosing one

  1. Does the work sleep / block? (allocate GFP_KERNEL, take a mutex, do I/O, msleep.) - Yes -> process context: workqueue (one-off deferrals) or kthread (a long-lived loop). - No -> atomic context: tasklet (simple) or softirq (only the fixed set, for hot paths).
  2. Is it a short, latency-sensitive bottom half fired from an interrupt? -> tasklet.
  3. Is it a long-running background loop that waits for events and owns a lifecycle? -> kthread.
  4. Is it a hot, high-rate path that must scale across CPUs? -> softirq (in practice you consume NAPI rather than register a new softirq).

Note: tasklets are the classic primitive and still widely used, but recent kernels discourage them for new code (BH-workqueues were added as a replacement around 6.9). For this learning project the tasklet is the clearest way to model an interrupt bottom half.

3.3 How VN1000 uses them

  • Engine = kthread (hal/vnic_hal_l1.c). It is a long-lived "device" that sleeps in wait_event_interruptible() until a doorbell, then does a run. It needs a sleepable context (it blocks on the wait queue) and owns a lifecycle (kthread_run at init, kthread_stop at exit). A kthread with its own event loop is the natural model for "hardware running independently, waiting for the driver to ring the doorbell".
  • Phase 4 simulated MSI handler = tasklet. It models a hardware interrupt handler: short, must not sleep, runs in softirq context, low latency. A workqueue was rejected (process context, can sleep, least interrupt-like) and so was calling napi_schedule() inline (no separation). See architecture.md 8.2 and D-20.
  • NAPI poll = softirq: vnic_poll runs in NET_RX_SOFTIRQ (we do not create this; the kernel runs the registered poll there). That is why it must not sleep - see topic 2.

4. Memory barriers - ordering the descriptor handshake

4.1 The problem: things are reordered

For speed, both the compiler (compile time) and the CPU (run time - store buffers, out-of-order execution) may reorder memory accesses. On weakly-ordered CPUs (ARM, PowerPC) this reordering is aggressive; even x86 can reorder a store followed by a load. That breaks any "publish then signal" protocol between a producer and a consumer:

producer:  desc->addr = ...; desc->len = ...;   // payload
           desc->status = DD;                    // the "ready" flag
consumer:  if (desc->status & DD)                // sees the flag
               use(desc->addr, desc->len);       // reads the payload

Without ordering, the producer's status = DD can become visible before the addr/len writes, or the consumer can read addr/len before it reads status - so the consumer acts on a half-written descriptor. No lock is involved (this is lock-free single-producer/single-consumer), so a barrier is what enforces the order.

Note - why does the compiler reorder at all, and what does it gain? The compiler's only contract is to preserve the behaviour of a single thread (the "as-if" rule): the observable result for this thread must be unchanged. Within that contract it is free to move, merge, or drop memory accesses, and it does so to make the code faster:

  • Hiding load latency. A load from memory can cost hundreds of cycles. The compiler hoists loads early and sinks the dependent use later, so the CPU can do other work while the value is in flight instead of stalling on it.
  • Better register allocation / fewer accesses. Reordering lets it keep a value in a register and reuse it (or merge two accesses into one, or delete a load whose result is already known), cutting memory traffic.
  • Scheduling for the pipeline. Independent instructions can be interleaved to keep the CPU's execution units busy and avoid pipeline bubbles (e.g. not placing a load immediately before the instruction that needs its result).

The catch: the "as-if" rule only looks at one thread. It has no idea another CPU or a device is watching the same memory and depends on the order of the writes. That is exactly where the producer/consumer handshake breaks, and why we must explicitly tell the compiler "do not reorder across this point" with a barrier. (The CPU reorders at run time for the same latency-hiding reasons, under the same single-thread assumption - which is why a compiler barrier alone is not always enough; see 4.3 and 5.2.)

4.2 The pattern (publish / observe, a.k.a. release / acquire)

  • Producer: write payload -> write barrier -> write the flag.
  • Consumer: read the flag -> read barrier -> read payload.

The write barrier keeps the payload writes ahead of the flag; the read barrier keeps the payload reads behind the flag. They must be used in pairs to work.

4.3 The Linux barriers

Barrier Orders Use when the other side is...
barrier() the compiler only (just stops compile-time reordering)
smp_wmb() / smp_rmb() / smp_mb() CPU stores / loads / both, between CPUs another CPU sharing normal memory
smp_store_release() / smp_load_acquire() the flag + everything before/after it the modern, less error-prone CPU pair
dma_wmb() / dma_rmb() accesses to DMA (coherent) memory, as seen by a device a device doing DMA
wmb() / rmb() / mb() strongest, includes MMIO broad ordering incl. device registers

Also relevant: READ_ONCE() / WRITE_ONCE() stop the compiler from tearing or refetching the flag itself; writel()/readl() already carry ordering for MMIO registers.

4.4 Choosing: who is the other side?

  • CPU <-> CPU (two threads/contexts sharing normal memory): smp_wmb/smp_rmb.
  • CPU <-> device via DMA (driver writes a descriptor in DMA memory that a device reads, or reads data a device DMA-wrote): dma_wmb/dma_rmb. On x86 these are often the same instruction as the smp_* ones, but on ARM/PowerPC dma_* is stronger (a wider coherence domain that includes the device), so using dma_* where a device is involved is required for portability.

4.5 How VN1000 uses them

The TX/RX rings are a lock-free producer/consumer with the DD bit as the flag and the descriptor addr/len as the payload (topic mirrors design-decisions D-18/D-19):

  • Driver core (core/vnic_netdev.c) uses dma_wmb/dma_rmb, because from the driver's side the peer is a device (the Level 1 engine today, the Level 2 QEMU device later). Examples: ndo_start_xmit does dma_wmb() before ringing TDT; vnic_tx_clean and vnic_rx_clean do dma_rmb() after seeing DD; the RX refill does dma_wmb() before clearing status. This keeps the core correct for both levels.
  • The Level 1 engine (hal/vnic_hal_l1.c) uses smp_wmb/smp_rmb, because it is a CPU (a kthread) - the L1 "device" side is CPU <-> CPU. The pairing still holds: dma_* is at least as strong as smp_*, so the driver's dma_wmb pairs with the engine's smp_rmb, and vice versa.
  • The flag reads use READ_ONCE(desc->status) so the compiler cannot tear or hoist them.

At Level 2 the engine disappears; the device is QEMU doing real DMA, and the driver's existing dma_* barriers are exactly what that needs - which is why switching the core to dma_* made it Level-2-ready with no datapath logic change.

4.6 The whole driver <-> device synchronization (it is not a lock)

Barriers are one piece. You cannot put a lock between the CPU and a device, so the TX/RX rings are race-free by a stack of four cooperating guarantees:

  1. Single-owner protocol (DD) - at any instant a descriptor is owned by exactly one side and only the owner writes it; DD plus the TDT/RDT doorbells are the handoff points, so the driver and device never write the same descriptor at once.
  2. Ordering across the handoff - producer dma_wmb() before publishing the flag, consumer dma_rmb() after reading it (4.2-4.5). Stops "saw DD=1 but read stale payload".
  3. READ_ONCE on the flag (compiler, not CPU) - READ_ONCE(desc->status) forces the compiler to actually reload status from memory each time, once, untorn - so it cannot cache the device's flag in a register and spin on a stale value. It emits no fence; it is the compiler-level companion to dma_rmb() (the CPU-level ordering). You need both: READ_ONCE so you observe the new DD, dma_rmb() so the payload reads are ordered after it.
  4. Reachable, consistent memory - the descriptor ring is coherent (dma_alloc_coherent, L2), so no manual cache management is needed (only the ordering barriers); the packet buffers are streaming-mapped and dma_unmap'd (which syncs them for the CPU) before the driver reads RX data, and are never freed while the device might still DMA them (the driver waits for DD). D-26 keeps this portable (the RX unmap happens before delivery).

The driver's own tx_ring.lock (D-23) is a separate concern: it serialises the driver's two contexts (ndo_start_xmit in process context vs vnic_tx_clean in the NAPI softirq), not the driver against the device; RX needs no such lock (consume + refill are one NAPI pass). Drop any of the four above and you get a classic bug - a torn handoff, a missed completion, stale data, or a use-after-free of a buffer the device is still writing.

5. MMIO register access - why readl/writel, not a raw pointer

At Level 2 the driver talks to the device's registers (CTRL, the TDT/RDT doorbells, ICR/IMS /IMC, ...) through MMIO in a PCI BAR. pci_iomap() / ioremap() returns a void __iomem * that maps the device's BAR (as uncacheable device memory, not RAM). The only correct way to touch it is the accessor family - readl()/writel() (32-bit), readw/writew, readb/writeb - never by dereferencing the pointer as if it were normal memory.

"Memory-mapped" but not RAM - resolving the apparent paradox. It can seem contradictory that MMIO is memory-mapped yet you must not access it like memory. The resolution: "memory-mapped" means it lives in the memory address space and is reached by ordinary load/store instructions (as opposed to a separate I/O-port space, e.g. x86 IN/OUT) - and readl/writel are exactly those load/store instructions, so there is no conflict with the design. What "memory-mapped" does not mean is that the region behaves like RAM. Memory is not RAM: a BAR is device registers - reads/writes have side effects (ICR read-to-clear, the TDT doorbell) and must not be cached, reordered, merged, split or elided. So you do reach it with load/store (it is memory-mapped), but through accessors that suppress the RAM-style compiler/CPU optimisations and add the right width, endianness and barriers (5.1-5.2). On x86 readl/writel reduce to about a volatile access; on weakly-ordered arches (ARM, PowerPC) they also add byte-swapping and memory barriers.

5.1 Why a raw *ptr dereference is wrong

  1. Elision / reordering (the dangerous one). A plain *ptr is not volatile, so the compiler may cache it in a register, merge two accesses into one, drop a "dead" read, or reorder it. If the access is optimised away, the load/store is never emitted - the device never sees it. (Worse: it can pass in a debug build and silently break at -O2.) readl/writel use a volatile access, so exactly one access of the right width is always emitted.
  2. Access width. readl is a 32-bit access. A struct/pointer dereference may emit byte-sized or merged accesses that a device rejects or mis-handles (real devices, and emulated ones, declare the access sizes they accept).
  3. Ordering / barriers. readl/writel carry a compiler barrier and, on weakly-ordered CPUs, a memory barrier, so MMIO is ordered with respect to other MMIO. This is essential for things like the doorbell: the descriptor in memory must be visible before the TDT write that tells the device to read it.
  4. Endianness. readl/writel are little-endian by definition (they byte-swap on a big-endian CPU); a raw dereference is native-endian and breaks on BE.
  5. __iomem typing. The token is annotated __iomem precisely so that dereferencing it is a sparse / type error - the annotation exists to force the accessors.

5.2 What the ordering covers - and what it does not (vs topic 4)

readl/writel give two guarantees:

  • The access is emitted (volatile): never optimised away, cached, or merged.
  • MMIO is ordered against MMIO: successive register accesses reach the device in program order - writel(a, REG_A); writel(b, REG_B); is seen by the device as A-then-B. Devices often require this ("clear ICR before re-enabling IMS", "write the index register before the data register", ...).

For that second guarantee, two different kinds of reordering have to be stopped - and they are not the same thing:

  • Compiler barrier - stops the compiler reordering / removing / merging accesses at compile time (the instruction stream). readl/writel always include this.
  • CPU / bus barrier - stops the CPU reordering accesses at run time. x86 is strongly ordered (TSO) so this is nearly free; weakly-ordered CPUs (ARM, PowerPC) need a real barrier instruction, which readl/writel include. A compiler barrier alone does not stop CPU reordering.

What this ordering does not cover: an MMIO access vs a normal-memory (DMA) access. readl/writel order MMIO-vs-MMIO only; they say nothing about when a write to a descriptor in ordinary DMA memory becomes visible to the device relative to a doorbell MMIO write. That is the job of dma_wmb()/dma_rmb() (topic 4). So the doorbell needs both - the descriptor barrier and the ordered MMIO write:

/* ndo_start_xmit (core/vnic_netdev.c) */
... fill the descriptor (addr/len/flags) ...     /* writes to normal DMA memory */
dma_wmb();                               /* those become device-visible before the doorbell */
vnic_hal_wr32(priv, VN_REG_TDT, next);   /* writel(): MMIO doorbell; device now DMA-reads the desc */

writel's compiler barrier already stops the compiler sinking the descriptor write below the doorbell, but only dma_wmb() guarantees the device actually observes the descriptor first.

When you deliberately batch and want to skip the per-access barrier, the relaxed variants readl_relaxed/writel_relaxed drop the CPU/bus barrier (keeping the volatile + compiler guarantee) - then you must place the barriers yourself.

5.3 The QEMU connection

This is also why the device sees the right thing at Level 2. The MMIO access traps to QEMU's BAR callback at the physical-address level (KVM faults it out; TCG routes it through softmmu) - but only if the access is actually emitted (point 1), at the right width (point 2), in the right order (point 3). So readl/writel are not merely guest-side hygiene; they are what makes the emulated device observe the intended register operation. See qemu-device-model.md section 3.

5.4 How VN1000 uses them

  • Level 1: vnic_hal_rd32/vnic_hal_wr32 are a plain C switch over an in-memory fake register block - no MMIO, so no accessor is involved (the "device" is a kthread in the same address space).
  • Level 2: the same vnic_hal_rd32/vnic_hal_wr32 become readl/writel on the pci_iomap'd BAR (D-12 keeps core/ unchanged; only the HAL backend differs). The doorbell ordering above and the ICR read-to-clear / IMS/IMC register dance then rely on exactly these accessor guarantees.

6. TUN/TAP - virtual interfaces, and the QEMU tap backend

6.1 The problem it solves

A network interface is normally backed by hardware - a real NIC driver puts frames on a wire. TUN/TAP is the kernel mechanism that lets a userspace program be the "hardware" behind an otherwise-ordinary network interface: the kernel sees a normal netdev, but the frames are handed to / taken from a process through a file descriptor. This is the building block behind VMs, VPNs, tunnels, containers and sandboxes - anything that needs to move packets between the kernel's network stack and a userspace program without a physical NIC. For us, it is how the QEMU process (host userspace) connects the emulated VN1000 to the host's real network stack (Phase 10).

6.2 TUN vs TAP - L3 vs L2

Both are exposed through the same char device /dev/net/tun; you choose the mode when you open it:

  • TUN (IFF_TUN) = L3: the fd carries bare IP packets (no Ethernet header). Point-to-point and routed - the typical VPN/tunnel device (tun0).
  • TAP (IFF_TAP) = L2: the fd carries full Ethernet frames (with MAC headers). Behaves like a real Ethernet port - can be bridged, has a MAC, sees ARP/broadcast. The VM case.

A NIC's datapath is Ethernet frames (our vn_desc buffers hold whole frames, the device does an Ethernet-MAC swap in loopback), so the QEMU NIC backend uses TAP.

6.3 The two ends of a tap

A tap has two ends, and keeping them straight is the key to the whole thing:

  • kernel end - a normal netdev (e.g. vnic-tap0): it has a MAC, you can give it an IP, and ip link / tcpdump / routing / iptables treat it exactly like a physical interface.
  • userspace end - an open fd on /dev/net/tun bound to that interface: read(fd) returns the next frame the kernel transmitted out the interface; write(fd) injects a frame as if it had been received on the interface.

So the directions are mirror-imaged across the fd:

   kernel-side TX  ==  program's read(fd)     (host sends out vnic-tap0  -> QEMU reads it)
   kernel-side RX  ==  program's write(fd)    (QEMU writes a frame       -> appears as RX on vnic-tap0)

6.4 How QEMU uses the tap (data flow)

guest:    app -> guest IP stack -> vnic0 (emulated PCIe NIC)
                                      |   QEMU device:  TX = qemu_send_packet,  RX = receive()
WSL2 host:                       QEMU process  <->  /dev/net/tun (fd)
                                      |
                                 vnic-tap0  (host netdev: MAC, IP 192.168.20.1, tcpdump-able)
                                      |
                                 host IP stack (routing / NAT / firewall)
  • Guest TX: vnic0 transmits -> the device's qemu_send_packet -> write() to the tap fd -> the frame appears as received on vnic-tap0 -> the host stack handles it (replies, routes, NATs).
  • Guest RX: the host sends a frame out vnic-tap0 -> kernel hands it to QEMU via the fd -> QEMU's receive() callback pci_dma_writes it into the RX ring + raises RXAVAIL.

A consequence worth remembering: tcpdump -ni vnic-tap0 is kernel-side - it shows frames at the interface whether or not QEMU has read them. During Phase 10, the host's ICMP replies were visible to tcpdump on the tap even though the emulated device never received them (the read side was paused); the tap capture was a red herring (see lessons-learned L-08). Confirm guest delivery with the device's own RX stats / interrupt count, not just host-side tcpdump.

6.5 Persistent vs transient, and permissions

  • Transient: QEMU (via a helper script) creates the tap on start and removes it on exit; needs CAP_NET_ADMIN at launch.
  • Persistent (ip tuntap add dev vnic-tap0 mode tap user "$USER"): created once and owned by a user, so it survives QEMU restarts and that user can open the fd without root. Creating it still needs CAP_NET_ADMIN (root) because it adds a kernel interface; /dev/net/tun itself is usually world-rw, so opening an existing user-owned tap does not need root (though KVM access often still does). We use a persistent, user-owned tap (run with script=no,downscript=no) so the guest can be rebooted against the same vnic-tap0.

6.6 Why a tap (not slirp) for "a real NIC"

tap puts real L2 on a host interface, so the host can ping the guest, tcpdump real frames, and the link can be bridged / routed / NAT'd - the guest behaves like a real NIC plugged into the host (the Level 2 goal, D-25). The alternative, user-mode slirp, is a userspace TCP/IP stack inside QEMU that NATs guest connections into host sockets: zero host setup and instant internet, but no real L2, and the host cannot initiate to the guest (only hostfwd). The device model is identical for both - only the -netdev differs (see qemu-device-model.md section 6 for the device<->backend API and the full backend comparison).

6.7 How VN1000 uses it

  • Level 1: no tap - the "network" is the in-kernel loopback engine in the same address space.
  • Level 2, loopback mode (-device vnic, the default; the test suite): no tap - the device loops each TX frame back into its own RX ring internally.
  • Level 2, tap mode (-device vnic,loopback=off -netdev tap,...): the device bridges to vnic-tap0 as above - qemu_send_packet on TX, the receive() callback on RX, with can_receive()/qemu_flush_queued_packets() as the RX flow control (D-30). Host-side tap setup + checks are in dev-setup.md section 8; the boot + guest-side ping/verify in qemu-build.md section 8.

7. Throughput mechanisms - TX doorbell batching, interrupt moderation, and virtio

This topic explains the real-NIC techniques behind throughput, prompted by the Level 2 measurement that TX (~350 Mbit/s) ran ~10x slower than RX (~3.77 Gbit/s) over the tap (lessons-learned L-09). The mechanisms below are what real NICs/drivers use; understanding them explains both that result and why a paravirtual design (virtio) exists.

7.1 The cost that actually differs: real hardware vs emulation

A NIC's two "edge" events are the TX doorbell (driver -> device: a write to a tail register saying "descriptors are ready up to N") and the completion interrupt (device -> driver). The mechanism is the same on real hardware and in QEMU; the cost is not:

event on real hardware under KVM emulation
TX doorbell (writel(TDT)) a posted PCIe write - fire-and-forget, often write-combined; ~tens of cycles an MMIO VM-exit - trap to the hypervisor + device model + return; ~thousands of cycles
completion interrupt a real MSI the LAPIC delivers msi_notify -> APIC injection into the vCPU (also an exit/inject)

So a per-packet doorbell is nearly free on bare metal but a major tax under virtualization - which is exactly why VN1000's TX (one writel(TDT) per ndo_start_xmit) stalls while RX, driven by the host/iothread and drained by NAPI batching, runs far faster (topic 2, topic 5, L-09).

7.2 TX doorbell batching (xmit_more) and friends

Real drivers do not ring the doorbell per packet. The stack hands the driver a burst (GSO segmentation, qdisc bulk dequeue) and sets netdev_xmit_more() on all but the last; the driver defers the tail write and rings it once per burst (igb/ixgbe/i40e/mlx5/virtio all do this). Related real-NIC TX economies: - BQL (Byte Queue Limits) - bound the bytes outstanding in the ring to cap latency/bufferbloat. - TX-completion coalescing - one interrupt per batch of sent frames, cleaned in one NAPI poll. - RS-bit thinning - set the TX descriptor's Report-Status bit only every Nth descriptor, so the NIC writes back DD (a DMA write) sparsely instead of per descriptor.

7.3 Interrupt moderation (ITR / adaptive DIM)

To avoid an interrupt per packet under load, NICs have an interrupt-throttle register (Intel ITR/EITR, in microseconds or interrupts/sec): raise at most one IRQ per N us, or after K packets or T us, whichever first. Modern drivers add DIM (Dynamic Interrupt Moderation, net/dim) - adjusting the rate by load (low latency when idle, heavy coalescing when busy). This trades a little latency for far fewer interrupts/NAPI invocations, i.e. host-CPU efficiency.

7.4 The virtio alternative - designed for virtualization

A register + descriptor-ring NIC (e1000, and our VN1000) pays the emulation tax in 7.1 on every doorbell and interrupt - which is why QEMU's emulated e1000 is famously slow under load. virtio avoids it by design. The key thing to understand is that virtio is not one layer - it is a split across guest and host with a shared-memory contract in between:

(a) Where it lives (the layers). It mirrors VN1000's stack but standardised:

VN1000 (real-hardware model) virtio-net
guest driver vnic-l2.ko (netdev) virtio_net.ko (netdev) - same layer
guest core/transport our MMIO regs + vn_desc rings virtio_ring (vring) + a transport: virtio-pci (also -mmio, -ccw)
contract (ABI) our vnic_regs.h the virtio spec: virtqueue layout, config space, feature bits
host device QEMU hw/net/vnic.c (MMIO/DMA/MSI callbacks) a backend in (1) QEMU userspace (hw/net/virtio-net.c), (2) the host kernel (vhost-net), or (3) another process (vhost-user/DPDK)

So the driver sits where any NIC driver sits in the guest; the device sits where our QEMU model sits - or is pushed into the host kernel (vhost) for speed.

(b) The virtqueue - the shared structure, in guest RAM, that both sides read/write. The split layout has a descriptor table ({addr,len,flags,next}, like our vn_desc), an available ring the guest publishes ("these descriptors are yours to process"), and a used ring the device publishes ("done with these"). Same producer/consumer + ownership idea as our TX/RX rings + DD bit, but the handshake is the avail_idx/used_idx indices rather than a per-descriptor done bit.

(c) Why it is fast - event suppression. The two expensive events are the kick (guest -> host: write a notify register after adding to avail = our TX doorbell, a VM-exit) and the interrupt (host -> guest after adding to used = our completion MSI). With VIRTIO_F_EVENT_IDX each side can tell the other "do not notify me until you reach index X": while the device is actively processing it suppresses the guest's kicks (no exit - the guest just writes the ring and the device picks it up), and while NAPI is polling the guest suppresses the device's interrupts. This is exactly the xmit_more batching (7.2) + NAPI + ITR (7.3) ideas, but built into the ring protocol and bidirectional/dynamic: under load, kicks and interrupts fall to near-zero (the busy side just polls shared memory); an exit/injection happens only to wake an idle peer.

(d) vhost - bypassing QEMU on the hot path. With vhost-net, the datapath (walking the vrings, moving frames to/from the tap) runs in a host-kernel thread; QEMU only does setup. The kick and interrupt use ioeventfd / irqfd, so KVM signals the vhost thread / injects the IRQ without going through QEMU userspace at all.

(In this project the guest's enp0s2 is exactly a virtio-net device - -device virtio-net-pci backed by -netdev user (slirp) - so it is cheap and fast, versus VN1000 deliberately on the real register path.)

7.5 How this maps to VN1000

VN1000 is deliberately a real-hardware-style register + ring NIC (e1000-like), so it does pay the emulation cost - which makes it the right place to study these mechanisms (and virtio the contrast it intentionally is not). Current state and plan: - It already declares VN_CAP_MODER and a VN_REG_ITR register - present in the spec, not yet implemented. - TX rang the doorbell per packet (one writel(TDT) per ndo_start_xmit) - the measured TX bottleneck (L-09). - Implemented + measured (Phase 12): (a) TX doorbell batching via netdev_xmit_more() (D-32) - ~4.5x TX (414 Mbit/s -> 1.86 Gbit/s, L-10); (b) device-side ITR honouring VN_REG_ITR via a QEMU timer (D-33) - ~60x fewer interrupts at flat throughput (L-11); and an MTU sweep (L-12) - ~2.5x throughput at 9000. The interrupt-rate findings are why the next two subsections matter.

7.6 Per-packet vs per-byte cost (why MTU lifts throughput but not interrupts)

NIC datapath work splits into two kinds of cost, and tuning is largely about which one you are paying:

  • Per-byte - scales with the bytes moved: the payload DMA/copy, the checksum, the wire/serialisation. MTU does not change this (the same bytes are moved either way).
  • Per-packet - a roughly fixed cost paid once per frame regardless of its size: a descriptor (DMA-read it, write back DD, the dma_wmb/dma_rmb, ring-index updates), a dma_map/unmap, an sk_buff alloc/free, a trip through the IP/TCP stack (eth_type_trans, GRO, routing, socket queueing), and - in emulation especially - a qemu_send_packet/receive() call + a tap-fd op + a host-stack entry per frame.

Larger MTU carries the same bytes in ~6x fewer packets (9000 vs 1500), so it cuts the count of the per-packet work - that freed CPU is what raises throughput (L-12: ~2.5x at 9000). Strikingly, at the higher jumbo throughput the packets/sec is actually lower (~257k pps at 1500/3.0 Gbit/s vs ~103k pps at 9000/7.4 Gbit/s), so per-second per-packet work drops even as bytes/sec climbs.

It does not cut interrupts, because the interrupt rate is decoupled from packet count: NAPI already batches many packets per interrupt, and the rate is set by how often NAPI drains below budget and re-arms (topic 2), not by how many packets there are. So MTU's saving shows up as throughput, while the interrupt rate is governed by NAPI/ITR (L-11/L-12). This per-packet-vs-per-byte split is the single most useful lens for the tuning choices below.

7.7 Tuning a NIC to the workload - throughput vs latency/IOPS

The knobs trade the same resources differently, so they are tuned to the application. The two poles:

  • Bulk throughput (storage replication, backup, video, big data): maximise bytes/sec and CPU efficiency; per-packet latency does not matter.
  • Latency / packet-rate (IOPS) (RPC, key-value stores, HFT, telco): minimise per-packet latency and sustain high packets/sec; spend CPU to do it.
Lever Bulk throughput Latency / IOPS Why (the trade)
Interrupt moderation (ITR) high coalescing low/off (or adaptive DIM) coalescing cuts interrupts/CPU but adds up to ITR us latency (L-11)
MTU / jumbo 9000 standard (1500) jumbo amortises per-packet cost (7.6); useless for already-small transactional packets
Coalescing offloads TSO/GSO + GRO/LRO on GRO/LRO off they hold/merge packets -> throughput up but latency up

The recurring theme: coalescing/batching (ITR, GRO, deep rings) buys throughput/CPU at the cost of latency; polling/shallow-queue/no-coalesce buys latency at the cost of CPU. "Adaptive" schemes (NAPI itself, and DIM for ITR) try to sit in between - low latency when idle, heavy coalescing under load.

Checksum offload is special - it is not on the latency-vs-throughput axis at all. Today VN1000 computes the IP/TCP/UDP checksum in software (the CPU reads every payload byte - a per-byte cost, 7.6 - on TX before sending and on RX to verify). A NIC with hardware checksum offload does it in silicon: on TX the stack writes a partial checksum and the NIC fills it; on RX the NIC verifies and sets CHECKSUM_UNNECESSARY so the stack skips it. Because it removes a per-byte CPU cost in both directions with no latency penalty, it is enabled in essentially every profile - and it is foundational: TSO/GSO and GRO/LRO depend on it (the NIC must checksum the segments it splits or merges), so without checksum offload the high-throughput offloads cannot work. In emulation it is even cheaper: between a guest and the host (or two VMs on one host) the data never crosses a real wire, so the checksum can be skipped entirely - this is exactly what virtio-net does with VIRTIO_NET_F_CSUM/GUEST_CSUM and a big reason it is fast. VN1000 has no offload yet (VN_CAP_CSUM is reserved), so it pays the full software-checksum cost; adding it is backlog.

Where VN1000 sits: it has the throughput levers we built and measured - MTU (ndo_change_mtu), ITR (D-33, the latency-vs-interrupt knob), TX doorbell batching (D-32), and NAPI. The IOPS/scaling and offload levers are deliberately backlog: multi-queue + RSS + MSI-X (A3), checksum/TSO/GRO offloads (VN_CAP_CSUM reserved), a runtime ring-size param, adaptive DIM, and busy-poll. That set is exactly the menu a real driver exposes (via ethtool -C/-G/-K/-L, SO_BUSY_POLL, ...) to tune for the workload.

7.8 TSO / GSO / GRO / LRO - moving segmentation and coalescing off the hot path

Two stateless offloads cut per-packet work by changing where segmentation (TX) and coalescing (RX) happen. They are the per-packet-cost lever (7.6) applied to the stack/driver, complementary to MTU.

TX - segmentation (TSO): - Today (no TSO; GSO in software): a large write() -> TCP/IP runs once on the big skb -> GSO segments it into MSS-sized skbs just before the driver -> the driver sees a burst of MSS frames (xmit_more, batched doorbell, L-10) -> N descriptors, N device sends. The upper-stack cost is paid once (GSO's win), but the driver/descriptor/doorbell cost is still per segment. - With TSO (NETIF_F_TSO|SG|HW_CSUM): the stack passes one ~64 KB GSO super-skb all the way to ndo_start_xmit (skb_is_gso, gso_size=MSS); the driver maps it with SG (header + page frags = a few descriptors, no skb_linearize) plus a TSO command (mss, header length); one doorbell; the device segments it into MSS wire frames (replicate the header, fix per-segment seq/IP-id/checksum). 64 KB then costs one xmit + a few descriptors + one doorbell - the per-packet driver/descriptor/doorbell/exit cost is amortised over 64 KB. This is jumbo's benefit (7.6) without jumbo on the wire (frames stay MSS-sized).

RX - coalescing (GRO already; LRO is the HW version): - GRO is already active: vnic_rx_clean delivers via napi_gro_receive(), which coalesces same-flow segments into a larger skb in the NAPI poll (software) - so RX already gets the fewer-trips-up-the-stack benefit. - LRO is the hardware version: the device coalesces same-flow segments before delivery (NETIF_F_LRO), handing the driver fewer/larger skbs. (GRO is preferred for forwarding - LRO is lossy to un-coalesce - so LRO is an end-host optimisation.)

Prerequisite: TSO requires checksum offload (the device must checksum each segment it creates) and SG. So checksum offload (7.7) is the foundation the high-throughput offloads build on.

VN1000 today / what would change (backlog A6): GRO is on (napi_gro_receive); there is no TSO/LRO/checksum offload. Adding them means the driver advertises the features and stops linearizing (maps frags as multiple descriptors), the descriptor format gains a TSO command + mss + header length, and the device TX engine becomes a segmenter (and, for LRO, an RX coalescer).

Checksum (7.7) and TSO/LRO/GRO (7.8) are the common stateless per-packet offloads, but a production NIC exposes far more - all viewable / toggleable with ethtool -k (list) and -K <dev> <feat> on|off. The landscape, by category:

8. Where the descriptor ring lives - host RAM vs device MMIO

VN1000 (like e1000/igb/ixgbe/virtio) puts the descriptor ring in host RAM and programs the device with a base address + length; the device DMA-reads/writes the descriptors. A natural question is whether real NICs ever keep the descriptors in the device's MMIO space (a fixed-length ring on the chip), with only the buffer addresses pointing into RAM. They do - it is the minority model, but there is a textbook datasheet example.

8.1 The two models

  • Ring in RAM (dominant): the descriptor array lives in coherent DMA memory; only the head/tail pointers (TDH/TDT/RDH/RDT) are MMIO registers. The CPU writes descriptors with cheap cached stores and rings one doorbell per batch; the device DMA-reads a burst of descriptors (often into an on-chip descriptor cache/prefetcher, which amortises the per-descriptor PCIe read).
  • Descriptor in device MMIO (minority): the descriptor itself is a register (or on-chip SRAM slot) the CPU writes over MMIO. The buffer still lives in RAM (its bus address is written into the descriptor). The ring is necessarily small and fixed (bounded by on-chip storage).

8.2 The canonical datasheet example: Realtek RTL8139

The RTL8139 (ubiquitous, and the chip many people read first; QEMU emulates it in hw/net/rtl8139.c) has no TX ring in RAM. Its transmit descriptors are four fixed register pairs in the chip:

Register Offset Role
TSAD0..TSAD3 (Transmit Start Address) 0x20/24/28/2C the packet buffer's DMA address (-> host RAM)
TSD0..TSD3 (Transmit Status of Descriptor) 0x10/14/18/1C write length/flags (this also starts the send); read back status

The driver uses exactly NUM_TX_DESC = 4 and round-robins cur_tx % 4 (see Linux 8139too.c). This matches the question precisely: descriptors in MMIO, ring length fixed (= 4), buffer address into RAM. (RX is yet another model: a single linear ring buffer in RAM at RBSTART, managed by the CBR/CAPR MMIO pointers - no per-packet descriptors at all.)

The cost is only 4 in-flight TX and a ring you cannot enlarge - which is exactly why the successors RTL8169/8168/8125 switched to an in-RAM descriptor ring (a TNPDS-style base register), i.e. the VN1000/e1000 model. That evolution is the trade-off made visible.

8.3 Other real points on the spectrum

  • PIO NICs - even the buffers are on-chip: NE2000 / RTL8019, and embedded MACs like the SMSC LAN91C111, keep packet buffers in on-chip SRAM and the CPU moves data by I/O/MMIO - no DMA at all. The most extreme "everything on the device" case; storage is fixed and small.
  • Descriptor push into MMIO for latency: Mellanox ConnectX BlueFlame writes the WQE (descriptor) directly into device MMIO via write-combining (buffer still in RAM), to skip the device's DMA-read of the descriptor - a low-latency optimisation for RDMA/HPC, not a capacity model.

8.4 Trade-offs - why the RAM model won

Descriptor in RAM (VN1000, e1000, RTL8169) Descriptor in MMIO (RTL8139 TX)
CPU writes a descriptor cheap cached store expensive uncached/WC MMIO write
Device obtains a descriptor a DMA-read (batched/prefetched -> amortised) already on-chip -> no read, lower latency
Ring size large, programmable (VN1000 <= 4096) small, fixed (on-chip storage)
Doorbell one MMIO per batch the descriptor write is the kick
Best for general throughput tiny/embedded, or ultra-low latency

CPU MMIO writes are uncached and costly, so writing a full descriptor per packet over MMIO does not scale; writing to cheap cached RAM and ringing one doorbell - with the device prefetching descriptors by DMA - wins for throughput. MMIO-resident descriptors only pay off when the ring is tiny by design or when shaving the descriptor-read round-trip matters more than CPU write cost.

8.5 How VN1000 sits

VN1000 deliberately uses the mainstream RAM-ring model (the point is to teach the common driver flow): TDBAL/H + TDLEN program the ring base/length, the descriptors live in dma_alloc_coherent memory, and only TDH/TDT (and the RX pair) are MMIO. The slide deck's address-space diagram (slides/vnic-slides.html, 2.2) draws exactly this split: ring body in RAM, head/tail pointers in the BAR0 window. The RTL8139 is the useful contrast to keep in mind - same idea (buffers in RAM, reached by DMA), but the descriptors themselves are registers.

9. The netdev lifecycle - register vs open (why insmod does not bring the link up)

A common surprise: after insmod, the interface appears but is not running. Loading the module only registers the netdev; the datapath is brought up later, when the interface is administratively set up.

9.1 Two distinct steps

  • register_netdev() (at probe / module load): creates the net_device and makes it visible (ip link shows vnic0), but leaves it administratively DOWN. No buffers are posted, no IRQ is requested, the queue is stopped. The kernel does not call ndo_open here.
  • ndo_open() (at ip link set ... up): the kernel calls it only when something flips the interface UP (IFF_UP). This is where the real bring-up happens: post RX buffers, request the IRQ, program moderation, unmask interrupts, enable NAPI + the device, start the TX queue.

So loading the driver and using the driver are deliberately separate - the interface can exist without consuming an IRQ or DMA buffers until someone actually wants to use it. ndo_stop (on ip link set ... down) is the mirror: it quiesces, frees the IRQ, and drains the buffers.

9.2 Two things that look like "auto up" but are not

  • NetworkManager / systemd-networkd may bring a freshly appeared interface up (and DHCP it) on their own - but that is a userspace policy, not the driver. A driver never sets itself IFF_UP.
  • Carrier / link (STATUS.LINK_UP, netif_carrier_*) is the physical link state and is separate from administrative up; the datapath only runs once the interface is IFF_UP and has carrier.

9.3 How VN1000 uses it

Probe does only register_netdev() (level2/vnic_l2_main.c, level1/vnic_l1_main.c); the bring-up lives in vnic_open -> vnic_up() (core/vnic_netdev.c), which resets the TX indices, posts RX skbs (Model A, D-18), requests the IRQ (L2), programs ITR, unmasks via IMS, enables NAPI + CTRL (EN/TXEN/RXEN), and calls netif_start_queue(). vnic_up() is shared by ndo_open and the ndo_change_mtu re-init. Hence the test flow:

insmod vnic-l2.ko          # vnic0 appears, DOWN - ndo_open has NOT run
ip addr add 192.168.11.2/24 dev vnic0
ip link set dev vnic0 up   # <- triggers ndo_open / vnic_up: IRQ + RX buffers + queue go live

which is exactly what scripts/setup-vnic.sh does. (See also topic 2 for what NAPI/interrupts then do, and the architecture interrupt/polling section.)

10. Interrupt register semantics - why IMS/IMC (and ICR/ICS) are split pairs

The interrupt block has register pairs that look redundant - IMS/IMC for the mask, ICR/ICS for the cause. They are not two state words; each pair is two write ports into one internal register, designed so the driver can flip individual bits with a single, atomic, read-free write. This is the classic Intel e1000 design (and the general hardware W1S/W1C pattern).

10.1 The problem: one read/write mask forces read-modify-write

If the enable mask were a single read-write IMASK, enabling one cause means read-modify-write:

tmp = readl(IMASK); tmp |= VN_INT_TXDONE; writel(IMASK, tmp);

Two things go wrong: - Lost-update race: a NIC has two contexts touching the mask at once - the ISR top-half masks the cause it just took, while the NAPI poll re-enables on completion. Between one context's read and write, the other's update is clobbered. - Cost: an MMIO read is a full round-trip (a VM-exit under virtualisation), far more expensive than a posted write (topic 5); RMW forces that read on the hot path.

10.2 The fix: write-1-to-set / write-1-to-clear

Split the one mask into two write-only ports, where a written 1 acts on that bit and a 0 is ignored:

  • IMS (Mask Set): writel(IMS, TXDONE) enables TXDONE, leaving every other bit alone.
  • IMC (Mask Clear): writel(IMC, TXDONE) disables TXDONE, leaving the rest alone.

Now "flip just these bits" is one posted write, no read, no RMW - the hardware ORs (IMS) or AND-NOTs (IMC) the operand into the single internal mask atomically. Two contexts can each touch their own bits with no lock and no lost update. (Reading IMS returns the current mask, so it is RW1S; IMC is RW1C/write-only.)

10.3 The cause side mirrors it: ICR (read-to-clear) / ICS

The pending-cause register uses the same idea: - ICR is read-to-clear: reading it returns the pending causes and atomically clears them (one access = ack). ICS lets software (or, here, the device engine) set a cause to raise it. - So the ISR does one read of ICR to learn and acknowledge what fired - again no RMW.

What ICS is actually for. It is the software way to raise an interrupt - the steady RX/TX datapath never needs it (the device sets ICR itself when an event happens). On real (Intel-style) NICs ICS is used to self-test the MSI/ISR path at probe, to force a cause (e.g. a link-status interrupt), and for netpoll / ndo_poll_controller (netconsole, kgdb-over-net); the MSI-X form is EICS. The register exists on physical NICs - it is not a simulation artefact. VN1000 then differs by level: the Level 2 device writes ICR directly on completion (real- hardware behaviour; the ICS write path is implemented but unused by the datapath), while the Level 1 in-kernel engine - which reaches its fake device only through the MMIO accessors - deliberately raises completions by writing ICS, so the L1 path stays faithful (raise cause -> MSI -> ISR reads ICR) rather than calling napi_schedule behind the device's back. (Datasheet 4.9.)

10.4 How VN1000 uses it (and the residual window -> IAM)

  • ISR top-half: read ICR (ack), writel(IMC, cause) (mask just that cause), napi_schedule().
  • NAPI poll done: napi_complete_done(), writel(IMS, TXDONE|RXAVAIL) (re-arm) - re-fires if a cause arrived while masked (topic 2, D-11/D-20).

Each path writes only its own bits, reads nothing it does not need, and needs no lock for the register. One small window remains - between "read ICR" and "write IMC" a new cause could slip in; real NICs close it with IAM (Interrupt Auto Mask), where reading ICR auto-masks the just-reported causes in one atomic step. That is backlog A1 here.

11. DMA mappings - coherent vs streaming

A device DMAs to bus (DMA) addresses, not CPU pointers, so any memory it will touch must be mapped first. The DMA API gives two kinds of mapping for two access patterns - and a NIC uses both: one for the descriptor ring, the other for the packet buffers.

11.1 Why map at all

The device sees bus addresses (an IOMMU, if present, translates them), not kernel virtual addresses, and on non-cache-coherent architectures the CPU's caches and the device's view can differ. dma_alloc_* / dma_map_* produce a dma_addr_t the device can use, install any IOMMU entry, and handle cache coherency / bounce buffers. On x86+KVM it is mostly an identity map with no cache ops, but using the API keeps the driver portable (D-26).

11.2 Coherent (consistent) mapping - dma_alloc_coherent

Allocates a block and maps it for the device's whole lifetime; CPU and device see a consistent view with no explicit sync (the kernel hands back coherent - often uncached - memory). Best for long-lived, small, frequently and mutually accessed structures. Trade-off: CPU access can be slower (uncached) and it allocates in coarse units. Freed with dma_free_coherent. - Use for: the descriptor ring (both sides poll it constantly).

11.3 Streaming mapping - dma_map_single / dma_map_page

Maps an existing buffer for one transfer with a direction (DMA_TO_DEVICE / DMA_FROM_DEVICE); you must dma_unmap_* when the transfer completes (the unmap also syncs/bounces for the CPU). The ownership rule: between map and unmap the buffer belongs to the device - the CPU must not touch it; after unmap (or a dma_sync_*) it is the CPU's again. More efficient for one-shot transfers, works on arbitrary kernel buffers (e.g. skb data), and may transparently use a bounce buffer (SWIOTLB) if the buffer is not device-addressable. Always check dma_mapping_error(). - Use for: the packet buffers (mapped per TX/RX).

What SWIOTLB is. "Software I/O TLB" - the kernel's bounce-buffer fallback for when a device cannot reach a buffer (a narrow dma_mask, no IOMMU to remap, or a confidential VM where guest memory is not host/device-visible). The kernel reserves a small pool of low, device-addressable memory at boot; dma_map_single then hands the device a copy in that pool instead of the original buffer - DMA_TO_DEVICE copies in at map time, DMA_FROM_DEVICE copies back at unmap time. It is transparent to the driver, and is exactly why the rules above hold: you must use the DMA API (the device may be reading a bounce copy, not your buffer), must not touch the buffer between map and unmap, and must dma_unmap an RX buffer before reading it (the copy-back happens at unmap). On x86+KVM with a 64-bit mask VN1000 rarely bounces, but using the API keeps it correct if it does (D-26). The cost is that copy - which is why bouncing is a fallback, not the norm.

11.4 Choosing - and a caveat

coherent (dma_alloc_coherent) streaming (dma_map_single)
lifetime long-lived (alloc + map together) per-transfer (maps existing memory)
sync none (consistent view) dma_unmap / dma_sync at the handoff
CPU access may be uncached / slower normal cached memory
typical use descriptor rings, control structs packet buffers

Caveat: coherent != ordered. Coherency removes the need for cache sync, but you still need the dma_wmb()/dma_rmb() barriers (topic 4) to order the DD flag against the payload. Coherency is about visibility; barriers are about ordering - both are required for the ring handshake.

11.5 How VN1000 uses them

  • Level 2 (hal/vnic_hal_l2.c): the ring is dma_alloc_coherent (its bus address goes into TDBAL/H, RDBAL/H); packet buffers are dma_map_single (vnic_hal_map) + dma_unmap_single (vnic_hal_unmap). Core's RX clean unmaps the old buffer before delivering it - the unmap syncs it for the CPU - which keeps RX correct under a vIOMMU / non-coherent host, not only on x86 (D-26); dma_mapping_error is checked (D-22).
  • Level 1 (hal/vnic_hal_l1.c): neither - the ring is kcalloc and the "map" just returns the pointer, because the engine is an in-kernel kthread in the same address space (no real DMA).
  • The HAL hides the choice behind vnic_hal_ring_alloc / vnic_hal_map / vnic_hal_unmap, so the shared core/ datapath is identical across levels (D-12). (Slide deck 3.6 summarises this.)

12. Bring-up and teardown ordering - enabling NAPI, the controller, and interrupts

ndo_open and ndo_stop are not just "turn things on / off" - the order matters, because three things depend on one another: the NAPI poll context (topic 2), the device / controller (it produces completions), and the interrupt that connects them (topic 10). Get the order wrong and a completion is lost, or the close path never converges.

12.1 The one rule

Enable NAPI before the interrupt that can schedule it; on teardown, stop that interrupt source before draining NAPI.

The poll is run from the ISR via napi_schedule(). napi_enable() clears NAPI_STATE_SCHED, which makes the poll schedulable; until that is done a napi_schedule() does not start a poll. So if an interrupt fires before napi_enable(), the completion it signals is never polled - it sits in the ring with its DD set and nothing reaps it (a stuck queue / lost interrupt). Hence NAPI must be ready first.

Note that enabling the controller is not the same as arming the interrupt. On VN1000 the device producing completions (CTRL.EN) only latches causes in ICR; no MSI fires until IMS arms it (topic 10). So the hard constraint is napi_enable before IMS; placing it before CTRL.EN too is just the clean, conservative choice.

12.2 Bring-up order (vnic_up)

napi_enable()                     # 1. poll context ready (schedulable)
wr32(CTRL, EN | TXEN | RXEN)       # 2. controller on - device may DMA / latch causes
wr32(IMS, TXDONE | RXAVAIL)        # 3. arm interrupts - now MSI -> ISR -> napi_schedule can fire
netif_start_queue()               # 4. allow the stack to hand us TX

"Inside-out": make the poll context ready, then the producer (device), then the wire that connects them (interrupt), then finally the TX queue. (request_irq happens earlier still - registering the handler is harmless while interrupts are masked.)

12.3 Teardown order (vnic_down) - the mirror, with one subtlety

netif_tx_disable()                # 1. stop the queue (+ wait out any in-flight xmit)
wr32(CTRL, 0)                     # 2. controller off - device stops producing / latching causes
quiesce()                         # 3. wait out any in-flight engine run (no new completions)
napi_disable()                    # 4. flush the in-flight poll; blocks until it clears SCHED
wr32(IMC, <all causes>)           # 5. mask interrupts - LAST

Two ordering rules live here:

  1. Disable the source before napi_disable. napi_disable() blocks until the current poll finishes and prevents new ones; it can only converge once the device has stopped producing completions and raising interrupts (steps 2-3). Disable first, drain second.
  2. Mask interrupts (IMC) after napi_disable, not before. A NAPI poll re-arms IMS at the end of each run (topic 10). If you masked before the last poll completed, that poll's re-arm would simply switch interrupts back on. After napi_disable there is no poll left to re-arm IMS, so the IMC mask finally sticks.

"Outside-in": queue, then producer, then NAPI, then the interrupt mask - the exact reverse of bring-up.

12.4 How VN1000 uses it

  • vnic_up: napi_enable() → CTRL = EN|TXEN|RXEN → IMS → netif_start_queue() (exactly 12.2).
  • vnic_down: netif_tx_disable() → CTRL = 0 → vnic_hal_quiesce() → napi_disable() → IMC → vnic_hal_irq_free(); the comment on the IMC line spells out rule 2 ("no poll left to re-arm IMS").
  • Both ndo_open and the ndo_change_mtu re-init go through this same vnic_up / vnic_down pair, so the discipline lives in one place. This is the runtime counterpart of the register-vs-open split in topic 9 - topic 9 is when the datapath comes up; topic 12 is the order within that step.

12.5 The TX queue flow-control (start / stop / wake)

netif_start_queue() is the last step of bring-up for a reason, and it is one of a family of calls (start / stop / wake / tx_disable) that together form a small concurrency protocol, not standalone on/off switches. They control the __QUEUE_STATE_DRV_XOFF bit that gates whether the stack may call ndo_start_xmit.

It assumes the whole TX path is ready. The moment netif_start_queue() runs, the stack may call ndo_start_xmit immediately, possibly on another CPU. So everything that vnic_start_xmit touches must already be valid: the TX ring + its indices, the device enabled (CTRL.TXEN), and NAPI + interrupts armed (so completions get reaped and a full queue gets woken). Starting the queue before the device is enabled would let a frame be submitted to a device that is not consuming - a stuck queue. Hence it comes after napi_enable -> CTRL.EN -> IMS (12.2).

start vs wake - not interchangeable. - netif_start_queue() - used once at bring-up; it just clears the bit, assuming nothing is queued yet. - netif_wake_queue() - used to resume after a runtime netif_stop_queue(); it clears the bit and re-schedules the qdisc (kicks the TX softirq) to flush packets that piled up while stopped.

Using start where you needed wake clears the bit but does not re-run the qdisc, so backlog sits until the next natural xmit.

Stop must be balanced by wake, or the TX watchdog fires. Once started, the netdev TX watchdog (dev->watchdog_timeo, ~5 s) watches for progress; a netif_stop_queue() that is never matched by a wake triggers ndo_tx_timeout. So every stop on the full-ring path must eventually be answered by a wake from the completion path.

There is a stop -> wake race to respect. stop happens in ndo_start_xmit; wake happens in the NAPI TX-clean on another CPU. The classic lost-wake bug is: clean checks "stopped? has room?" before xmit has stopped the queue, so the wake is missed and the queue stays stopped forever. These calls are therefore used under a lock + ordering, not as free-standing toggles.

Teardown uses netif_tx_disable, not netif_stop_queue. netif_tx_disable() (12.3, step 1) both stops the queue and waits out any in-flight ndo_start_xmit (it takes the tx lock), so the ring is safe to quiesce and free afterwards; netif_stop_queue() only sets the bit and is for the hot path.

Single default queue only. netif_start_queue acts on txq 0; before register_netdev there is no queue, and a DOWN interface is not transmitted to anyway. Multi-queue NICs use netif_tx_start_all_queues / netif_start_subqueue; VN1000 is single-queue, so netif_start_queue is the right call.

How VN1000 wires it: - bring-up: netif_start_queue() (vnic_up), last. - full ring: netif_stop_queue() in vnic_start_xmit (on no room, and again after the fill that fills the ring). - completion: vnic_tx_clean does if (cleaned && netif_queue_stopped(netdev) && !vnic_tx_full(ring)) netif_wake_queue(netdev) - the wake half of the protocol, serialised against xmit by tx_ring.lock (+ the dma_wmb on the publish path, topic 4). - teardown: netif_tx_disable() (vnic_down).

13. Reading TCP retransmissions - real loss vs a late (spurious) retransmit

A nonzero TCP "Retr" is not proof of packet loss. This project's tests learned that twice (lessons L-04 under load, L-13 even single-stream): with ring backpressure a segment / ACK can be late rather than lost, so TCP retransmits something that then arrives anyway. This topic is how to tell the two apart.

The fundamental point: the sender cannot know at retransmit time whether the original was lost or merely late - that is why it retransmits (it assumes lost). "Lost vs late" is decided after the fact, from whether the receiver turns out to have gotten the data twice.

13.1 The key discriminator - DSACK (RFC 2883)

  • Late (not lost): the original and the retransmit both arrive, so the receiver has a duplicate and sends a DSACK ("I already had this"). The sender counts TCPDSACKRecv.
  • Truly lost: the receiver never gets a duplicate, so no DSACK comes back.

So the single strongest test: TCPDSACKRecv ≈ TCPRetransSegs -> the retransmits were spurious (late); no DSACK -> lean toward real loss.

Expect RetransSegs ≥ DSACKRecv, not exact equality. A small excess is normal (e.g. 03-concurrency C1 once showed RetransSegs=58 vs DSACKRecv=55) - not every retransmit produces a counted duplicate. The reasons, roughly in order of how often they bite:

  1. Measurement-window lag (usually the main one). A DSACK returns ~1 RTT after its retransmit. When nstat reads right at the end of the run, the last few retransmits' DSACKs are still in flight (or the flow already closed), so the retransmit is counted but its DSACK lands outside the window.
  2. DSACK is best-effort. The dup-ACK carrying it can be delayed-ACK-coalesced or itself lost; the receiver's GRO can coalesce duplicate segments into fewer DSACK blocks; a receiver without SACK sends none at all.
  3. Some retransmits genuinely filled a gap -> no duplicate -> no DSACK. In an internal loopback that gap is not a device/wire drop (the device counters stay 0); it is a host-queue micro-drop (qdisc / socket-buffer overflow under -P8 saturation) or a real tail loss that TLP recovered. Real, but above the driver - not loss on the wire.
  4. Different units. TcpRetransSegs counts retransmitted segments; TCPDSACKRecv counts DSACK events. A segment retransmitted twice, or DSACKs coalesced, breaks the 1:1 mapping.

So read it as a ratio, not an equality: DSACKRecv close to RetransSegs (most retransmits were DSACK'd) and zero device drops = spurious. The opposite - a large RetransSegs with few DSACKs and rising device / qdisc drops - is what points to real loss.

13.2 Which mechanism fired (a clue, not proof)

Trigger Counter Leans
Fast Retransmit (3 dup-ACKs / a SACK hole) TCPFastRetrans, TCPSackRecovery real loss / reordering - the receiver signalled a gap
RTO (timer fired, no ACK at all) TCPTimeouts ambiguous - real loss or delay
TLP (Tail Loss Probe, before a full RTO) TCPLossProbes often spurious - the tail just was not ACKed yet

13.3 The counters: spurious vs real

  • Spurious (the stack itself concluded the retransmit was needless): TCPDSACKRecv / TCPDSACKRecvSegs, TCPSpuriousRTOs (F-RTO), TCPSpuriousRtxHostQueues (blamed on local qdisc/driver queueing), TCPDSACKIgnoredNoUndo / TCPDSACKUndo (saw the DSACK, undid the cwnd cut), DelayedACKLost.
  • Real loss: TCPLostRetransmit, TCPFastRetrans, TCPSACKReorder, and - decisively - a link-layer drop counter > 0 somewhere on the path: device rx_dropped / tx_dropped (sysfs / ethtool -S), qdisc drops (tc -s qdisc), a switch port drop.

13.4 The decision procedure

  1. Link-layer drop counters first. If device drops, qdisc drops and port drops are all 0, no real drop happened on the path - the retransmit was a late arrival.
  2. TCPDSACKRecv vs RetransSegs. Close (most retransmits DSACK'd) -> spurious. Expect RetransSegs ≥ DSACKRecv; a small excess is normal (13.1), so judge the ratio, not equality.
  3. The trigger. LossProbes / Timeouts (timer-based, common under latency) vs FastRetrans / SACK recovery (receiver-signalled - real loss / reordering).
  4. Spurious-specific counters (SpuriousRTOs, SpuriousRtxHostQueues, DSACKIgnoredNoUndo) with value -> even the stack judged it spurious.

The golden pair is link-layer drops + DSACK: both clean -> late; either points to a gap (drops > 0, or fast-retrans with no DSACK) -> real loss.

13.5 Tools

  • nstat / /proc/net/netstat (the TcpExt: counters) + /proc/net/snmp (Tcp:RetransSegs)
  • the MIB the tests sample via tcp_reset / tcp_report (tests/lib.sh).
  • ss -ti - per socket: retrans:, rto, rtt, dsack_dups, reord_seen.
  • tcpdump / Wireshark - the definitive view: it flags "TCP Spurious Retransmission", "Out-Of-Order", "Fast Retransmission", and shows the DSACK block coming back.
  • Device / qdisc drop counters - ip -s link, ethtool -S, tc -s qdisc.

13.6 How VN1000 uses it

The device backpressures on a full ring rather than dropping (D-19), so under load its drop counters stay 0 and any TCP Retr is latency, not loss. The tests therefore gate correctness on the device drop counters (rx_dropped/tx_dropped, sysfs) and treat Retr as informational, printing the TCP MIB to attribute it: 03-concurrency C1 and 01-smoke A2 both do this. Worked example - L-13's one retransmit was RetransSegs=1, TCPDSACKRecv=1 (matched), TLP-triggered, device drops 0 -> spurious late tail, not loss.

14. The sk_buff (skb) - the kernel's packet buffer

Every packet in the kernel is an sk_buff ("socket buffer", skb) - the unit the whole datapath (topic 1) passes around by pointer. On TX the stack builds it and hands it to the driver's ndo_start_xmit; on RX the driver builds it and hands it up with napi_gro_receive. One skb is one packet (a GSO super-skb stands in for many - topic 7.8).

14.1 Two parts: metadata + data

An skb is a control struct (struct sk_buff: total length, the layer-header offsets, the owning socket, checksum/offload flags, the routing dst, ...) plus the packet bytes. The bytes are not embedded in the struct - it points at a separate data area, so the same bytes can be referenced or cloned (skb_clone) without copying.

14.2 Linear vs non-linear (head + frags)

The data area has four pointers, head <= data <= tail <= end: - head..end is the allocated buffer; data..tail is the linear part in use; the space before data (headroom) and after tail (tailroom) is where headers are pushed / data is put. - A non-linear skb also carries data in page fragments (skb_shinfo(skb)->frags[], an array of {page, offset, len}) and/or a frag_list of further skbs. skb->len is the total; skb->data_len is the part held in frags; skb_is_nonlinear() == skb->data_len != 0.

Paged data is how the kernel avoids copies: sendfile() / splice() / MSG_ZEROCOPY reference page-cache or user pages directly as frags (zero-copy), and a GSO super-skb keeps its ~64 KB payload in frags until segmentation (topic 7.8).

14.3 Headers by pointer, not by copy

Layers add/strip headers by moving pointers, not by memcpy-ing the payload: - skb_reserve (reserve headroom), skb_push (prepend a header - TX), skb_pull (remove a header - RX), skb_put (extend the tail - e.g. RX skb_put(len) after the DMA write). - skb->{mac,network,transport}_header record where L2/L3/L4 begin once parsed. On RX, eth_type_trans pulls the Ethernet header, sets skb->protocol, and sets skb->pkt_type (PACKET_HOST, ...).

14.4 skb_linearize - and what it costs

skb_linearize(skb) flattens a non-linear skb into a single contiguous linear buffer: it allocates (if needed) and memcpys every fragment's bytes into the linear area, then releases the pages. So: - a non-linear skb -> it really copies: cost ~ one alloc + a memcpy proportional to the paged bytes (CPU + memory bandwidth), and it can fail (-ENOMEM); - an already-linear skb (data_len == 0) -> a no-op, no copy.

Code that needs the whole packet in one contiguous region uses it: a NIC whose DMA engine cannot do scatter-gather (no NETIF_F_SG), or any path that reads the frame as one buffer. It is the opposite of zero-copy - which is exactly why SG + TSO (topic 7.8) exist: an SG-capable NIC maps each frag as its own descriptor and skips the copy.

14.5 Why the SG check happens at transmit, not at allocation

A natural question: if a NIC lacks NETIF_F_SG, why not just build a linear skb up front instead of building a paged one and linearizing it later? Because: - The egress device is not known when the skb is built. Routing, bonding, bridging, VLANs, tunnels and qdiscs choose the real output device after the skb exists; a clone may even fan out to several devices (bridge flooding, an AF_PACKET / tcpdump tap). There is no single NIC - hence no SG capability - to consult at allocation time. - Paged / zero-copy is the point. sendfile / MSG_ZEROCOPY / GSO deliberately avoid the copy, and most NICs support SG/TSO; forcing a linear allocation everywhere would penalise the common case and defeat zero-copy and GSO. - So the stack checks per-skb, at transmit, against the chosen device: netif_skb_features() + validate_xmit_skb() -> skb_needs_linearize() -> __skb_linearize() only when that device cannot handle the layout. The copy is paid exactly where needed, not globally.

14.6 How VN1000 uses the skb

  • TX (vnic_start_xmit): dma_map_single(skb->data, skb->len) (topic 11), fill a descriptor, ring the doorbell; the real skb is kept in ring->bufs[] (the descriptor token is opaque, D-12) and freed on completion with napi_consume_skb (in NAPI) or dev_kfree_skb_any (elsewhere).
  • RX (vnic_rx_clean): the slot already holds a pre-posted skb (Model A, D-18); skb_put(len)
  • eth_type_trans, then napi_gro_receive; refill the slot with a fresh skb.
  • VN1000 does not advertise NETIF_F_SG, so the stack hands xmit a linear skb already; L1's in-kernel engine also skb_linearizes defensively for its linear loopback copy. SG/TSO (to skip the copy) is backlog A6.
