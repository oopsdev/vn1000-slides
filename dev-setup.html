Development environment SOP - Option B: WSL2 + QEMU Ubuntu guest

Run QEMU + an Ubuntu guest inside WSL2, and do all kernel-module development inside the guest. Rationale: see design-decisions.md D-14 (the WSL2 custom kernel has no matching headers in apt; the guest has headers, kgdb is handy, and a panic only kills the guest; Level 2 needs a QEMU guest anyway).

Run everything in the WSL2 Ubuntu shell (except the .wslconfig in step 0.2, which is on Windows).

0. Prepare the WSL2 host

sudo apt update
sudo apt install -y qemu-system-x86 qemu-utils cloud-image-utils gdb openssh-client

0.1 Check KVM (acceleration; without it things are much slower but still work)

ls -l /dev/kvm        # present -> use KVM; absent -> run-qemu.sh falls back to TCG

If there is no /dev/kvm, add this to %USERPROFILE%\.wslconfig on Windows:

[wsl2]
nestedVirtualization=true

Then run wsl --shutdown in Windows PowerShell, restart WSL, and re-check ls -l /dev/kvm.

1. Build the Ubuntu guest (cloud image + cloud-init)

mkdir -p ~/vnic-vm && cd ~/vnic-vm

# 1.1 Download the Ubuntu 24.04 server cloud image (qcow2)
wget -O base.img \
  https://cloud-images.ubuntu.com/noble/current/noble-server-cloudimg-amd64.img

# 1.2 Create a 20G overlay disk backed by base (base stays clean)
qemu-img create -f qcow2 -b base.img -F qcow2 disk.qcow2 20G

1.3 cloud-init: user and packages

cat > user-data <<'EOF'
#cloud-config
hostname: vnic-guest
users:
  - name: dev
    groups: sudo
    sudo: ALL=(ALL) NOPASSWD:ALL
    shell: /bin/bash
    lock_passwd: false
    plain_text_passwd: "dev"      # for practice; prefer ssh_authorized_keys for real use
ssh_pwauth: true
package_update: true
package_upgrade: false            # do not upgrade the kernel, to keep headers and uname consistent
packages: [ build-essential, make, gcc, gdb, git, kmod ]
runcmd:
  - [ bash, -lc, "apt-get install -y linux-headers-$(uname -r)" ]
EOF

cat > meta-data <<'EOF'
instance-id: vnic-guest
local-hostname: vnic-guest
EOF

# 1.4 Pack into a cloud-init seed disk
cloud-localds seed.img user-data meta-data

How the cloud image recognises seed.img (the NoCloud "signature"). There is no magic byte in the file - cloud-init's NoCloud datasource looks for a filesystem whose volume label is cidata (case-insensitive) with meta-data (and user-data) at its root; the presence of meta-data on a cidata-labelled disk is what marks it as a valid seed. cloud-localds builds exactly that: an ISO9660 image (default; --vfat for FAT) labelled cidata containing the two files. (cloud-init can also be pointed explicitly via the kernel cmdline ds=nocloud, but here the labelled disk is auto-detected.) Verify on the host: bash blkid seed.img # -> LABEL="cidata" TYPE="iso9660" mkdir -p /tmp/sm && sudo mount -o loop,ro seed.img /tmp/sm && ls /tmp/sm # meta-data user-data sudo umount /tmp/sm Whether it then re-provisions depends on meta-data's instance-id (pinned to vnic-guest), not on the label - see the instance-id note in §3. First boot + login below.

2. Boot script

The boot script is the repo's scripts/run-qemu.sh - use it as-is; do not hand-copy a separate version. A divergent copy silently ignores the Level 2 QEMU= / VNIC_DEVICE= env vars (it then boots the system QEMU with no -device vnic, and the device never appears in lspci). It honours VMDIR= (image dir, default ~/vnic-vm), QEMU= (binary path) and VNIC_DEVICE= (append -device vnic); it boots a q35 guest (needed when Level 2 attaches a PCIe device) on the serial console (-nographic) with the gdbstub open (-s, :1234; add -S to freeze at boot).

Symlink it into the image dir so ./run-qemu.sh always tracks the repo:

cd ~/vnic-vm
REPO=~/vnic                                   # your host checkout (or /mnt/c/Projects/Linux_NIC_Simulation)
ln -sf "$REPO/scripts/run-qemu.sh" run-qemu.sh

3. First boot + login

./run-qemu.sh
# On first boot cloud-init creates the user, installs packages, and grows the rootfs (a few minutes).
# The serial console shows a login prompt: user dev / password dev.
# Leave the QEMU console: Ctrl-A then X

Or ssh from another WSL terminal (after cloud-init finishes):

ssh -p 2222 dev@localhost          # password dev

Subsequent boots re-run cloud-init - this is normal. run-qemu.sh keeps seed.img attached, so cloud-init's services start on every boot. Because meta-data pins a fixed instance-id (vnic-guest) and /var/lib/cloud persists in disk.qcow2, the one-time modules (user, SSH key, package install, rootfs grow) run only on the first boot; later boots skip them and do just the lightweight per-boot work, so they are quick. If you instead see a full re-provision every boot, the state is not persisting - check cloud-init status --long and that cat /var/lib/cloud/data/instance-id is stable across boots.

Once the guest is set up, you can make boots a clean no-op by disabling cloud-init (or by removing the seed.img -drive line from run-qemu.sh): bash sudo touch /etc/cloud/cloud-init.disabled Do not run cloud-init clean unless you want to force a full re-provision on the next boot.

4. Verify: headers + hello module

Inside the guest:

uname -r
ls -d /lib/modules/$(uname -r)/build      # should exist (headers installed)
# If missing: sudo apt install -y linux-headers-$(uname -r)

mkdir -p ~/hello && cd ~/hello
cat > hello.c <<'EOF'
#include <linux/module.h>
static int __init h_init(void) { pr_info("hello: init\n"); return 0; }
static void __exit h_exit(void) { pr_info("hello: exit\n"); }
module_init(h_init);
module_exit(h_exit);
MODULE_LICENSE("GPL");
EOF
cat > Makefile <<'EOF'
obj-m += hello.o
all:
    $(MAKE) -C /lib/modules/$(shell uname -r)/build M=$(PWD) modules
clean:
    $(MAKE) -C /lib/modules/$(shell uname -r)/build M=$(PWD) clean
EOF
make
sudo insmod hello.ko && dmesg | tail -2 && sudo rmmod hello

Seeing hello: init / hello: exit -> the environment is OK.

5. Code sync (pick one)

5.1 git clone (private repo, needs a token or gh)

GitHub has disabled password login; a private repo requires a token or gh.

(a) PAT in an environment variable + a helper that reads it at runtime (recommended: token never lands on disk or in the URL)

# 1) env file [the 'export' is mandatory] otherwise it is only a shell variable
#    and the child process (git helper) cannot read it
printf 'export GITHUB_PAT=%s\n' 'your-PAT' > ~/.config/vnic/env
chmod 600 ~/.config/vnic/env
echo 'source ~/.config/vnic/env' >> ~/.bashrc
source ~/.config/vnic/env

# 2) confirm it is really exported into the ENVIRONMENT (not just the current shell)
env | grep -c GITHUB_PAT          # must be 1; if 0 -> the env file is missing 'export'

# 3) confirm the token is valid and can read this repo (hit the API directly, bypassing git)
curl -s -o /dev/null -w "%{http_code}\n" \
  -H "Authorization: Bearer $GITHUB_PAT" \
  https://api.github.com/repos/oopsdev/Linux_NIC_Simulation
#   200 = OK ; 401 = invalid/expired token or stray whitespace ; 404 = valid token but no access to this repo

# 4) clone (the helper reads the token from env only when it runs)
./scripts/clone-vnic.sh ~/vnic    # if this script does not exist yet -> use the inline version below
# inline (first time):
git -c credential.helper='!f(){ echo username=oopsdev; echo "password=${GITHUB_PAT}"; };f' \
    clone https://github.com/oopsdev/Linux_NIC_Simulation.git ~/vnic
git -C ~/vnic config credential.helper '!f(){ echo username=oopsdev; echo "password=${GITHUB_PAT}"; };f'

(b) gh (no need to fuss over scopes)

sudo apt install -y gh && gh auth login     # GitHub.com -> HTTPS -> Login with a web browser
git clone https://github.com/oopsdev/Linux_NIC_Simulation.git ~/vnic

PAT permission notes (read this when curl is not 200)

  • 401: token invalid / expired, or whitespace / newline got mixed in when pasting -> reset with the printf above, or regenerate.
  • 404 (most common): valid token but it cannot see this private repo:
  • fine-grained token: Resource owner = oopsdev, Repository access = Only select repositories -> check Linux_NIC_Simulation (do not pick "Public repositories only"), Repository permissions -> Contents: Read-only.
  • classic token: check the whole repo scope.

Later updates: commit/push on Windows -> in the guest cd ~/vnic && git pull (the helper is persisted; you only need GITHUB_PAT sourced into the environment).

Security: never paste the token into chat / logs / a file that will be committed; keep the env file in the home directory, chmod 600, never in the repo. If the token leaks, Revoke and regenerate it on GitHub immediately.

Do not want to deal with auth -> use the 9p share in 5.2: no clone, no token.

5.2 9p/virtfs (advanced: edit live on Windows, build in the guest)

Add this to the qemu args in run-qemu.sh (share a path WSL can see):

  -virtfs local,path=/mnt/c/Projects/Linux_NIC_Simulation,mount_tag=vnic,security_model=none,id=vnic \

security_model=none: /mnt/c (drvfs) does not support xattr, so mapped-xattr would fail. Mount inside the guest:

sudo mkdir -p /mnt/vnic
sudo mount -t 9p -o trans=virtio,version=9p2000.L vnic /mnt/vnic

Note: /mnt/c/... is already cross-filesystem in WSL, and 9p adds another layer, so it is on the slow side; for speed, share a native WSL path instead (clone the repo into ~/).

6. Kernel / module debugging (gdb)

6.1 Attach to the guest kernel (boot-flow debugging)

On the host (WSL), in another terminal:

# after run-qemu.sh is given -S to freeze at boot:
gdb
(gdb) target remote :1234
(gdb) continue

(To see kernel symbols you need the guest's matching vmlinux; most of the time we debug our own module - see 6.2.)

6.2 Debug your own module (most common)

After loading the module in the guest, get its .text load address:

sudo insmod vnic-l1.ko
cat /sys/module/vnic_l1/sections/.text       # e.g. 0xffffffffc0xxxxxx

Host gdb:

gdb
(gdb) target remote :1234
(gdb) add-symbol-file /path/to/vnic-l1.ko 0xffffffffc0xxxxxx
(gdb) b vnic_start_xmit
(gdb) continue

8. Level 2 tap interface (host side, for the real backend)

Only needed for the tap backend (loopback=off) - real guest <-> host traffic. The internal loopback (-device vnic, the default, what the test suite uses) needs no tap. The tap is a host interface (vnic-tap0): it is the host end of the link whose guest end is vnic0 - QEMU bridges raw Ethernet frames between them (see qemu-device-model.md section 6). All commands here run on the WSL2 host.

8.1 Prerequisites (check once)

ls -l /dev/net/tun                 # must exist (char dev); TAP is created through it
lsmod | grep -w tun || zcat /proc/config.gz 2>/dev/null | grep CONFIG_TUN   # TUN support present?
ip tuntap help >/dev/null 2>&1 && echo "ip tuntap ok"   # iproute2 has tuntap

8.2 Create + address the tap (once per host boot)

sudo ip tuntap add dev vnic-tap0 mode tap user "$USER"   # persistent L2 tap you own
sudo ip addr replace 192.168.20.1/24 dev vnic-tap0       # host end of the point-to-point link
sudo ip link set vnic-tap0 up

8.3 Check it

ip -br addr show vnic-tap0         # expect: UP + 192.168.20.1/24 (LOWER_UP once QEMU attaches)
ip -br link show vnic-tap0         # state + the host-side MAC
ip neigh show dev vnic-tap0        # should be empty; if a stale PERMANENT entry lingers from an
                                   # old static-neigh test, delete it: sudo ip neigh del <ip> dev vnic-tap0
# while the guest pings, watch real frames cross the tap:
sudo tcpdump -ni vnic-tap0 -e

Notes: - vnic-tap0 shows DOWN/NO-CARRIER until QEMU opens it; it goes LOWER_UP once the guest is attached. tcpdump here is kernel-side - it shows frames at the interface even if the emulated device has not read them (this masked the Phase 10 RX bug; see lessons-learned L-08). - The guest's MAC is random per boot (eth_hw_addr_random), so do not pin a static neighbour on the host for the guest IP - let ARP resolve (a stale PERMANENT entry breaks host->guest).

8.4 Boot the guest on the tap, and tear down

Booting with VNIC_TAP= and the guest-side IP + ping/verify are in qemu-build.md section 8. To remove the tap:

sudo ip tuntap del dev vnic-tap0 mode tap

Troubleshooting

  • /dev/kvm missing -> see 0.1; or accept TCG (slow).
  • ssh will not connect -> make sure cloud-init finished (wait a few minutes on first boot); hostfwd is 2222->22, connect to localhost:2222.
  • linux-headers-$(uname -r) cannot be installed -> do not apt upgrade the guest (it upgrades the kernel and causes a mismatch); or after installing headers, sudo reboot to realign.
  • module build cannot find build/ -> headers not installed properly, go back to section 4.
  • tap: ip tuntap add fails -> no TUN support, see 8.1. Guest shows no vnic0 under VNIC_TAP= -> the launched run-qemu.sh lacks the VNIC_TAP branch (run the repo's scripts/run-qemu.sh), or sudo dropped the env (use sudo env VAR=...); confirm with ps -eo args | grep qemu that -netdev tap + -device vnic,...,loopback=off are present.
  • tap: host <-> guest ping fails -> check vnic-tap0 is UP with its IP and no stale PERMANENT neigh; tcpdump -ni vnic-tap0 to see which direction is missing (a host-side reply that the guest never receives is the L-08 RX case - rebuild QEMU with the fix).
